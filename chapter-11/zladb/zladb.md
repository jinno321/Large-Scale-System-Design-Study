## 1. 뉴스 피드 시스템이란?

홈 화면에 지속적으로 업데이트되는 콘텐츠 스트림으로, 상태 업데이트, 사진, 동영상, 링크, 팔로우한 사람·페이지·그룹의 활동 등을 모아 보여주는 시스템입니다 (페이스북 정의 기준).

## 2. 설계 범위

- 주요 기능 정의 - 피드 발행, 뉴스 피드 생성
- 피드 정렬 기준 (시간순 vs 토픽 관련도순 등) - 시간 역순
- 최대 팔로우 가능 인원수 - 5,000명
- 예상 트래픽 규모 - 하루 천만 명
- 이미지·동영상 첨부 지원 여부 - 가능

## 3. 핵심 설계

### 3-1. 피드 발행 (Feed Publishing)

사용자가 글을 올리면 캐시와 데이터베이스에 기록한 뒤, 팔로워들의 뉴스 피드로 전파합니다.

- `POST /v1/me/feed` — 글 작성 (본문 + 인증 헤더)
- `GET /v1/me/feed` — 피드 조회 (인증 헤더)

**팬아웃(Fanout) 방식 비교**

| 구분 | 쓰기 시점 팬아웃 (Push 모델) | 읽기 시점 팬아웃 (Pull 모델) |
| --- | --- | --- |
| 동작 | 글 작성 즉시 팔로워 캐시에 반영 | 사용자가 피드를 열람할 때 그 시점에 생성 |
| 장점 | 실시간 반영, 피드 조회 속도 빠름 | 비활성 사용자에게 자원 낭비 없음, 핫키(특정 키에 트래픽이 몰리는) 문제 없음 |
| 단점 | 핫키 문제 발생 가능, 비활성 사용자에게도 갱신하니 자원 낭비 | 피드 조회 시 지연 발생 |

→ **결론**: 대부분은 쓰기 시점(Push)으로 처리하되, 팔로워가 매우 많은 인플루언서 계정은 읽기 시점(Pull)으로 전환하는 **하이브리드 방식**이 실무적. 추가로 안정 해시(Consistent Hashing)를 적용해 특정 키에 부하가 쏠리는 핫키 문제를 완화할 수 있음.

### 3-2. 뉴스 피드 생성

팔로우한 모든 사람의 게시물을 모아 하나의 피드로 구성하는 단계 (원문에 다이어그램 이미지 첨부).

### 3-3. 캐시 구조

캐시 계층 구조에 대한 다이어그램이 포함되어 있음 (레이어별 캐싱 전략).

## 4. 추가로 논의해볼 만한 주제들

- 데이터베이스 확장: 스케일업 vs 스케일아웃
- 관계형 데이터베이스(SQL, Structured Query Language) vs 비정형 데이터베이스(NoSQL)
- 마스터-슬레이브(master-slave) 다중화 구조
- 레플리카(replica)를 통한 읽기 연산 분산
- 데이터 일관성 모델
- 데이터베이스 샤딩
- 웹 계층을 무상태(stateless)로 운영하는 방법
- 최대한 많은 데이터를 캐싱하는 전략
- 다중 데이터센터 지원 방안
- 메시지 큐를 활용한 컴포넌트 간 결합도 완화
- 핵심 지표 모니터링 (예: 트래픽 몰리는 시간대의 초당 쿼리 수(QPS, Queries Per Second), 피드 새로고침 지연시간 등)

## 1. 피드 캐시는 사용자별로 따로 존재한다

쓰기 시점 팬아웃(push 모델)에서는 A가 글을 올리면, 팔로워인 B·C·D 각각의 "봐야 할 포스팅 ID 목록"에 미리 밀어 넣어요. 즉 **하나의 공용 캐시가 아니라 `user_id`를 키로 하는, 사용자 수만큼의 캐시 엔트리**예요.

```
feed:cache:{user_B_id} → [post_101, post_98, post_95, ...]
feed:cache:{user_C_id} → [post_101, post_87, post_82, ...]
```

**실무 최적화 포인트**

- 포스팅 전체 내용이 아니라 **post_id만** 캐시에 저장 (내용은 포스팅 캐시/DB에서 별도 조회)
- 사용자당 캐시 개수를 **제한**(예: 최근 100~500개)해서 무한정 쌓이지 않게 함
- 팔로워가 아주 많은 계정은 push를 끄고 pull로 전환 (핫키 문제 완화)

---

## 2. 캐시 용량 어림 계산 (Back-of-envelope Estimation)

### 가정

| 항목 | 값 |
| --- | --- |
| DAU (Daily Active Users, 일일 활성 사용자) | 1,000만 명 |
| 사용자당 캐시 보관 포스팅 ID 개수 | 500개 |
| 엔트리당 실제 메모리 사용량 (Redis Sorted Set 오버헤드 포함) | 100바이트 (원본 데이터는 16바이트지만 스킵리스트 등 내부 오버헤드로 실제는 훨씬 큼) |

### 계산

```
사용자 1명: 500개 × 100바이트 = 50,000바이트 ≈ 49KB
전체(원본 1벌): 1,000만 × 49KB ≈ 490GB
고가용성용 레플리카 1개 추가: 490GB × 2 ≈ 980GB (~1TB)
```

### 비용 추정 (AWS ElastiCache 기준)

| 항목 | 수치 |
| --- | --- |
| 노드 종류 | cache.r7g.xlarge |
| 노드당 월 비용 (온디맨드) | 약 $319 (시간당 $0.437) |
| 노드당 실사용 가능 용량 | 약 19.74GiB (스펙상 26.32GiB의 75% — AWS가 복제 버퍼·OS 오버헤드용으로 25% 예비 권장) |
| 필요 노드 수 | 약 48대 (primary 24 + replica 24) |
| 월 비용 (온디맨드) | 약 $15,000 |
| 연 비용 (온디맨드) | 약 $180,000 |
| 연 비용 (1년 예약 인스턴스, 약 40% 할인) | 약 $110,000 |

**결론**: DAU 1,000만 규모 서비스의 전체 인프라 비용 대비로는 감당 가능한 수준이지만, 그냥 넘길 만큼 작지도 않은 금액. 그래서 실무에서는 다음을 적용함:

1. **캡을 더 타이트하게** — 500개 → 100~200개로 줄이면 비용도 그만큼 비례해서 감소
2. **비활성 사용자 TTL 만료** — 최근 활동 없는 사용자는 캐시에서 제거, 재로그인 시 pull로 재생성
3. **데이터 티어링** — 자주 안 쓰는 데이터는 메모리에서 SSD로 자동 이전해 비용 절감
4. **하이브리드 팬아웃** — 팔로워 많은 계정은 push 대상에서 제외

## 3. 친구 수 상한(5,000명)이 있으면 달라지는 점

**스토리지 비용 계산은 거의 그대로.** 위 계산은 "읽는 사람이 캐시에 몇 개를 들고 있냐"(500개 캡)로 정해진 것이라 친구 수와 무관함.

**달라지는 건 쓰기(팬아웃) 처리량.** 한 명이 글을 올렸을 때 발생하는 쓰기 건수는 그 사람의 친구 수에 비례함.

```
가정: DAU 1,000만, 10%가 하루 1건 포스팅 → 하루 100만 건

평균 친구 300명일 때:
  하루 팬아웃 쓰기 = 100만 × 300 ≈ 3억 건/일 ≈ 초당 3,500건

최악의 경우 (전원 상한선 5,000명 친구):
  하루 팬아웃 쓰기 = 100만 × 5,000 = 50억 건/일 ≈ 초당 5.8만 건 (피크는 2~3배)
```

이 정도는 메시지 큐 + 워커로 수평 확장하면 감당 가능한 범위. **핵심은 상한선이 있다는 것 자체가 최악의 시나리오를 예측 가능하게 만들어준다는 점.**

- 페이스북식 "친구(양방향, 상한 5,000명)" 모델 → 팬아웃 한 번에 최대 5,000건으로 상한이 정해져서, 대부분 사용자에게 순수 push를 써도 핫키 문제가 통제 가능
- 인스타그램/트위터식 "팔로우(단방향, 무제한)" 모델 → 팔로워 수천만~수억 명인 계정이 나올 수 있어서, 그런 계정은 push를 포기하고 pull로 전환하는 하이브리드가 필수

→ **친구 관계가 양방향/상한이 있는지, 팔로우 관계가 단방향/무제한인지가 아키텍처 선택 자체에 영향을 주는 설계 요소**

## 4. 인스타그램은 어떤 방식? → 하이브리드가 맞음, 다만 한 겹 더 복잡

일반 사용자는 write 시점 팬아웃(push), 팔로워가 아주 많은 계정(셀럽 등)은 읽는 시점에 가져오는(pull) 하이브리드 구조가 업계 공통 패턴.

**실무에서 교과서보다 복잡한 부분**

- 피드는 단순 시간순이 아니라 **참여율 예측 기반 ML(Machine Learning, 기계학습) 랭킹 모델**이 노출 순서를 다시 정렬함. push로 캐시에 넣는 건 "후보 목록"에 가깝고, 실제 노출 순서는 읽는 시점에 랭킹 서비스가 재계산하는 경우가 많음 → push를 하더라도 read 시점 연산이 완전히 사라지진 않음
- 일부 자료에 따르면 페이스북은 교과서적인 하이브리드보다 실제로는 **읽기 시점(pull/fanout-on-load) 비중이 더 크다**는 지적도 있음 (다만 메타 공식 자료가 모호해서 외부에서 정확히 재구성하긴 어려움)

---

## 요약 한 줄씩

- 피드 캐시는 **사용자별 개별 존재** — 저장 공간은 "읽는 사람 수 × 개인 캐시 용량"에 비례
- 1,000만 DAU 기준 캐시 스토리지는 **~1TB, 연 10만~20만 달러대** — 감당 가능하지만 최적화 유인은 충분
- **친구 수 상한**은 저장 비용엔 영향 없지만, **팬아웃 쓰기 처리량의 최악 시나리오를 예측 가능하게 만들어주는** 설계 장치
- 인스타그램/페이스북은 **하이브리드(push+pull)가 기본**, 그 위에 ML 랭킹이 추가로 얹혀 있어 실무는 교과서보다 한 겹 더 복잡함
