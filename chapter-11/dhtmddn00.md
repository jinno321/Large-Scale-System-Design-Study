# 뉴스 피드 시스템(News Feed System) 설계

## 뉴스 피드란

페이스북의 정의를 빌리면, 뉴스 피드는 **홈페이지 중앙에 지속적으로 업데이트되는 콘텐츠 스트림**이다. 사용자가 팔로우하는 사람들의 사진, 비디오, 글, 상태 변경 등이 시간순 또는 알고리즘 순으로 나열된다.

페이스북, 인스타그램, 트위터의 타임라인이 모두 이 구조다.

## 문제 이해 및 설계 범위 확정

질문을 통해 요구사항을 확정한다.

- **모바일 앱인가 웹인가?** 둘 다 지원한다.
- **중요한 기능은?** 사용자가 새 스토리를 올릴 수 있고, 친구들이 올리는 스토리를 뉴스 피드에서 볼 수 있어야 한다.
- **뉴스 피드 정렬 기준은?** 시간 역순(reverse chronological). 즉 최신 글이 위로 온다.
- **한 사용자가 가질 수 있는 친구 수는?** 최대 5,000명
- **트래픽 규모는?** DAU(일간 활성 사용자) 1,000만 명
- **어떤 콘텐츠를 지원해야 하나?** 이미지, 비디오, 텍스트 전부

> **왜 정렬 기준을 먼저 확정하나?**
> 정렬 방식이 설계 전체를 뒤흔들기 때문이다. 시간 역순이면 "최신 글을 앞에 붙이면 끝"이라 캐시에 리스트 하나만 유지하면 된다. 하지만 페이스북처럼 **관련도(relevance) 기반 알고리즘 정렬**이면, 사용자마다 다른 점수를 매기기 위한 랭킹 서비스와 머신러닝 파이프라인이 통째로 필요해진다.
> 면접에서 이 질문을 먼저 던져 범위를 좁히는 게 중요한 이유다. 이 장에서는 시간 역순으로 가정한다.

## 뉴스 피드 API

클라이언트가 서버와 통신하는 창구다. HTTP 기반 REST로 설계하며, 가장 중요한 건 두 가지다.

### 1. 피드 발행 API (Feed Publishing)

새 스토리를 포스팅하기 위한 API.

```
POST /v1/me/feed
파라미터:
  - content: 포스트 내용
  - auth_token: API 인증에 사용
```

### 2. 피드 읽기 API (Feed Retrieval)

뉴스 피드를 가져오는 API.

```
GET /v1/me/feed
파라미터:
  - auth_token: API 인증에 사용
```

## 피드 발행 흐름

사용자가 글을 올렸을 때 어떤 일이 벌어지는지 본다.

```
사용자 → 로드밸런서 → 웹 서버 → ┬→ 포스팅 저장 서비스 → 포스트 DB / 캐시
                                  ├→ 포스팅 전송(팬아웃) 서비스 → 뉴스 피드 캐시
                                  └→ 알림 서비스 → 친구들에게 알림
```

각 컴포넌트의 역할:

- **사용자**: `POST /v1/me/feed`로 새 포스트를 올린다.
- **로드밸런서**: 트래픽을 웹 서버들로 분산한다.
- **웹 서버**: HTTP 요청을 내부 서비스로 라우팅한다. 인증과 처리율 제한을 여기서 담당한다.
- **포스팅 저장 서비스(post service)**: 새 포스트를 DB와 캐시에 저장한다.
- **포스팅 전송 서비스(fanout service)**: 새 포스트를 친구들의 뉴스 피드에 전달한다. **이 장의 핵심 컴포넌트다.**
- **알림 서비스(notification service)**: 친구들에게 새 포스트가 올라왔음을 알린다. 푸시 알림을 보낸다.

> 알림 서비스는 앞선 10장에서 설계한 그 시스템이다. 여기서는 뉴스 피드 시스템의 하위 컴포넌트로 재사용된다. 웹 서버의 **인증과 처리율 제한**은 4장의 처리율 제한 장치가 그대로 적용되는 부분이다.

## 뉴스 피드 생성: 팬아웃(Fanout)

여기가 뉴스 피드 설계의 핵심이다. **"내 친구가 글을 썼을 때, 그 글이 내 피드에 언제 어떻게 들어오는가?"** 를 결정하는 문제다.

팬아웃(fanout)은 한 사용자의 포스트를 그 사용자와 연결된 모든 사용자에게 전달하는 과정을 뜻한다. 두 가지 접근법이 있다.

### 접근법 1: 쓰기 시점에 팬아웃 (Push 모델)

새 포스트를 작성하는 **순간**, 친구들 각자의 뉴스 피드 캐시에 그 포스트를 밀어 넣는다.

```
루시아가 글 작성
  ↓
루시아의 친구 300명을 조회
  ↓
300명 각각의 뉴스 피드 캐시에 이 포스트 ID를 추가
  ↓
(친구들이 접속하면 자기 캐시만 읽으면 됨)
```

**장점**
- **읽기가 매우 빠르다.** 사용자가 피드를 요청하면 이미 만들어진 캐시를 그대로 반환하면 된다. 실시간으로 뭘 계산할 필요가 없다.
- 뉴스 피드가 미리 생성되어 있으므로 **읽기 지연시간이 짧다.**

**단점**
- **친구가 많으면 느리고 비싸다.** 친구 목록을 가져와 전부의 피드를 갱신하는 작업이 오래 걸린다. 이를 **핫키(hotkey) 문제**라 부른다.
- **비활성 사용자에게도 낭비된다.** 몇 달째 로그인하지 않는 사용자의 피드까지 갱신하느라 자원을 쓴다.

> **핫키 문제가 구체적으로 뭔가?**
> 팔로워가 1억 명인 셀럽이 글 하나를 올렸다고 하자. Push 모델이면 그 순간 **1억 개의 캐시를 갱신**해야 한다. 몇 분이 걸릴 수도 있고, 그동안 시스템 전체가 마비된다.
> 이걸 **팬아웃 폭발(fanout explosion)** 이라고도 한다. 쓰기 한 번이 쓰기 1억 번으로 증폭되는 것이다. 트위터가 초기에 겪었던 유명한 확장성 문제가 정확히 이것이다.

### 접근법 2: 읽기 시점에 팬아웃 (Pull 모델)

피드를 **요청하는 시점**에 친구들의 최신 포스트를 그때그때 모아서 만든다.

```
루시아가 피드 요청
  ↓
루시아의 친구 300명 조회
  ↓
각 친구의 최신 포스트를 가져와 병합
  ↓
시간순 정렬해서 반환
```

**장점**
- **비활성 사용자에게 자원을 낭비하지 않는다.** 접속하지 않으면 아무 작업도 일어나지 않는다.
- 데이터를 친구 각각에게 밀어 넣지 않으므로 **핫키 문제가 생기지 않는다.**

**단점**
- **읽기가 느리다.** 요청할 때마다 여러 친구의 데이터를 모으고 정렬해야 한다.
- 사용자가 피드를 볼 때마다 이 비용이 발생하므로, 읽기가 많은 서비스에서는 부담이 크다.

### 접근법 3: 하이브리드 (실제로 쓰는 방식)

두 방식의 장점만 취한다. **핵심 통찰은 "모든 사용자를 똑같이 대할 필요가 없다"** 는 것이다.

```
일반 사용자 (친구 수백 명)
  → Push 모델 사용
  → 미리 피드를 만들어둬서 읽기를 빠르게

셀럽/인플루언서 (팔로워 수백만 명)
  → Pull 모델 사용
  → 팬아웃 폭발을 피함
  → 팔로워가 접속할 때 셀럽의 글을 그때 가져와 병합
```

즉 피드를 읽을 때, **미리 만들어진 캐시(일반 친구들 글)** 와 **실시간으로 가져온 것(셀럽 글)** 을 합쳐서 보여준다.

또한 **읽기 캐시를 활용해 성능을 높이는** 방향도 함께 적용한다. 일관된 해싱을 통해 요청과 데이터를 보다 고르게 분산시켜 핫키 문제를 완화할 수 있다.

> 앞서 다룬 **안정 해시(consistent hashing)** 가 여기서 등장한다. 뉴스 피드 캐시를 여러 노드에 분산할 때, 특정 노드에 부하가 몰리지 않도록 하는 데 쓰인다.

## 팬아웃 서비스 상세 설계

Push 모델에서 팬아웃 서비스가 실제로 어떻게 동작하는지 본다.

```
포스팅 저장
   ↓
① 그래프 DB에서 친구 ID 목록 조회
   ↓
② 사용자 캐시에서 친구 정보 조회 (설정 확인)
   ↓
③ 친구 목록과 포스트 ID를 메시지 큐에 삽입
   ↓
④ 팬아웃 작업 서버(worker)가 큐에서 꺼내 처리
   ↓
⑤ 각 친구의 뉴스 피드 캐시에 <포스트 ID, 사용자 ID> 추가
```

**① 그래프 DB에서 친구 목록 조회**

친구 관계는 그래프 구조라서 그래프 DB(Neo4j 등)가 적합하다. 친구 관계 조회, 친구 추천 같은 연산에 유리하다.

**② 설정 확인**

여기서 두 가지를 걸러낸다.
- **뮤트(mute) 설정**: 친구지만 그 사람의 글은 안 보겠다고 설정한 경우
- **공개 범위 설정**: 특정 친구에게만 공개하는 경우

**③~④ 메시지 큐 활용**

친구 목록과 포스트 ID를 메시지 큐에 넣고, 작업 서버가 비동기로 처리한다.

> 10장 알림 시스템에서 봤던 그 구조다. 메시지 큐가 **강한 결합을 끊는** 역할을 한다. 사용자는 "글이 등록됐다"는 응답을 즉시 받고, 실제 팬아웃은 백그라운드에서 진행된다. 친구가 5,000명이라고 사용자가 5,000번의 캐시 갱신을 기다릴 이유가 없다.

**⑤ 뉴스 피드 캐시에 저장**

여기서 중요한 최적화가 있다. **포스트 내용 전체가 아니라 `<포스트 ID, 포스터 ID>` 쌍만 저장한다.**

```
뉴스 피드 캐시 구조:
| post_id | user_id |
| 12345   | 999     |
| 12344   | 777     |
| 12343   | 999     |
```

이유는 두 가지다.
- **메모리 절약**: 포스트 본문까지 5,000명의 캐시에 복제하면 같은 내용이 5,000번 저장된다. ID만 저장하면 훨씬 가볍다.
- **데이터 일관성**: 포스트가 수정되거나 삭제되면, 본문을 복제해뒀을 경우 5,000곳을 전부 갱신해야 한다. ID만 있으면 원본 하나만 고치면 된다.

또한 캐시 크기를 무한정 늘릴 수 없으므로 **크기 제한을 둔다.** 대부분의 사용자는 피드의 맨 위 몇백 개만 보기 때문에, 오래된 항목은 잘라낸다.

## 뉴스 피드 조회 흐름

사용자가 피드를 요청했을 때의 흐름이다.

```
사용자 → 로드밸런서 → 웹 서버 → 뉴스 피드 서비스
                                      ↓
                          ① 뉴스 피드 캐시에서 포스트 ID 목록 조회
                                      ↓
                          ② 사용자 캐시에서 작성자 정보 조회
                          ③ 콘텐츠 캐시에서 포스트 본문 조회
                                      ↓
                          ④ 완전한 형태의 뉴스 피드로 조립
                                      ↓
                                 JSON 반환
```

캐시에는 ID만 있으므로, **사용자 정보와 포스트 본문을 별도로 가져와 조립(hydration)** 하는 과정이 필요하다. 이 과정을 거쳐야 클라이언트가 렌더링할 수 있는 완전한 데이터가 된다.

미디어 콘텐츠(이미지, 비디오)는 **CDN**에 저장해 빠르게 제공한다.

## 캐시 구조

뉴스 피드 시스템에서 캐시는 매우 중요한 컴포넌트다. 캐시를 5개 계층으로 나눈다.

| 계층 | 저장 내용 | 용도 |
|---|---|---|
| **뉴스 피드(News Feed)** | 뉴스 피드의 포스트 ID 목록 | 사용자별 피드 |
| **콘텐츠(Content)** | 포스트 데이터. 인기 콘텐츠는 별도 보관 | 본문 조회 |
| **소셜 그래프(Social Graph)** | 사용자 간 관계 정보 | 친구 목록 조회 |
| **행동(Action)** | 좋아요, 답글, 신고 등의 행위 | 상호작용 표시 |
| **횟수(Counters)** | 좋아요 수, 답글 수, 팔로워 수, 팔로잉 수 | 숫자 표시 |

각 계층을 분리하는 이유는 **데이터마다 갱신 빈도와 접근 패턴이 다르기 때문**이다. 좋아요 수(Counters)는 초당 수천 번 바뀌지만 포스트 본문(Content)은 거의 안 바뀐다. 이를 한 캐시에 섞으면 자주 바뀌는 데이터 때문에 안 바뀌는 데이터까지 무효화된다.

## 마무리: 추가로 논의할 점

### 데이터베이스 확장

- 수직적 규모 확장 vs 수평적 규모 확장
- SQL vs NoSQL
- 주-부(master-slave) 다중화
- 복제본(replica)에 대한 읽기 연산: 읽기를 복제본으로 분산
- 일관성 모델: 어느 수준의 일관성을 택할 것인가
- 데이터베이스 샤딩

> 앞서 다룬 CAP 정리와 정족수 합의가 여기서 다시 등장한다. 뉴스 피드는 **약간의 지연이나 불일치를 허용할 수 있는** 서비스다. 내 친구의 글이 1초 늦게 보여도 큰 문제가 없다. 그래서 강한 일관성보다 **결과적 일관성 + 높은 가용성(AP)** 을 택하는 것이 자연스럽다.

### 기타

- 웹 계층을 무상태로 관리하기: 수평 확장의 전제 조건
- 가능한 한 많은 데이터를 캐시할 방법 찾기
- 여러 데이터 센터를 지원할 방법 찾기
- 컴포넌트 간 결합도를 낮추기: 메시지 큐 활용
- 핵심 메트릭 모니터링: QPS, 피드 로딩 시간 등

## 정리

뉴스 피드 시스템의 핵심 질문은 하나다. **"피드를 언제 만들 것인가?"**

- **쓸 때 만들면(Push)** 읽기는 빠르지만, 팔로워가 많을 때 팬아웃 폭발이 일어난다.
- **읽을 때 만들면(Pull)** 팬아웃 폭발은 없지만, 읽을 때마다 비용이 든다.
- **실제로는 하이브리드**로, 일반 사용자는 Push, 셀럽은 Pull을 적용한다.

그 외의 설계는 앞선 장들의 재조합에 가깝다. 메시지 큐로 팬아웃을 비동기화하고(10장), 안정 해시로 캐시 부하를 분산하며(5장), 처리율 제한으로 남용을 막고(4장), 알림 서비스를 하위 컴포넌트로 재사용한다(10장). 시스템 설계가 결국 **검증된 부품들의 조합**이라는 걸 잘 보여주는 사례다.
