# [11장] 뉴스 피드 시스템 설계

## 문제 설정
* **플랫폼 지원:** 모바일 앱, 웹 둘 다 지원 가능해야 한다.
* **핵심 기능:** 뉴스 피드 페이지에 새로운 스토리를 올릴 수 있어야 하고, 친구들이 올리는 스토리를 볼 수 있어야 한다.
* **정렬 기준:** 뉴스 피드는 시간 역순(Reverse chronological order)으로 표시된다.
* **친구 수 제한:** 한 명의 사용자는 최대 5000명의 친구를 가질 수 있다.
* **트래픽 규모:** 매일 천만 명(10million DAU)이 방문하는 트래픽이 쏟아진다.
* **미디어 지원:** 스토리에는 이미지나 비디오 등이 지원된다.

---

## 개략적 설계안 

전체 시스템은 크게 **피드 발행(Feed publishing)**과 **뉴스 피드 생성(News feed building)** 두 가지 부분으로 나뉜다.
* **피드 발행:** 사용자가 스토리를 포스팅하면 해당 데이터를 캐시와 데이터베이스에 기록하고, 친구의 뉴스 피드에 전송한다.
* **뉴스 피드 생성:** 뉴스 피드는 모든 친구의 포스팅을 시간 흐름 역순으로 모아서 만든다.

### 뉴스 피드 API
클라이언트가 서버와 통신하기 위해 사용하는 수단이다. HTTP 프로토콜 기반이고, 상태 정보를 업데이트하거나, 뉴스 피드를 가져오고, 친구를 추가하는 등 다양한 작업에 사용된다. 그중 핵심적인 2개의 API는 다음과 같다.

#### 1. 피드 발행 API
새 스토리를 포스팅하기 위한 API이다. REST API 형태로 HTTP POST 메서드를 사용하며, Body에 포스팅 내용을 넣어서 요청을 보낸다.
* **형태:** `POST /v1/me/feed`

#### 2. 피드 읽기 API
뉴스 피드를 가져오는 API이다. HTTP GET 메서드를 사용하며, Response로 피드의 메타 데이터들을 반환한다.
* **형태:** `GET /v1/me/feed`

### 개략적 설계 컴포넌트

#### [피드 발행 컴포넌트]
* **사용자:** 모바일 앱이나 브라우저로 새 포스팅을 올리는 주체이며, 피드 발행 API를 호출한다.
* **로드밸런서:** 트래픽을 여러 웹 서버들로 분산시킨다.
* **웹 서버:** HTTP 요청을 내부 서비스로 중계하는 역할을 담당한다.
* **포스팅 저장 서비스:** 새 포스팅을 데이터베이스와 캐시에 저장한다.
* **포스팅 전송(팬아웃) 서비스:** 새 포스팅을 친구의 뉴스 피드에 푸시한다. 뉴스 피드 데이터는 속도를 위해 캐시에 보관한다.
* **알림 서비스:** 친구들에게 새 포스팅이 올라왔음을 알리고, 푸시 알림을 발행하는 역할을 한다.

#### [뉴스 피드 생성 컴포넌트]
* **사용자:** 뉴스 피드를 읽는 주체이다.
* **로드 밸런서:** 트래픽을 웹 서버들로 분산시킨다.
* **웹 서버:** 트래픽을 뉴스 피드 서비스로 보낸다.
* **뉴스 피드 서비스:** 캐시에서 뉴스 피드를 가져오는 핵심 로직을 처리한다.
* **뉴스 피드 캐시:** 뉴스 피드를 빠르게 렌더링할 때 필요한 피드 ID 목록을 보관한다.

## 상세 설계

기본적인 흐름을 바탕으로 주요 컴포넌트의 상세 구현과 최적화 방안을 살펴본다.

### 1. 피드 발행 흐름 상세 설계
대부분의 컴포넌트는 개략적 설계 수준에서 처리될 수 있으나, **웹 서버**와 **포스팅 전송 서비스(팬아웃 서비스)**는 추가적인 상세 설계가 필요하다.

#### 웹 서버의 역할 확장
* 웹 서버는 단순히 요청을 중계하는 것 외에 **인증(Authentication)**과 **처리율 제한(Rate Limiting)** 기능을 포함해야 한다.
* 인증 토큰을 HTTP 헤더에 넣어 유효성을 검사하며, 스팸을 막고 특정 사용자가 무분별하게 유해 콘텐츠를 자주 올리는 것을 방지하기 위해 처리율 제한 장치를 함께 수행한다.

#### 포스팅 전송(팬아웃) 서비스
팬아웃(Fan-out)은 어떤 사용자의 새 포스팅을 그 사용자의 모든 친구에게 전달하는 과정이다. 이 메커니즘에는 두 가지 모델이 존재한다.

---

### 쓰기 시점에 팬아웃하는 모델 (Push 모델)
새로운 포스팅을 기록하는 시점에 해당 사용자의 친구들의 뉴스 피드 캐시를 미리 갱신해 두는 방식이다.

* **장점:** 
  * 뉴스 피드가 실시간으로 갱신되며 친구 목록에 있는 사용자에게 즉시 전송된다. (즉시성)
  * 새 포스팅이 기록되는 순간에 뉴스 피드가 이미 계산되어 캐시에 채워지므로, 뉴스 피드를 읽을 때(Read 시점) 드는 시간이 매우 짧다.
* **단점:** 
  * **핫키(Hot키) 문제:** 팔로워나 친구가 아주 많은 인플루언서의 경우, 대량의 친구 목록을 가져와서 그 모두의 뉴스 피드 캐시를 일일이 갱신하는 데 너무 많은 시간이 소요된다.
  * 서비스를 자주 사용하지 않는(비활성 상태) 사용자의 피드까지 미리 갱신하므로 시스템 리소스가 낭비된다.

---

### 읽기 시점에 팬아웃하는 모델 (Pull 모델)
사용자가 뉴스 피드를 읽어야 하는 시점에 실시간으로 뉴스 피드를 갱신하는 요청 기반(On-demand) 모델이다. 사용자가 본인의 홈페이지나 타임라인을 로딩하는 시점에 새로운 포스트를 땡겨온다.

* **장점:** 
  * 비활성화된 사용자나 앱을 거의 사용하지 않는 사용자의 경우 이 모델이 절대적으로 유리하다. 로그인해서 피드를 보지 않으면 리소스를 전혀 쓰지 않기 때문이다.
  * 데이터를 친구 각각에게 일일이 푸시하는 작업이 없어지므로 인플루언서로 인한 핫키 문제가 발생하지 않는다.
* **단점:** 
  * 사용자가 피드를 요청할 때마다 실시간으로 데이터를 긁어와서 조합해야 하므로 뉴스 피드를 읽는 데 많은 시간이 소요된다.

---

#### 하이브리드(Hybrid) 접근법을 통한 트레이드 오프
뉴스 피드 시스템에서는 **피드를 빠르게 가져오는 것(Read 성능)**이 비즈니스적으로 가장 중요하므로 두 가지 방식을 결합한다.
* **일반 사용자:** 대부분의 사용자에 대해서는 읽기 성능이 뛰어난 **푸시(Push) 모델**을 기본으로 사용한다.
* **인플루언서:** 친구나 팔로워가 너무 많은 사용자의 경우, 해당 사용자가 포스팅할 때 푸시하지 않고 팔로워들이 필요할 때 가져가도록 하는 **풀(Pull) 모델**을 사용하여 시스템 과부하를 방지한다.
* **부하 분산:** 아울러, **안정 해시(Consistent Hashing)**를 도입하여 요청과 데이터를 전송 서버 및 캐시 서버에 보다 고르게 분산함으로써 핫키 문제를 한층 더 완화한다.

---

#### 팬아웃 서비스의 세부 작업 단계
1. **소셜 그래프 데이터베이스**에서 해당 사용자의 친구 ID 목록을 추출한다. 그래프 DB는 친구 관계나 친구 추천 등의 복잡한 관계 네트워크를 관리하는 데 가장 적합하다.
2. **사용자 정보 캐시**에서 친구들의 세부 정보를 가져온다. 이때 사용자 설정에 따라 차단(Block)했거나 알림 제한을 걸어둔 친구 관계의 정보는 필터링하여 배제한다.
3. 필터링을 거친 친구 목록과 새 스토리의 포스팅 ID를 **메시지 큐(Message Queue)**에 넣는다.
4. **팬아웃 작업 서버(Worker)**가 메시지 큐에서 데이터를 꺼내어 친구들의 **뉴스 피드 캐시**에 매핑한다. 
   * *주의:* 캐시 메모리 요구량이 과도해지는 것을 막기 위해 캐시에는 포스팅 정보 전체를 넣지 않고 **포스팅 ID만 보관**한다. 
   * 사용자가 뉴스 피드에 올라온 수천 개의 스토리를 끝까지 스크롤하여 모두 읽을 확률은 매우 낮으므로, 최근 글 위주로 캐시 크기에 제한을 두어도 캐시 미스(Cache Miss)는 거의 일어나지 않는다.

---

### 2. 피드 읽기 흐름 상세 설계
이미지나 비디오 같은 용량이 큰 미디어 콘텐츠는 클라이언트와 가까운 **CDN**에 저장하여 최대한 빠르게 로딩되도록 구축한다.

1. 사용자가 뉴스 피드를 읽으려는 요청(`GET /v1/me/feed`)을 보낸다.
2. 로드밸런서가 인입된 요청들을 여러 웹 서버로 분산시킨다.
3. 웹 서버는 피드 조회를 위해 **뉴스 피드 서비스**를 호출한다.
4. 뉴스 피드 서비스는 **뉴스 피드 캐시**에서 해당 사용자를 위해 미리 준비된 포스팅 ID 목록을 빠르게 가져온다.
5. 가져온 ID들을 기반으로, 뉴스 피드에 표시할 사용자 이름, 프로필 사진, 포스팅 텍스트, 이미지 경로 등을 **사용자 캐시**와 **포스팅 콘텐츠 캐시**에서 각각 조회하여 하나의 완전한 뉴스 피드 객체로 조립한다.
6. 구조화된 JSON 형태로 클라이언트(모바일 앱/웹 브라우저)에게 최종 응답값을 반환한다.

## 캐시 구조
뉴스 피드 시스템의 성능을 극대화하기 위해 캐시를 아래와 같이 **다섯 가지 계층**으로 세분화하여 관리한다.

* **뉴스 피드 캐시:** 사용자별 뉴스 피드의 포스팅 ID 목록을 보관한다.
* **콘텐츠 캐시:** 포스팅 데이터(텍스트, 미디어 URL 등) 본문을 보관한다.
* **소셜 그래프 캐시:** 사용자 간의 친구 관계, 팔로우 목록 등을 보관한다.
* **행동(Action) 캐시:** 포스팅에 대한 사용자의 좋아요, 공유, 답글(댓글) 등의 상태를 보관한다.
* **횟수(Counter) 캐시:** 좋아요 수, 댓글 수, 팔로워 수 등의 카운터성 데이터를 보관한다.

## 💡 면접 마무리 팁
시스템 설계 면접에 절대적인 정답은 없으며, 회사나 서비스가 마주한 독특한 제약과 요구조건에 따라 아키텍처는 유연하게 달라진다. 

특히 뉴스 피드 설계는 **데이터베이스의 수평적 규모 확장(Sharding/Partitioning)** 및 **캐시 전략**과 밀접하게 연관되어 있으므로, 데이터가 기하급수적으로 늘어날 때 스토리지와 메모리를 어떻게 효율적으로 확장할 것인지(샤딩 키 선정, DB 레포지토리 최적화 등)에 대해 깊이 있게 논의하고 어필하면 면접에서 좋은 점수를 받을 수 있다.
