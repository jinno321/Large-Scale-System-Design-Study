### 데이터베이스 다중화

- 데이터베이스 서버를 여러개 구축하고 master와 slave로 관계를 설정
- 쓰기 연산 → master에서만 쓰기
- slave → master의 사본 전달 받음 + 읽기 연산 처리

<aside>

- 즉, 쓰기 연산은 master에서 모두 처리하지만, 읽기 연산은 slave로 분산 시켜서 병렬로 처리될 수 있는 쿼리 수를 늘림. → 성능이 좋아짐
- 자연 재해 등의 이유로 데이터베이스 서버 가운데 일부가 파괴 되어도 데이터가 보존됨.
    - 지역적으로 떨어진 여러 곳에 다중화 시켜놓을 수 있기 때문
- 데이터를 여러 지역에 복제해뒀으므로, 하나의 데이터베이스에 장애가 생겨도 다른 서버에 있는 데이터를 가져와서 계속 서비스 지속가능.
</aside>

    # 데이터베이스 다중화 (Replication)
    
    데이터베이스 서버를 여러 대 두고, 데이터를 복제(replication)해서 운영하는 방식. 크게 **단일 마스터(Master–Slave)** 와 **멀티 마스터(Multi-Master)** 로 나뉜다.
    
    ---
    
    ## 1) 단일 마스터 방식 (Master–Slave)
    
    - 쓰기 연산 → **master 한 곳에서만** 처리
    - slave → master의 사본을 전달받고 **읽기 연산**을 처리
    - 즉, 쓰기는 master가 전담하고 읽기는 slave로 분산시켜 병렬 처리되는 쿼리 수를 늘림 → **성능 향상**
    
    **장점**
    
    - 읽기 부하 분산으로 성능이 좋아짐
    - 자연재해 등으로 서버 일부가 파괴돼도 데이터 보존 (지역적으로 떨어진 여러 곳에 다중화 가능)
    - 한 DB에 장애가 생겨도 다른 서버 데이터로 서비스 지속 가능
    
    **한계**
    
    - 쓰기 주체가 하나라 **쓰기 확장(write scaling)이 안 됨** → master가 병목
    - 대신 쓰기 주체가 하나뿐이라 **데이터 일관성(consistency) 유지가 쉬움**
    
    ---
    
    ## 2) 멀티 마스터 방식 (Multi-Master)
    
    - 여러 노드를 두고 **그룹의 어떤 멤버든 쓰기 가능**. 모든 멤버가 읽기 요청에도 응답한다.
    - 한 노드에서 일어난 변경을 나머지 노드로 **전파(propagate)** 하고, 동시에 발생한 변경끼리의 **충돌(conflict)을 해결**하는 것이 시스템의 핵심 역할.
    - 단일 마스터와 대비: 단일 마스터는 일관성이 쉽지만 유연성이 떨어지고, 멀티 마스터는 유연하지만 충돌·일관성 관리가 어렵다.
    
    **주 목적 (왜 쓰는가)**
    
    | 항목 | 내용 |
    | --- | --- |
    | 가용성 (Availability) | 마스터 하나가 죽어도 다른 마스터들이 계속 쓰기를 처리 |
    | 분산 접근 (Distributed access) | 여러 물리적 지역에 마스터를 배치 가능 |
    | 쓰기 확장 | 쓰기 주체가 여러 개 → 단일 마스터의 쓰기 병목 해소 |
    
    **대가 (trade-off)**
    
    - 대부분 **느슨한 일관성(loosely consistent)**, 즉 지연(lazy)·비동기(asynchronous) 방식이라 **ACID 속성을 위배**한다.
        - ACID = Atomicity(원자성), Consistency(일관성), Isolation(격리성), Durability(지속성)
    - 즉시 반영(eager) 방식은 구조가 복잡하고 통신 지연(latency)이 늘어난다.
    - 노드 수가 늘고 지연이 커질수록 충돌 해결 같은 **무결성(integrity) 문제**가 다루기 어려워진다.
    
    > 대표 구현 예: Active Directory(LDAP), Amazon Aurora Multi-Master, Apache CouchDB, MySQL Group Replication, Galera Cluster 등
    > 
    
    ---
    
    ## 3) MySQL NDB Cluster — 양방향 / 순환 복제
    
    > NDB = Network DataBase (MySQL의 클러스터용 스토리지 엔진)
    > 
    
    ### ① 양방향 / 순환 복제 (Bidirectional / Circular)
    
    - 두 클러스터 사이의 양방향 복제, 그리고 여러 클러스터를 고리로 잇는 순환 복제가 가능.
    - 예시: 클러스터 1 → 2, 2 → 3, 3 → 1 순으로 서로 source(원본) 역할을 한다.
    - **핵심 조건**: source이면서 동시에 replica 역할을 하는 모든 SQL 노드는 `log_slave_updates` 변수를 켜야 한다.
        - `log_slave_updates`: 복제로 받은 변경을 자신의 binary log에도 다시 기록하게 하는 옵션. 켜져야 변경이 다음 노드로 계속 전파됨.
    
    ### ② 다중 소스 복제 + 장애 조치 (Multi-source & Failover)
    
    - 한 클러스터가 여러 클러스터로 동시에 복제 가능. 예: 클러스터 1이 2와 3 모두에게 복제, 클러스터 2도 3에게 복제.
    - 이 경우 데이터가 클러스터 1 → 3으로 **두 경로**(직접 / 클러스터 2 경유)로 전달된다.
    - 장애 조치의 핵심 문제: 클러스터 간 복제가 **비동기**라서, 직접 받은 업데이트가 경유해서 받은 것보다 최신이라는 보장이 없다.
    → 클러스터 1이 죽으면, 클러스터 3을 클러스터 2와 동기화시켜 누락된 업데이트를 따라잡게 해야 한다.
    
    ---
    
    ## 한 줄 정리
    
    **단일 마스터(읽기 분산·일관성 쉬움, 쓰기 확장 불가) → 멀티 마스터(쓰기 확장·고가용성, 대신 충돌·일관성 관리 어려움) → MySQL의 실제 구현(양방향·순환·다중소스)**

### 캐시

> 값비싼 연산 결과 또는 자주 참조되는 데이터를 메모리 안에 두고, 뒤이은 요청이 보다 빨리 처리될 수 있도록 하는 저장소
> 
- 애플리케이션의 성능은 데이터베이스를 얼마나 자주 호출하느냐에 크게 좌우됨(디스크 I/O)
    
    → 이를 해결하기 위해 캐시 사용.
    

#### 캐시 계층

- 캐시 계층(cache tier)는 데이터가 잠시 보관되는 곳으로 데이터베이스보다 훨씬 빠름. (메모리)
    - 별도의 캐시 계층 → 데이터베이스의 부하를 줄여줆. 캐시 규모 독립 확장 가능

<aside>

1. 요청을 받은 웹서버가 캐시에 응답이 저장되어 있는지 확인

→ 2.1. 있으면 바로 반환

→ 2.2. 없으면 데이터베이스 질의를 통해 데이터를 찾아 캐시에 전환한 뒤 클라이언트 반환

</aside>

캐싱 전략 (Caching Strategies)

#### 캐시 사용 시 유의할 점

- 캐시는 어떤 상황에 바람직한가?
    
    데이터 갱신은 자주 일어나지 않지만, 참조는 빈번한 경우
    
- 어떤 데이터를 캐시에 두어야 하는가?
    
    캐시는 데이터 휘발성 메모리에 두므로, 영속적으로 보관할 캐시는 지속적 저장소에 두어야함.
    
- 캐시에 보관된 데이터는 어떻게 만료되는가?
    
    만료 정책이 너무 짧으면 DB를 너무 많이 읽어야하고,
    반대로 너무 길면 DB 원본과 차이가나서 정합성 문제가 생김
    
- 일관성은 어떻게 유지되는가?
    
    일관성 = 데이터 저장소의 원본과 캐시 내의 사본이 같은지 여부
    
    원본과 캐시 갱신을 단일 트랜잭션 내에서 처리하지 않으면 일관성이 깨짐.
    → 여러 지역에 걸쳐 시스템을 확장해 나가는 경우 이 일관성 유지가 어려워짐.
    
    페이스북 캐시 일관성 유지 방법
    
- 장애에는 어떻게 대처할것인가?
    
    캐시 서버 한대 → SPOF가 될 수 있음
    SPOF 예방 → 서버를 어려 지역에 걸쳐 분산
    
- 캐시 메모리는 얼마나 크게 잡을 것인가?
    
    너무 작으면 데이터가 너무 자주 캐시에서 밀려나므로 성능이 떨어짐.
    → 캐시 메모리를 과할당하면 해결 가능 (but, 비용문제)
    
- 데이터 방출 정책은 무엇인가?
    
    LRU - 마지막으로 사용된 시점이 가장 오래된 데이터 방출
    LFU - 사용 빈도가 가정 낮은 데이터를 방출
    FIFO - 가장 먼저 캐시에 들어온 데이터를 가장 먼저 방출

동적 콘텐츠도 캐싱 가능 → 요청 경로, 질의 문자열, 쿠키, 요청 헤더 등의 정보에 기반해서 HTML 페이지를 캐싱

CDN 동적 콘텐츠 전송 (Amazon CloudFront)

6. 만료되지 않은 이미지에 대한 요청은 캐시를 통해 처리됨.

> 즉, 정적파일을 캐싱해두는 서버

### 무상태(stateless) 웹 계층

→ 웹 계층을 수평적으로 확장하고 싶을 때 필요함.

- 상태 정보 → 사용자 세션 데이터 같은 정보임
    - 이 상태정보를 RDB나 NoSQL 같은 저장소에 보관하고 필요할때 가져와 쓰도록 만드는게 바람직

#### 상태 정보 의존적인 아키텍처

→ 클라이언트 정보를 보관하여 상태를 유지하며, 요청들 사이에 공유해서 사용함.

- 즉, 같은 클라이언트 요청은 항상 같은 서버로 전송되어야함.
- 로드밸런서가 이를 지원하기 위해서 고정 세션 기능을 제공
    - 로드밸런서 부담스러움 (서버 추가 제거 까다로움 + 서버 장애 처리 복잡)

#### 무상태 아키텍처

- HTTP 요청이 어떤 웹 서버로도 전달 될 수 있음.
- 상태 정보가 필요할 경우 공유 저장소로부터 데이터를 가져옴.
    - 상태 정보가 웹 서버로부터 물리적으로 분리됨.
    
    > 단순, 안정적, 규모 확장 easy
    >

### 메시지 큐

- 메시지의 무손실을 보장하는, 비동기 통신을 지원하는 컴포넌트!
- 메시지의 버퍼 역할, 비동기적으로 전송

> 기본 아키텍처 → 생산자/발행자 + 소비자/구독자
> 
- 메시지 큐를 이용하면 서비스 또는 서버 간 결합이 느슨해짐.
- 규모 확장성이 보장가능
    - 생산자는 소비자가 다운되어도 발행가능
    - 소비자는 생산자가 다운되어도 메시지 수신 가능

### 로그, 메트릭 그리고 자동화

- 웹 사이트 규모가 커지면 필수적

#### 로그

- 에러 로그 모니터링 → 시스템의 오류와 문제들을 보다 쉽게 찾아냄.
- 서버 단위 or 여러 서버 로그를 단일 서비스로 모아주는 도구 활용

#### 메트릭

- 사업 현황에 관한 유용한 정보 추출, 시스템의 현재 상황 쉽게 파악
    - 호스트 단위 메트릭 → cpu, 메모리, 디스크 I/O에 관한 메트릭
    - 종합 메트릭 → 데이터베이스 계층의 성능, 캐시 계층 성능
    - 핵심 비즈니스 메트릭 → 일별 능동 사용자, 수익, 재방문

#### 자동화

- CI/CD 등
- 빌드, 테스트, 배포 자동화 → 개발 생산성 크게 향상

### 데이터베이스의 규모 확장

- 규모 확장 방법 → 수직 / 수평

#### 수직적 확장

- 기존 서버에 더많은 고성능 자원을 증설 - cpu, ram, 디스크 등
    - 아마존 RDS는 24TB RAM을 갖추었음 ㄷㄷ
    - 스택 오버플로도 한 해동안 천만 명의 사용자를 전부 한 대의 마스터 데이터베이스로 처리함
- 하지만 하드웨어에 한계가 있으므로 CPU, RAM 무한 증설 불가능
- SPOF 위험
- 비용 많이 듬.

#### 수평적 확장

- 샤딩! → 더 많은 서버를 추가함으로써 성능을 향상
- 대규모 데이터베이스를 샤드라고 부르는 작은 단위로 분할
    - 같은 스키마를 쓰지만 보관되는 데이터에는 중복이 없음
    - 어느 샤드에 넣을지는 사용자 ID 사용
        - user_id%4
- 샤딩키(=파티션키)를 어떻게 정하느냐가 가장 중요함
    - 가장 데이터를 고르게 분할 할 수 있는 키를 선택 하는것이 중요.

<aside>

샤딩을 도입하면 생기는 문제

1. 데이터의 재 샤딩
    1. 데이터가 너무 많아져서 하나의 샤드로 감당 어려울 때
    2. 샤드 간 데이터 분포가 균등하지 못하여 어떤 샤드에 할당된 공간 소모가 다른 샤드에 비해 빨리 진행될 때. (샤드 소진)
    
    ⇒ 샤드 키를 계산하는 함수를 변경하고 데이터를 재배치 해야함. 
    
2. 유명인사 문제 (=핫스팟 키 문제)
    1. 특정 샤드에 질의가 집중되어 서버 과부하가 걸리는 문제
    2. 케이티 페리, 저스틴 비버 등 유명인사가 전부 같은 샤드에 저장되어있는 경우 등…
3. 조인과 비정규화
    1. 일단 여러 샤드로 쪼개고 나면, 여러 샤드에 걸친 데이터를 조인하기 힘들어진다. 
    ⇒ 데이터 베이스를 비정규화하여 하나의 테이블에서 질의가 수행될수있도록 해결
</aside>

### 백만 사용자, 그리고 그 이상

1. 웹 계층은 무상태 계층으로
2. 모든 계층에 다중화 도입
3. 가능한 많은 데이터를 캐시할 것
4. 여러 데이터 센터를 지원할 것
5. 정적 콘텐츠는 CDN을 통해 서비스할 것
6. 데이터 계층은 샤딩을 통해 그 규모를 확장할 것
7. 각 계층은 독립적 서비스로 분할할 것
8. 시스템을 지속적으로 모니터링하고, 자동화 도구들을 활용할 것
