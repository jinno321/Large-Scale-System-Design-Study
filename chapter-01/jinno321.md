# 1명이 사용하는 시스템에서 백만 명이 사용하는 시스템으로

학부 시절 1명이 쓰는 시스템도 벅차했지만, 이제는 어떻게 해야 백만 명이 들어와서 만족하고 돌아갈 수 있는지를 연구하고 싶다는 생각이 들었다. 1명이 사용하는 시스템과 백만 명이 사용하는 시스템이 같을 수는 없다.

### 단일서버

먼저, 단 한대의 서버에서 실행되는 간단한 시스템부터 본다.
1. 클라이언트가 웹사이트에 접속을 하면 이때 DNS 서버에 질의 하여 IP 주소로 변환이 된다.
2. 이 IP 주소를 가지고 웹서버에 요청(HTTP 등)을 보낸다.
3. 이렇게 되면 웹서버는 해당 요청을 보고 정적 응답이나 JSON응답을 보내준다.

여기서 하나하나씩 질문을 거쳐가며 사이즈를 늘려보자.

### 데이터베이스
그렇다면 서버가 꺼지면 어떻게 될까 모든 사용자들의 정보가 날라가고 다시 시작해야한다. 이런것도 치명적일 것이고, 웹 서버를 늘리면 서로 간에 데이터 정합성도 맞지 않는다. 찔끔 생각한게 이정도고 여러여러 이유들을 통합했을 때, 데이터베이스를 놓는 이유는 너무나 자명하다.

#### 관계형 vs 비관계형

사실 어떤 데이터베이스를 사용하는지에 대한 기준은 너무 많아서, 서비스의 특성을 고려하는 것이 가장 중요해 보인다. 
관계형이 대부분 보편적이므로, 비관계형을 언제 쓸지를 생각해보는게 좋아보인다.
- 아주 낮은 응답 지연시간 요구
- 비정형이라 관계형 데이터가 아님 -> 이부분은 내 생각엔 데이터의 스키마가 많이 바뀌는 경우라고 생각해도 좋을 것 같다.(로그, 사용자 행동 분석 등)
- 직렬화하거나 역직렬화 할수 있기만 하면 될때
- 아주 많은 양의 데이터를 저장할 필요가 있을때 

이 외에도 정말 밀리초단위의 요구사항이나 사용량이 기하급수적으로 늘어날 때에는 비관계형을 사용하는 것이 좋아보인다. 데이터베이스의 Scale-up이 이루어진다면 데이터베이스 내부에서도 정합성을 맞추는것이 어려워지고, 조인연산도 늘어나며 트랜잭션의 사용이 이루어지면 데이터베이스의 부하가 늘어나는 것이 보편적인데 이는 대용량 처리에선 곧 '독'이 된다고 생각한다. 


### 규모확장: 수직 vs 수평
- **수직(Scale up)** : 서버의 성능을 업그레이드 시키는것 
- **수평(Scale out)** : 서버 개수를 추가하는 것

수직적 규모확장이 안좋은 것은 아니다. 성능이 간당하게 부족한데, 이를 위해 수평적 규모확장을 하는 것은 추후를 위해서는 좋겠지만 오버스펙일 수도 있다. 그럴때는 수직 확장이 훨씬 좋다.
그러나, 완벽한 대안거리는 되지 않는다. 가성비가 좋은것은 수평 확장이며, 이렇게 되면 서버가 여러대가 되며 이를 위해 로드밸런서의 도입이 최선이다.

#### 로드밸런서 역할
이전에 로드밸런서에 대해서 공부해본 적이 있어서 자세하게는 넘어간다.
쉽게 말하면, 사용자의 요청을 어느 서버로 보내줄 지를 결정하는 도구이다.

### 데이터베이스 다중화
master-slave 데이터베이스 전략을 사용해 주 데이터베이스는 주로 쓰기 연산을, 부 데이터베이스들은 주로 읽기 연산을 수행하도록 하는 것이다. 대부분의 요청은 읽기인것을 생각해보면 효과적으로 보인다. 
- **더 나은 성능** : 읽기 연산들이 부 데이터베이스들로 분산되며 병렬로 처리가 될 수 있으며 성능이 좋아진다.
- **가용성** : 하나의 데이터베이스가 파괴되어도, 다른 데이터베이스들이 그 역할을 도맡아 해줄 수 있어진다.
- **안정성** : 하나의 데이터베이스가 날라가도, 다른 데이터베이스들에 어느정도의 내용들이 보존되어 있다.

#### 장애 상황 대처
- **부 서버가 다운된 경우?** => 한시적으로 master가 모든 역할 수행 하거나, 다른 부 데이터베이스가 역할 대체
- **주 데이터베이스 다운?** => 하나의 부 데이터베이스가 master로 승격 , 이때 부 데이터베이스는 최신의 데이터가 아닐 수 있는데 이를 위해 복구 스크립트를 작성해야한다. 

이렇게 웹 계층과 데이터 계층을 분리하며, 각 계층별로의 scale-out을 알아보았다.

### 캐시

이때까지 짜둔 설계에 의하면 매번 데이터베이스를 방문해야 한다. 그러나, 실제 서비스에서 데이터베이스 까지 가는 시간은 사용자에게 많이 걸릴 것이며, 데이터베이스로의 요청을 줄이는 것이 가장 큰 숙제중 하나이다. 이를 위한 것이 캐시이다.

캐시 계층은 데이터가 잠시 보관되는 곳으로 데이터베이스보다 훨씬 빠르다.

#### 캐시를 언제 사용해야 할까? 
- 데이터 갱신은 자주 안일어나지면 읽기는 많이 일어날 때.
- 영속적이지 않은 데이터들을 다룰 때. -> 캐시는 보통 휘발성이다.

#### 또한 생각할 점도 많다.
- **캐시 데이터의 만료 정책**을 마련해 두어야 한다. -> TTL 설정 정책이 필수적
- **일관성 유지** : 데이터베이스의 내용과 캐시의 정합성이 맞지 않다면 캐시는 오히려 '독'으로 작용한다. 갱신 로직을 단일 트랜잭션으로 처리해 주어야 한다.
  > **Q) 만약, 단일 트랜잭션이 아닐 경우 어떻게 해야할까?**
  > 애플리케이션 계층에서 Cache-Aside(Look-Aside) 패턴을 정교하게 다듬거나, 데이터 변경 시 캐시를 먼저 만료(Evict)시키고 DB를 업데이트하는 방식을 쓴다. 혹은 분산 환경이라면 캐시와 DB의 정합성을 맞추기 위해 메시지 큐(Kafka 등)나 CDC(Change Data Capture)를 활용해 비동기적으로 캐시를 동기화하는 것도 방법이다.
- **SPOF 대처** : 단일 장애 지점이 무너지면, 모든 서비스가 무너진다. 이를 위해 캐시 서버를 분산시키는 것도 필수적이다.
- **캐시 메모리는 얼마나 크게?** : 너무 작으면 캐시의 데이터들이 자주 바뀌어서 별 필요가 없는 느낌이 된다. -> 캐시 메모리 과할당이 필요함.
- **데이터 방출 정책** : 캐시가 꽉찼는데, 캐시에 쓰기 연산을 해야 한다면 하나의 데이터를 추방시켜야 한다. 이때 여러가지 알고리즘이 있는데 LRU가 가장 널리 쓰인다. LFU나 FIFO도 존재함.

#### 🛠️ 인메모리 캐시와 공유 캐시의 차이와 언제 각각 선택해야할까.

| 구분 | 인메모리 캐시 (Local Cache) | 공유 캐시 (Global / Distributed Cache) |
| :--- | :--- | :--- |
| **대표적인 예시** | **Caffeine Cache**, Ehcache, Google Guava Cache 등 | **Redis**, Memcached 등 |
| **개념** | 웹 서버 자체의 메모리(RAM) 공간에 데이터를 저장함. | 웹 서버 외부에 별도의 캐시 서버를 두고 공유함. |
| **속도** | 네트워크를 타지 않으므로 극도로 빠름. | 네트워크 통신(I/O)이 발생하므로 로컬보다는 상대적으로 느림. |
| **정합성** | 서버가 여러 대가 되면 각 서버마다 캐시 데이터가 달라져 정합성이 깨짐. | 모든 서버가 하나의 캐시 서버를 바라보므로 데이터 일관성이 유지됨. |

- **언제 선택해야 할까?**
  - **인메모리 캐시(Caffeine 등):** 단일 서버이거나, 서버 간 데이터가 조금 달라도 상관없는 경우, 혹은 서비스 점검 공지사항처럼 거의 변경이 없는 static한 설정 데이터를 극도로 빠르게 조회해야 할 때 쓰기 좋다.
  - **공유 캐시(Redis 등):** 백만 명이 사용하는 시스템처럼 서버가 여러 대로 scale-out되는 환경에서, 사용자 로그인 세션 정보나 전역적인 유저 잔액 정보처럼 **모든 서버가 무조건 동일한 데이터를 캐시로 조회하고 업데이트해야 할 때** 필수적이다.

### 콘텐츠 전송 네트워크(CDN)
정적 콘텐츠(사진 등)를 캐시할 때 쓰이는 기술이다.
어떤 사용자가 웹사이트를 방문하면, 그 사용자에게 가장 가까운(지리적) CDN 서버가 정적 콘텐츠를 딜리버리 한다. 한국의 CDN을 사용한다면, 서울에서 요청할때가 LA의 사용자보다 빨리 도착할 것이다.

1. 사용자 A가 이미지 URL을 이용해 image.png에 접근한다. -> 이때, URL의 도메인은 CDN 서비스 사업자가 제공
2. CDN 서버의 캐시가 해당 콘텐츠가 없는 경우, 서버는 원본 서버에 요청하여 파일을 가져온다. -> DB일수도 있고, S3와 같은 원격 스토리지 일 수도 있다.
3. 원본 서버가 파일을 CDN서버에 반환하면 캐시하고 사용자에게 반환
4. 이후 요청부터는 해당 캐싱된 이미지를 반환


#### CDN 사용 시 고려해야 할 사항
- **비용** : 외부 업체를 사용할 경우가 많으며, 자주 사용되지 않는 콘텐츠를 캐싱하는 것은 돈낭비에 불과하다. 이런 경우를 잘 체크해야한다.
- **적절한 TTL** : 시의성이 중요할 경우 TTL이 너무 길지도 않고, 짧지도 않아야 한다.
  > **예시:** 실시간으로 스코어가 바뀌는 스포츠 중계의 선수 프로필 이미지나 이벤트 배너는 TTL을 짧게 가져가야 하고, 서비스 로고나 고정된 UI 아이콘 등은 TTL을 길게 잡아도 무방하다.
- **장애 대처** : 갑자기 CDN이 죽으면, 이에만 의존하고 있는 서비스는 이미지를 하나도 렌더링 하지 못할 것이다. 그래서, 이가 장애 났을 경우를 대비하여 원본 콘텐츠를 파싱할 수 있는 로직을 구성하는 것이 좋다.
- **콘텐츠 무효화 방법** : 아직 만료되지 않은 콘텐츠라 하더라도 CDN에서 제거할 수 있어야 한다.
  > **필요성 예시:** 서비스에 중대한 보안 이슈가 있어서 이벤트 이미지를 급하게 내려야 하거나, 전면 교체된 UI 소스코드가 즉시 반영되어야 할 때 무효화가 필요하다.
  > 
  > **방안:** CDN 서비스가 제공하는 API/콘솔을 통해 **Invalidation(무효화) 요청**을 날려 캐시를 강제로 비우거나, 애초에 파일명 뒤에 버전을 붙이는 **콘텐츠 버저닝(e.g., `image_v2.png`, `style.css?v=1.1`)** 방식을 사용해 새로운 자원을 읽어가도록 유도한다.

### 무상태 웹 계층
수평적으로 스케일 아웃을 하기 위해서 가장 중요한 것은 무상태 웹 계층을 가져야 한다는 것이다. 이를 위해 상태 정보를 웹 계층에서 제거해야한다. 서버는 사용자에 대해 편견이 있으면 안된다. 그렇기 위해서 가장 쉬운 것은 데이터베이스나 전역 공유 저장소에 상태정보를 보관하고 필요할때 가져오는 것이다. 서버가 상태를 아예 관리하지 않는 JWT 기반의 AT(Access Token)를 활용해 각 서버가 독립적으로 검증하게 만들고, 보안상 탈취 위험이 있는 RT(Refresh Token)만 Redis 같은 외부의 공유 저장소에 저장해 관리하는 인증 방식이 그 좋은 예다.

상태 정보가 저장되어 있는 서버가 스케일 아웃을 하면 한사용자는 특정 서버에만 접근을 해야한다. A사용자가 A+ 서버로 요청을 보내다가 어느 순간 B+서버에 보내게 된다면, B+ 서버에는 A 사용자의 접근 정보가 없다. 이렇게 되면 새로운 사용자라고 판단을 하고, 뭐 쉽게는 로그아웃이 이루어 질 것이다. 이를 위해서 로드밸런서 알고리즘 중에 해시, 고정 세션(Sticky Session)이라는 기능 등이 제공되고 있는데, 이는 로드밸런서에 부담을 준다.

책(p.19) 에서는 NoSQL을 사용하는데, 규모 확장이 간편해서이다.

### 데이터센터
만약 전 세계적으로 우리의 서비스를 사용하기 위해서는 데이터 센터를 지원해야 한다. 사용자는 지리적으로 가장 가까운 데이터 센터 내부에 있는 서비스들을 사용하게 된다. 지리적 라우팅에서 geoDNS는 사용자의 위치에 따라 도메인 이름을 어떤 IP 주소로 변환할지 결정해준다. 만약 하나의 데이터센터가 화재가 나면, 사용자들의 요청을 다른 데이터센터로 옮길 것이다. 물론, 지리적으로 먼 곳의 데이터센터로 요청이 가면 그만큼 처리속도는 낮지만 가용성이 낮아지는 것보다는 차선의 선택이다.

- **트래픽 우회** : 장애가 발생하지 않은 가장 가까운 데이터 센터로 트래픽을 유도하는 기술이 필요하다.
- **데이터 동기화** : 하나의 데이터 센터가 장애가 나서 다른 데이터센터로 요청이 우회되어도 데이터의 동기화는 이루어져야 한다. 다만 백만 명이 사용하는 대용량 환경에서 물리적으로 멀리 떨어진 데이터 센터 간에 실시간으로 완전 동기화를 맞추는 것은 네트워크 지연 시간 때문에 불가능에 가깝다. 따라서 보통 비동기식 복제 방식을 취하며 최종 일관성(Eventual Consistency)을 만족시키는 전략을 취하게 된다.
- **테스트, 배포** : 애플리케이션 또는 웹을 여러 위치에서 테스트 해보는 것이 중요하다.

### 메시지 큐와 이벤트 기반 아키텍처(EDA)

백만 명이 사용하는 대규모 시스템에서 MSA를 도입하게 되면, 서비스 간의 통신을 어떻게 처리할지가 가장 큰 숙제가 된다. 시스템 초기에는 가볍게 HTTP/REST API를 이용해 동기(Synchronous) 방식으로 서버끼리 요청을 주고받는 것이 일반적이다. 

하지만 동기 방식은 치명적인 단점이 있다. 주문 서비스가 결제 서비스를 호출하고, 결제 서비스가 알림 서비스를 호출하는 구조라면 중간에 서버 하나만 죽어도 전체 요청이 실패하게 된다. 또한 결제 처리가 끝날 때까지 주문 서비스는 아무것도 못 하고 마냥 대기해야 하므로 응답 지연(Latency)이 기하급수적으로 늘어난다. 이러한 문제를 해결하기 위해 메시지 큐를 도입하고, 시스템의 패러다임을 **이벤트 기반 아키텍처(Event-Driven Architecture, EDA)**로 전환하게 된다.

메시지 큐는 기본적으로 발행자(producer)와 소비하는 소비자(consumer)로 이루어진다. 발행자가 메시지를 발행하면 메시지 큐는 소비자가 꺼낼 때까지 안전히 보관하여 이를 소비자가 소비할 수 있도록 도와준다. 이때, 소비자는 어떤 메시지 큐를 읽어들일지 정할 수 있다. 이 구조가 MSA와 EDA에서 핵심적으로 활용되는 이유는 다음과 같다.

- **느슨한 결합(Decoupling)과 비동기 처리** : 이벤트 기반 아키텍처에서 각 마이크로서비스는 서로를 직접 호출하지 않는다. 시스템에서 일어나는 중요한 변화(예: 주문 완료, 회원가입 등)를 하나의 '이벤트(Event)'로 정의하고, 이를 메시지 큐에 발행할 뿐이다. 주문 서비스는 주문이 들어오면 "주문 완료" 이벤트를 메시지 큐에 던지고 자신의 할 일을 끝낸다. 결제나 알림 서비스는 큐를 구독하고 있다가 비동기적으로 이를 가져가 처리하므로, 서버 간 결합도가 느슨해지고 사용자 응답 속도가 극도로 빨라진다. 서버 하나가 다운되어도 전체 서비스가 터지지 않는 고가용성의 기반이 된다.
- **트래픽 폭주에 대한 완충 장치** : 선착순 이벤트나 수만 건의 결제 요청이 기하급수적으로 몰릴 때, 동기식 API 통신은 부하를 견디지 못하고 서버가 뻗어버린다. 이때 메시지 큐는 중간에서 거대한 댐 역할을 해준다. 발행자가 초당 수만 건의 이벤트를 밀어 넣더라도 큐가 이를 안전하게 버퍼링(Buffering)해주고, 뒤에 있는 마이크로서비스들은 자신들이 처리할 수 있는 한계량만큼만 조절해가며 안정적으로 메시지를 소비하기 때문에 시스템 붕괴를 막을 수 있다.
- **데이터 일관성 해결(Saga 패턴)** : 데이터베이스를 여러 샤드로 쪼개거나 분산된 MSA 환경에서는 하나의 거대한 트랜잭션으로 묶는 조인 연산이 대용량 처리에 큰 독이 된다. EDA를 활용하면 주문 완료 이벤트가 결제를 일으키고, 결제 완료 이벤트가 배송을 일으키는 식으로 메시지 큐를 통해 이벤트를 체이닝하면서 점진적으로 데이터의 정합성을 맞춰나갈 수 "있다. 분산 환경에서 트랜잭션의 무거움을 덜어내기 위한 최선의 선택이라고 생각한다.

### 로그, 메트릭, 자동화
웹사이트의 규모가 커지게 되면 로그, 메트릭, 자동화 같은 도구에 필수적인 투자가 필요하다. 

- **로그** : 시스템의 오류 and 문제들을 보다 쉽게 찾아낼 수 있도록 해준다. 서버 단위로 모니터링 할 수도 있지만, 단일 서비스로 모아주는 도구를 활용하면 더 좋다.(datadog)
- **메트릭** : 사업 현황에 관한 유용한 정보와 인프라적 요소에 대한 정보를 제공받을 수 있다.
  - **호스트 단위 메트릭** : CPU, 메모리, 디스크 I/O 관한 메트릭
  - **종합 메트릭** : 데이터베이스 계층의 성능, 캐시 계층의 성능
  - **핵심 비즈니스 메트릭** : 일별 능동 사용자(DAU), 수익, 재방문 
- **자동화** : 시스템이 크고 복잡해지면 생산성을 높이기 위해 CI/CD와 같은 자동화 도구를 활용해야 한다. (github action, jenkins 등)

### 데이터베이스의 규모 확장 
데이터의 크기가 증가하면 데이터베이스에 대한 부하도 증가한다. 데이터베이스를 증설할 방법을 찾아야 한다.
이때도, 수직적 확장(=스케일 업)과 수평적 확장(=스케일 아웃)으로 나눌 수 있다.

- **수직적 확장** : 데이터베이스 서버에 고성능의 자원을 증설하는 방법이다. 아마존 AWS의 RDS는 이런 고성능의 제품을 제공해주고 있다. 예를 들어 스택오버플로는 옛날에 천만명의 사용자 전부를 단 한대의 마스터 데이터베이스로 처리했다. 하지만, 이러한 점에는 약점이 있다.
  - 데이터베이스 서버 하드웨어는 한계가 있어 무한 증설할 수가 없다.
  - SPOF(단일 장애 지점)에 대한 위험성이 증가한다.
  - 비용이 많이 든다.

- **수평적 확장(샤딩)** : 대규모 데이터베이스를 샤드라고 부르는 작은 단위로 분할한다. 샤딩에 대해 공부할때 가장 많이 쓰는 예시는 user_id에 대한 해시값으로 데이터베이스를 분리하는 것이다. user_id % 4를 하여 결과값으로 몇번 샤드에 데이터를 저장할지를 정한다. 이를 통해 우리는 수직적 확장이 제한되어 있는 데이터베이스를 효과적으로 확장시킬 수 있다.

샤딩에서 가장 중요한 점은 파티션 키(샤드 키)의 선택이다. 어떻게 분산될지 정하는 하나 이상의 칼럼으로 구성이 된다. 샤딩을 도입하면서 새로운 문제들이 많이 생긴다.

- **데이터의 재 샤딩** : 데이터가 너무 많아져서 하나의 샤드로 감당하기 어려울 때나, 샤드 간 데이터 분포가 균등하지 못하여 어떤 샤드에 할당된 공간 소모가 다른 샤드에 비해 빨리 진행될 때, 샤드 소진이 발생하면 샤드 키를 계산하는 함수를 변경하고 데이터를 재배치 하여야 한다.
- **유명인사 문제** : 특정 샤드에 질의가 집중되어 서버에 과부하가 걸리는 핫스팟 현상 문제를 말한다. 예를 들어 인스타그램이나 트위터 같은 대규모 서비스에서 샤드 키를 '유저 ID'로 잡았다고 가정해보자. 일반 유저들의 샤드는 트래픽이 완만하겠지만, 만약 손흥민이나 BTS 같은 초유명 셀럽의 데이터가 저장된 특정 샤드가 있다면 그 셀럽이 글을 올리는 순간 수백만 명의 팔로워가 동시에 해당 샤드로만 접근하게 된다. 결과적으로 다른 샤드 서버들은 널널한데, 그 셀럽이 속한 특정 샤드 하나만 트래픽 폭주로 마비되는 현상이 발생하게 된다. 이를 해결하기 위해서는 유명인사 데이터에 대해서만 더 잘게 쪼개진 샤드 키를 부여(데이터 ID 뒤에 랜덤 비트를 추가하는 등)하여 분산시키거나, 해당 셀럽 데이터만 따로 전역 캐시 계층에서 읽어가도록 처리하는 특별한 읽기 분산 전략을 세워야 대용량 처리가 원활해진다고 생각한다.
- **조인과 비정규화** : 일단 하나의 데이터 베이스를 여러 샤드 서버로 쪼개고 나면, 여러 샤드에 걸친 데이터를 조인하기가 힘들어진다. 이를 해결하기 위해서는 NoSQL을 적절히 활용하거나, 데이터베이스의 적절한 비정규화가 필요하다.
