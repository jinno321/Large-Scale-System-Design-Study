# 분산 시스템을 위한 유일 ID 생성기 설계

## 왜 관계형 DB의 auto_increment를 쓰면 안 되나

단일 서버라면 관계형 데이터베이스의 기본 키(auto_increment)로 유일 ID를 만들면 그만이다. 하지만 분산 환경에서는 이 접근법이 통하지 않는다.

- 데이터베이스 서버 한 대로는 대규모 ID 생성 요구를 감당할 수 없다.
- 여러 데이터베이스 서버를 쓰면, 서버 간에 번호가 겹치지 않도록 조율해야 하는데 이 과정에서 지연시간을 낮추기가 무척 힘들다.

## 요구사항 정의

- ID는 유일(unique)하고 정렬 가능(sortable)해야 한다.
- 시간의 흐름에 따라 ID 값은 커지지만, 정확히 1씩 증가한다고는 할 수 없다.
- 숫자로만 구성된다.
- 초당 10,000개의 ID를 생성할 수 있어야 한다.

## 접근법 1: 다중 마스터 복제(Multi-master Replication)

### 설계

데이터베이스의 auto_increment를 활용하되, 다음 ID를 구할 때 1씩 증가시키는 대신 **데이터베이스 서버 수(k)만큼 증가**시킨다.

```
서버가 2대라면 (k=2):
서버1: 1, 3, 5, 7, ... (홀수)
서버2: 2, 4, 6, 8, ... (짝수)
→ 번호가 절대 겹치지 않음
```

### 장점

- 데이터베이스의 기존 auto_increment 메커니즘을 그대로 활용하므로 구현이 비교적 단순하다.
- 여러 서버가 병렬로 ID를 생성하므로 초당 처리량을 늘릴 수 있다.

### 단점

- **여러 데이터 센터로 확장하기 어렵다**: 서버가 동적으로 추가/삭제되면 증가 폭(k)을 다시 조정해야 한다.
- **시간순 정렬이 보장되지 않는다**: 여러 서버에 걸쳐 보면 ID가 시간에 따라 커진다는 보장이 깨진다.

## 접근법 2: UUID(Universally Unique Identifier)

### 설계

UUID는 128비트짜리 유일 식별자다. 각 서버가 서로 조율 없이 독립적으로 UUID를 생성한다.

```
예: 09c93e62-50b4-468d-bf8a-c07e1040bfb2
```

### 장점

- 서버 간 조율이 전혀 필요 없어 동기화 병목이 없다.
- 서버를 추가해도 서로 협의할 필요가 없으니 확장이 자유롭다.

### 단점

- **128비트로 길다**: 요구사항인 "숫자로만 구성"을 만족하지 못한다.
- **시간순으로 정렬되지 않는다**: 랜덤성에 기반하므로 ID만 보고 생성 순서를 알 수 없다.

## 접근법 3: 티켓 서버(Ticket Server)

### 설계

ID 생성만 전담하는 **단일 데이터베이스 서버(티켓 서버)**를 하나 두고, 모든 애플리케이션 서버가 이 서버에서 auto_increment로 다음 ID를 받아간다. (플리커(Flickr)가 사용한 방식으로 알려져 있다.)

```
앱 서버1 ─┐
앱 서버2 ─┼──→ 티켓 서버 (auto_increment) → 유일한 순번 발급
앱 서버3 ─┘
```

### 장점

- 숫자로만 된 ID를 만들 수 있다.
- 구현이 단순하고, 중소 규모 애플리케이션에 적합하다.

### 단점

- **티켓 서버가 단일 장애점(SPOF)이 된다**: 이 서버가 죽으면 전체 시스템의 ID 생성이 멈춘다.
- 티켓 서버를 여러 대 두면 서버 간 데이터 동기화 문제가 다시 생긴다.

## 접근법 4: 트위터 스노플레이크(Snowflake)

### 설계

트위터가 만든 방식으로, 64비트 ID를 여러 구획(section)으로 나눠 각 구획이 서로 다른 의미를 갖게 한다.

```
| 1비트 |    41비트     | 5비트     | 5비트  |   12비트   |
| 부호  |  타임스탬프    | 데이터센터 | 머신   |  일련번호   |
|      |              | ID        | ID    |            |
```

- **부호 비트(1비트)**: 항상 0. 미래에 음수/양수 구분이 필요할 때를 위해 예약.
- **타임스탬프(41비트)**: 기준 시각(epoch)으로부터 흐른 밀리초. 상위 비트를 차지하기 때문에 ID를 시간순으로 정렬 가능하게 만드는 핵심 구획.
- **데이터 센터 ID(5비트)**: 2^5 = 32개의 데이터 센터를 구분.
- **머신 ID(5비트)**: 데이터 센터당 2^5 = 32개의 머신을 구분.
- **일련번호(12비트)**: 같은 밀리초 안에서 같은 머신이 여러 ID를 만들 때 구분. 매 밀리초마다 0으로 리셋되며 최대 4096개(2^12).

### 장점

- **시간순 정렬이 가능하다**: 타임스탬프가 상위 비트라 나중에 만든 ID가 항상 더 큰 값.
- 숫자로만 구성된다.
- 데이터 센터 ID와 머신 ID가 서버마다 다르므로 서버 간 통신 없이도 ID가 겹치지 않는다.
- 각 구획의 비트 길이를 요구사항에 맞게 조정 가능하다.

### 단점

- **시계 동기화에 의존한다**: 각 서버의 시계가 어긋나면 문제가 생길 수 있다.
- 구현이 앞의 방식들보다 복잡하다.

## 상세 설계: 스노플레이크 각 구획 이해

### 타임스탬프(41비트)가 왜 핵심인가

타임스탬프가 상위 비트에 위치하기 때문에, ID를 숫자로 비교하면 곧 시간순 비교가 된다.

41비트로 표현할 수 있는 밀리초의 범위는 약 69년이다.

```
2^41 밀리초 ≈ 69년
```

보통 기준 시각(epoch)을 서비스 시작 시점으로 잡아서, 그때부터 약 69년간 유일 ID를 생성할 수 있다.

### 일련번호(12비트)의 역할

같은 머신이 **같은 밀리초 안에** 여러 ID를 생성하는 경우, 타임스탬프·데이터센터·머신 구획이 전부 동일해진다. 이때 ID가 겹치지 않도록 구분해주는 게 일련번호다.

```
같은 밀리초, 같은 머신에서:
1번째 ID → 일련번호 0
2번째 ID → 일련번호 1
...
4096번째 ID → 일련번호 4095 (12비트 최대)
밀리초가 바뀌면 → 일련번호 다시 0으로 리셋
```

12비트라 한 밀리초에 한 머신당 최대 4096개의 ID를 만들 수 있다. 이를 넘으면 다음 밀리초까지 대기한다.

## 마무리: 추가로 고려할 점

### 시계 동기화(Clock Synchronization)

스노플레이크는 여러 서버가 같은 시계를 사용한다고 가정하지만, 서버마다 시계가 미세하게 다르게 흐를 수 있다. **NTP(Network Time Protocol)**가 가장 보편적인 수단으로, 여러 서버의 시계를 표준 시간에 맞춰 주기적으로 동기화해준다.

### 각 구획의 길이 최적화

앞서 제시한 비트 배분(41/5/5/12)은 고정된 정답이 아니다.

- 동시성이 낮고 수명이 긴 시스템이라면 일련번호 비트를 줄이고 타임스탬프 비트를 늘린다.
- 초당 생성량이 매우 많다면 일련번호 비트를 늘린다.

### 고가용성(High Availability)

ID 생성기는 필수 컴포넌트이므로 아주 높은 가용성을 제공해야 한다. ID 생성이 멈추면 그에 의존하는 모든 서비스가 영향을 받기 때문에, 생성기 자체를 이중화하고 장애에 견디도록 설계해야 한다.

## 4가지 접근법 비교 요약

| 접근법 | 숫자만 | 시간순 정렬 | 분산 조율 불필요 | 주요 단점 |
|---|---|---|---|---|
| 다중 마스터 복제 | O | X | △ | 확장 시 증가폭 조정 복잡, 전역 정렬 불가 |
| UUID | X | X | O | 128비트로 길고 정렬 불가 |
| 티켓 서버 | O | O | X | 단일 장애점(SPOF) |
| 스노플레이크 | O | O | O | 시계 동기화 의존, 구현 복잡 |

요구사항(숫자, 정렬 가능, 초당 1만 개, 분산 환경)을 모두 만족하는 건 스노플레이크 접근법이다. 실무의 분산 ID 생성기는 대부분 스노플레이크를 변형해서 쓴다.
