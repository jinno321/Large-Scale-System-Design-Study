# 분산 시스템에서의 유일 ID 생성기 설계

유일 ID라고 했을 때 가장 먼저 떠오르는 것은 `AUTO_INCREMENT` 방식이다. 하지만 이 방식은 대규모 시스템이나 분산 환경에서 다음과 같은 명확한 단점을 가진다.

## 1. 기존 AUTO_INCREMENT 방식의 한계
1. **내부 데이터 규모 노출:** 숫자가 순차적으로 일정하게 증가하기 때문에, URL이나 API 요청을 통해 외부에서 내부 데이터의 규모를 쉽게 추측할 수 있다.
2. **분산 환경(Sharding)의 걸림돌:** 서비스가 커져 데이터베이스를 여러 대(Sharding)로 분산할 때, ID 중복을 막기가 매우 어려워진다.
3. **동시성 및 병목 현상:** 여러 트랜잭션이 동시에 `INSERT`를 시도할 때, ID 중복을 방지하기 위해 데이터베이스 락(Lock)을 걸게 되며 이로 인해 병목 현상이 발생한다.

---

## 2. 해결을 위한 대안 기술들

### ① 다중 마스터 복제 (Multi-Master Replication)
`AUTO_INCREMENT` 방식을 활용하되, 데이터베이스의 개수($k$)만큼 ID를 건너뛰며 증가시키는 방식이다.
- **예시 (DB가 3개인 경우):**
  - DB A : 1, 4, 7, ...
  - DB B : 2, 5, 8, ...
  - DB C : 3, 6, 9, ...

- **장점:** 데이터베이스의 수만큼 생성 성능을 나누어 가질 수 있어 규모 확장성 문제를 어느 정도 해결한다.
- **단점:**
  - 여러 데이터 센터에 걸쳐 규모를 늘리기 어렵다.
  - ID의 유일성은 보장되지만, 시간 흐름에 맞추어 값이 커지도록 보장할 수 없다. (발급 날짜 순 정렬 불가능)
  - **서버 추가/삭제 시 문제:** 서버가 추가되거나 삭제되면 ID의 증가 폭($k$)과 시작 값을 전체 데이터베이스에 걸쳐 다시 재설정하고 동기화해야 하기 때문에 유연한 확장이 어렵다.

### ② UUID (Universally Unique Identifier)
컴퓨터 시스템에 저장되는 정보를 식별하기 위한 128비트짜리 수이다. 중복될 확률이 천문학적으로 희박하여, 각 웹 서버가 독립적으로 ID를 생성할 수 있다.

- **장점:**
  - 구현이 단순하며, 서버 간의 동기화 이슈를 신경 쓸 필요가 없다.
  - 각 서버가 알아서 ID를 생성하므로 규모 확장이 매우 쉽다.
- **단점:**
  - ID가 128비트로 길어서 데이터베이스 인덱스 크기가 커지고 성능이 저하될 수 있다.
  - 시간순으로 정렬할 수 없다.
  - ID에 숫자가 아닌 문자(진수 표현 등)가 포함된다.

### ③ 티켓 서버 (Ticket Server)
Flickr(플리커)에서 분산 기본 키를 생성하기 위해 사용한 기술로, ID 생성만을 담당하는 전용 데이터베이스 서버(티켓 서버)를 하나 두고 모든 웹 서버가 여기서 ID를 받아오는 방식이다.

- **장점:** 구현하기가 비교적 쉽고, 중복 없는 순차적인 숫자를 보장받을 수 있다.
- **단점:**
  - **SPOF (Single Point of Failure) 위험:** 티켓 서버 자체가 단일 장애점이 된다. 이 서버에 장애가 발생하면 전체 시스템의 쓰기(Insert) 작업이 모두 마비된다. (이를 해결하기 위해 티켓 서버를 여러 대 두면 다시 동기화 문제가 발생한다.)

## 3. 트위터 스노플레이크(Snowflake) 접근법
트위터에서 개발한 독창적인 ID 생성 방식으로, **분할 정복(Divide and Conquer)** 기술을 사용하여 문제를 해결한다. 생성할 64비트 ID의 구조를 여러 섹션으로 분할하여 고유성을 보장한다.

### ID 구조 분할 (총 64비트)
- **사인(Sign) 비트 (1비트):** 음수와 양수를 구분 (보통 0으로 고정).
- **타임스탬프 (41비트):** 기원 시각(Epoch) 이후로 몇 밀리초(ms)가 경과했는지를 나타낸다. 이를 통해 **'시간에 따라 ID가 커진다'**는 요구사항을 완벽히 충족한다.
- **데이터센터 ID (5비트):** $2^5 = 32$개의 데이터센터를 지원할 수 있다.
- **서버 ID (5비트):** 데이터센터당 32개의 서버를 지원할 수 있다.
- **일련번호 (12비트):** 각 서버에서 동일한 밀리초 내에 ID를 생성할 때마다 1씩 증가시키며, 1밀리초가 경과하면 다시 0으로 초기화한다.

### 특징 및 장점
- 데이터센터 ID와 서버 ID는 시스템 구축 시 고정되므로, 분산 환경에서도 서버 간에 ID가 겹칠 일이 없다.
- 41비트의 타임스탬프 덕분에 시간별 정렬이 기본적으로 가능하다.
- **유동적인 아키텍처 설계 가능:** 만약 동시 처리량이 낮고 수명이 긴 애플리케이션이라면, 일련번호 비트를 줄이고 타임스탬프 비트를 늘리는 등 비트 레이아웃을 서비스 특성에 맞춰 효과적으로 변형하여 사용할 수 있다.
