# 알림 시스템(Notification System) 설계

## 알림 시스템이란

사용자에게 중요한 정보를 알려주는 시스템. 뉴스 속보, 상품 업데이트, 이벤트, 배송 알림 등이 해당한다. 오늘날 많은 앱에서 없어서는 안 될 기능이 되었다.

알림은 크게 세 가지 형태가 있다.

- 모바일 푸시 알림(iOS/Android)
- SMS 메시지
- 이메일

단순히 "메시지를 보낸다"로 보이지만, 하루 수백만~수천만 건 규모가 되면 여러 제3자 서비스 연동, 안정성, 중복 방지, 사용자 설정 존중 등 고려할 것이 많아진다.

## 문제 이해 및 설계 범위 확정

질문을 통해 요구사항을 확정한다.

- 어떤 종류의 알림을 지원하나? 푸시 알림, SMS 메시지, 이메일
- 실시간 시스템이어야 하나? **연성 실시간(soft real-time)** 시스템. 가능한 한 빨리 전달하되, 시스템 부하가 높을 때 약간의 지연은 허용한다.
- 지원해야 하는 단말은? iOS, 안드로이드, 랩톱/데스크톱
- 알림은 무엇이 만드나? 클라이언트 앱이 만들 수도 있고, 서버 측에서 스케줄링할 수도 있다.
- 사용자가 알림을 받지 않도록 설정할 수 있어야 하나? 그렇다. 옵트아웃(opt-out)한 사용자는 알림을 받지 않는다.
- 하루에 몇 건을 보내나? 천만 건의 모바일 푸시, 백만 건의 SMS, 5백만 건의 이메일

> **연성 실시간(soft real-time)이 무슨 뜻인가?** 마감 시한을 넘겨도 시스템이 실패로 간주되지는 않고, 다만 가치가 떨어지는 것을 말한다. 알림이 3초 늦게 도착해도 여전히 유용하다. 반대로 **경성 실시간(hard real-time)** 은 시한을 놓치면 곧 실패다(예: 항공기 제어, 자동차 에어백). 알림 시스템은 연성 실시간이므로, 부하가 몰릴 때 큐에 쌓아두고 조금 늦게 보내는 설계가 허용된다. 이 점이 뒤에 나오는 메시지 큐 도입의 근거가 된다.

## 알림 유형별 지원 방안

각 알림 유형이 어떻게 사용자 단말까지 도달하는지 살펴본다.

### iOS 푸시 알림

세 가지 컴포넌트가 필요하다.

```
알림 제공자(Provider) → APNS → iOS 단말
```

- **알림 제공자(Provider)**: 알림 요청을 만들어 APNS로 보내는 주체. 우리가 만들 서버다. 요청에는 두 가지가 필요하다.
  - **단말 토큰(device token)**: 알림 요청을 보내는 데 필요한 단말 고유 식별자
  - **페이로드(payload)**: 알림 내용을 담은 JSON 딕셔너리
- **APNS(Apple Push Notification Service)**: 애플이 제공하는 원격 서비스. 푸시 알림을 iOS 단말로 보내는 역할.
- **iOS 단말**: 알림을 최종적으로 수신하는 사용자 단말

```json
{
  "aps": {
    "alert": {
      "title": "게임 초대",
      "body": "루시아님이 게임에 초대했습니다."
    },
    "sound": "default"
  }
}
```

### 안드로이드 푸시 알림

구조는 iOS와 동일하다. APNS 대신 **FCM(Firebase Cloud Messaging)** 을 사용한다.

```
알림 제공자(Provider) → FCM → 안드로이드 단말
```

> **왜 APNS/FCM을 거쳐야 하나? 우리 서버가 단말에 직접 보내면 안 되나?** 안 된다. 모바일 단말은 대부분 고정 IP가 없고, 이동하면서 네트워크가 바뀌며, 배터리를 아끼려고 대부분의 시간을 절전 상태로 보낸다. 서버가 "지금 이 폰"에 직접 연결할 방법이 없다. 그래서 OS 제조사가 단말과 하나의 지속 연결을 유지해두고, 모든 앱의 알림을 그 통로로 대신 전달한다. 앱마다 각자 연결을 유지하면 배터리가 남아나지 않기 때문이다. 결과적으로 APNS/FCM은 우리가 통제할 수 없는 외부 의존성이 된다. 이 서비스가 느려지거나 장애가 나면 우리 알림도 막힌다. 뒤에 나오는 "알림 종류별로 큐를 분리한다"는 설계가 바로 이 위험에 대한 대응이다.

### SMS 메시지

보통 직접 구축하지 않고 트윌리오(Twilio), 넥스모(Nexmo) 같은 제3자 서비스를 이용한다. 대부분 상용 서비스다.

### 이메일

직접 이메일 서버를 구축할 수도 있지만, 상용 서비스를 쓰는 것이 일반적이다. 센드그리드(SendGrid), 메일침프(Mailchimp) 등이 널리 쓰인다.

이유는 이렇다.
- **전송 성공률(delivery rate)이 높다**: 직접 구축한 서버는 스팸으로 분류되기 쉽다. 상용 서비스는 IP 평판(reputation)을 관리해준다.
- **데이터 분석 서비스를 제공한다**: 열람률, 클릭률 등을 제공한다.

### 통합 구조

세 가지를 모두 지원하면 다음과 같은 모양이 된다.

```
                    ┌→ APNS       → iOS 단말
알림 시스템 ────────┼→ FCM        → 안드로이드 단말
                    ├→ SMS 서비스  → SMS 단말
                    └→ 이메일 서비스 → 이메일
```

## 연락처 정보 수집 절차

알림을 보내려면 단말 토큰, 전화번호, 이메일 주소 등이 필요하다.

사용자가 앱을 설치하거나 처음 계정을 등록하면, API 서버가 해당 정보를 수집하여 데이터베이스에 저장한다.

```
user 테이블
| user_id | email | country_code | phone_number | created_at | last_login |

device 테이블
| id | user_id | device_token | last_logged_in_at |
```

핵심은 **한 사용자가 여러 단말을 가질 수 있다**는 점이다(폰, 태블릿, 노트북). 그래서 `user`와 `device`를 1:N 관계로 분리한다. 알림은 그 사용자의 모든 단말에 전송되어야 한다.

## 알림 전송 및 수신 절차

### 1차 초안

```
서비스 1~N → [알림 시스템] → 제3자 서비스 → 단말
```

- **서비스 1~N**: 마이크로서비스, 크론잡, 분산 시스템 컴포넌트 등 알림을 발생시키는 주체들. (예: 결제 서비스가 "결제 완료" 알림 요청)
- **알림 시스템**: 알림 전송/수신 처리의 핵심. 서버 1대.
  - 서비스 1~N에 알림 전송 API를 제공
  - 제3자 서비스에 전달할 페이로드를 만듦
- **제3자 서비스**: APNS, FCM, SMS 서비스, 이메일 서비스
- **단말**: 최종 수신자

### 1차 초안의 문제점

1. **단일 장애점(SPOF)**: 알림 서버가 한 대뿐이라, 이 서버가 죽으면 알림이 전부 멈춘다.
2. **규모 확장의 어려움**: DB, 캐시, 알림 처리 로직을 전부 한 서버에서 처리한다. 규모를 키우기 어렵다.
3. **성능 병목**: 알림 처리는 자원을 많이 쓴다. 이메일 HTML 페이지를 만들고, 제3자 서비스의 응답을 기다려야 한다. 이 모든 걸 한 시스템에서 처리하면 트래픽이 몰리는 시간대에 시스템이 과부하된다.

### 개선안

세 가지 방향으로 개선한다.

- 데이터베이스와 캐시를 알림 시스템의 주 서버에서 분리한다.
- 알림 서버를 증설하고 자동으로 수평적 규모 확장이 이루어지도록 한다.
- 메시지 큐를 도입해 시스템 컴포넌트 사이의 강한 결합을 끊는다.

```
서비스 1~N
    ↓
[알림 서버들] ←→ 캐시 / DB
    ↓
[메시지 큐] (알림 종류별로 분리)
    ↓
[작업 서버들(worker)]
    ↓
제3자 서비스 → 단말
```

**알림 서버의 역할:**
- 알림 전송 API 제공: 스팸 방지를 위해 보통 사내 서비스 또는 인증된 클라이언트만 이용 가능하도록 한다.
- 알림 검증(validation): 이메일 주소, 전화번호 등 기본적인 검증
- DB/캐시 질의: 알림 렌더링에 필요한 데이터를 가져온다.
- 알림 전송: 알림 데이터를 메시지 큐에 넣는다.

**캐시**: 사용자 정보, 단말 정보, 알림 템플릿 등을 캐시한다.

**메시지 큐**: 시스템 컴포넌트 간 의존성을 제거한다. 대량의 알림을 보내야 할 때 버퍼 역할도 한다. 알림 종류별로 별도의 큐를 둔다.

**작업 서버(worker)**: 메시지 큐에서 알림을 꺼내 제3자 서비스로 전달한다.

> **메시지 큐가 "강한 결합을 끊는다"는 게 무슨 의미인가?** 큐가 없으면 알림 서버가 제3자 서비스의 응답을 직접 기다려야 한다. APNS가 느려지면 알림 서버의 스레드가 거기 묶여서, 다른 요청을 못 받게 된다. 즉 외부 서비스의 장애가 곧바로 우리 시스템의 장애가 된다. 큐를 두면 알림 서버는 "큐에 넣었다"까지만 하고 즉시 응답한다. 실제 전송은 작업 서버가 나중에 처리한다. APNS가 느려도 큐에 메시지가 쌓일 뿐, 알림 서버는 계속 요청을 받는다. 앞서 확인한 "연성 실시간이라 약간의 지연은 허용된다"는 요구사항이 이걸 가능하게 한다.
>
> 알림 종류별로 큐를 나누는 이유도 같은 맥락이다. 하나의 큐를 공유하면 SMS 서비스에 장애가 났을 때 그 메시지가 큐 앞을 막아, 정상인 이메일 알림까지 밀리게 된다. 큐를 분리하면 한 종류의 제3자 서비스에 장애가 나도 다른 종류의 알림은 정상 동작한다. **(장애 격리)**

## 상세 설계

### 안정성(Reliability)

#### 데이터 손실 방지

알림 시스템의 가장 중요한 요구사항 중 하나는 어떤 상황에서도 **알림이 소실되면 안 된다**는 것이다. 알림은 지연되거나 순서가 뒤바뀌어도 되지만, 사라지면 안 된다.

이를 위해 알림 로그(notification log) 데이터베이스를 유지한다. 알림 데이터를 DB에 보관하고, 실패 시 재시도하는 메커니즘을 구현한다.

```
알림 요청 → 로그 DB에 기록 → 큐에 삽입 → 작업 서버가 전송
                ↑                              ↓
                └──────── 실패하면 재시도 ──────┘
```

#### 알림 중복 전송 방지

정확히 한 번(exactly once)만 전송되도록 보장하기는 어렵다. 대부분의 경우 알림은 딱 한 번만 전송되지만, 분산 시스템의 특성상 가끔 중복 전송되기도 한다.

빈도를 줄이려면 중복 탐지 메커니즘을 도입한다. 보내야 할 알림이 도착하면 그 이벤트 ID를 검사해서 이전에 본 적 있는 이벤트인지 확인한다. 중복이면 버리고, 아니면 전송한다.

> **왜 "정확히 한 번"이 어려운가?** 작업 서버가 제3자 서비스에 전송 요청을 보냈는데 응답이 오지 않는 상황을 생각해보자. 두 가지 가능성이 있다.
> - 요청이 도달하지 않았다 → 재시도해야 함
> - 요청은 도달해서 알림이 발송됐는데, 응답만 유실됐다 → 재시도하면 중복 발송
>
> 작업 서버는 이 둘을 구분할 수 없다. 재시도하면 중복 위험, 안 하면 손실 위험이다. 알림 시스템은 "손실 > 중복"이 더 큰 문제이므로, 재시도하는 쪽(at-least-once)을 택하고 대신 중복을 탐지해서 걸러낸다.
>
> 이벤트 ID로 중복을 거르는 이런 성질을 **멱등성(idempotency)** 이라고 한다. 같은 요청을 여러 번 처리해도 결과가 한 번 처리한 것과 같아지는 성질이다.

### 알림 템플릿

대형 알림 시스템은 하루에 수백만 건의 알림을 발송하는데, 그 상당수는 형식이 비슷하다. 알림 템플릿은 모든 알림을 처음부터 다시 만들 필요가 없도록 하기 위한 것이다. 인자(parameter), 스타일, 추적 링크(tracking link)만 조정해서 사용한다.

```
BODY: 지금 주문하시면 {할인율}% 할인!
CTA:  지금 주문하기
```

장점:
- 전송될 알림들의 형식을 일관성 있게 유지할 수 있다.
- 오류 가능성과 알림 작성에 드는 시간을 절약한다.

### 알림 설정

사용자는 이미 너무 많은 알림을 받고 있어서 쉽게 피로감을 느낀다. 그래서 많은 웹사이트와 앱이 알림 설정을 세밀하게 제어할 수 있도록 한다.

사용자별 알림 설정은 별도 테이블에 저장한다.

```
notification_setting 테이블
| user_id | channel | opt_in |
```

- `channel`: 알림이 전송될 채널(푸시 알림 / 이메일 / SMS)
- `opt_in`: 해당 채널로 알림을 받을지 여부

알림을 보내기 전에 반드시 이 설정을 확인해서, 수신을 거부한 사용자에게는 보내지 않는다.

### 전송률 제한

사용자에게 너무 많은 알림을 보내지 않도록, 한 사용자가 받을 수 있는 알림의 빈도를 제한한다.

너무 많이 보내면 사용자가 알림 기능을 아예 꺼버릴 수도 있기 때문이다. 한 번 꺼진 알림은 다시 켜기 어렵다.

> 앞서 4장에서 다룬 처리율 제한 장치가 여기서 그대로 재사용된다. 다만 목적이 다르다. 4장에서는 "공격/과부하로부터 서버를 보호"하는 게 목적이었다면, 여기서는 **"사용자를 알림 피로에서 보호"** 하는 게 목적이다. 스코프도 다르다. IP나 API 엔드포인트가 아니라 사용자별·채널별로 제한을 건다. `rate:notification:{user_id}:{channel}:{date}` 같은 Redis 키를 쓰면 그대로 구현된다. 구현 기법(INCR + EXPIRE)은 동일하고, 어디에 적용하느냐만 다르다.

### 재시도 방법

제3자 서비스가 알림 전송에 실패하면, 해당 알림을 재시도 전용 큐에 넣는다. 같은 문제가 계속 발생하면 개발자에게 알림(alert)을 보낸다.

### 푸시 알림과 보안

iOS나 안드로이드 앱의 경우, 알림 전송 API는 appKey와 appSecret을 사용하여 보안을 유지한다. 인증된 혹은 승인된 클라이언트만 해당 API를 사용해 알림을 전송할 수 있다.

### 큐 모니터링

알림 시스템 모니터링 시 중요한 메트릭 하나는 큐에 쌓인 알림의 개수다.

이 수가 너무 크면 작업 서버들이 이벤트를 빠르게 처리하지 못하고 있다는 뜻이다. 이 경우 작업 서버를 증설해야 한다.

### 이벤트 추적

알림 확인율(open rate), 클릭율(click rate), 실제 앱 사용으로 이어지는 비율 같은 메트릭은 사용자를 이해하는 데 중요하다. 데이터 분석 서비스는 보통 이벤트 추적 기능도 제공하므로, 알림 시스템과 데이터 분석 서비스를 통합하는 것이 일반적이다.

```
알림 생성 → 발송 → 열람 → 클릭 → 앱 사용
   각 단계를 이벤트로 기록하여 분석
```

## 수정된 설계안

지금까지의 논의를 반영하면 다음이 추가된다.

- 알림 서버에 인증(authentication) 과 전송률 제한(rate limiting) 기능 추가
- 전송 실패에 대응하기 위한 재시도 기능 추가. 실패한 알림은 다시 큐에 넣고 재시도한다.
- 알림 템플릿 도입 — 일관성 유지, 시간 절약
- 모니터링과 이벤트 추적 시스템 추가 — 시스템 상태 확인 및 개선

```
서비스 1~N
    ↓
[알림 서버] — 인증, 전송률 제한, 검증, 설정 확인
    ↓            ↕ 캐시 / DB / 알림 템플릿
[알림 로그 DB] ← 기록
    ↓
[메시지 큐] (종류별 분리) ←── [재시도 큐]
    ↓                              ↑
[작업 서버] ──── 실패 시 ──────────┘
    ↓
제3자 서비스(APNS/FCM/SMS/Email) → 단말
    ↓
[이벤트 추적 / 분석 시스템]
```

## 마무리

알림 시스템 설계에서 최종적으로 짚어야 할 점들:

- **안정성**: 재시도 메커니즘을 도입해 전송 실패율을 낮춘다. 알림 로그 DB로 손실을 방지한다.
- **보안**: appKey/appSecret을 이용해 인증된 클라이언트만 알림을 전송할 수 있도록 한다.
- **이벤트 추적 및 모니터링**: 알림 생성 후 전송 성공/실패까지 추적하고, 큐 길이 등으로 시스템 상태를 모니터링한다.
- **사용자 설정 존중**: 알림을 받지 않겠다고 한 사용자는 제외한다. 알림 설정을 먼저 확인한다.
- **전송률 제한**: 사용자가 받는 알림의 빈도를 제한해 피로감을 줄인다.

## 정리

알림 시스템은 겉보기엔 단순하지만, 우리가 통제할 수 없는 제3자 서비스(APNS, FCM, SMS, 이메일)에 의존한다는 점이 설계의 핵심 제약이다. 이 제약 때문에 메시지 큐로 결합을 끊고, 종류별로 큐를 분리해 장애를 격리하며, 재시도와 중복 탐지로 안정성을 확보한다.

또한 기술적 안정성만큼 사용자 경험이 중요한 시스템이기도 하다. 알림 설정 존중과 전송률 제한이 없으면, 기술적으로 완벽해도 사용자가 알림을 꺼버리면서 시스템 자체가 무의미해진다.

앞선 장들과의 연결점도 뚜렷하다. 4장의 처리율 제한 장치가 사용자 보호 목적으로 재사용되고, 메시지 큐와 수평 확장, 무상태 서버 같은 개념은 앞서 다룬 분산 시스템 원칙들이 그대로 적용된 것이다.
