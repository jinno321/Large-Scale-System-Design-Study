애플리케이션 프로그램에서 여러가지 최신 뉴스, 제품 업데이트, 이벤트 등 정보를 비동기적으로 알림 서비스로 제공해준다. 단순히 모바일 푸시 알림에 한정되지 않고, 모바일 푸시 알림, SMS 메시지, 이메일 등 세가지로 분류할 수 있다

사용자가 꽤 많은 서비스에는 알림도 하루에 백만건 이상이 발생하게 된다. 이럴 때, 확장성 높은 시스템을 구축하는건 어렵다.

## 시스템 요구사항
* **지원 시스템:** 모바일 푸시 알림(iOS, 안드로이드), SMS 메시지, 이메일
* **실시간성:** 연성 실시간 시스템(Soft real-time) - 가능한 한 빨리 전달되어야 하지만, 시스템 과부하 시 약간의 지연은 무방함
* **단말 지원:** iOS, 안드로이드, 랩톱/데스크톱
* **사용자 설정:** 사용자가 알림 아웃풋(수신 여부 등) 설정 가능
* **처리량:** 하루에 천만 건의 모바일 푸시 알림, 백만 건의 SMS 메시지, 5백만 건의 이메일 처리 가능해야 함

## 개략적 설계안

### 1. 알림 유형별 지원 방안
운영체제와 기기별로 알림을 지원하는 메커니즘이 다르다.

#### iOS 푸시 알림
세 가지 컴포넌트가 필요하다.
* **알림 제공자 (Provider):** 알림 요청을 만들어 애플 푸시 알림 서비스(APNS)로 보내는 주체. 단말 토큰(Device Token)이라는 고유 식별자와 알림 내용을 담은 JSON 페이로드(Payload)를 전달함.
* **APNS (Apple Push Notification service):** 애플이 제공하는 원격 서비스로, 푸시 알림을 iOS 장치로 보내는 역할.
* **iOS 단말 (Client):** 푸시 알림을 최종 수신하는 사용자 단말.

#### 안드로이드 푸시 알림
* iOS와 유사한 구조를 가지나, APNS 대신 **FCM (Firebase Cloud Messaging)**을 사용하여 안드로이드 단말로 푸시 알림을 보낸다.

#### SMS 메시지
* 보통 Twilio, Nexmo 같은 상용 3차 서비스(Third-party service)를 이용하여 사용자 전화번호로 메시지를 발송한다.

#### 이메일
* 대부분의 회사는 고유 이메일 서버를 구축할 역량은 갖추고 있지만, 배달 성공률과 안정성을 위해 SendGrid, Mailchimp 같은 상용 이메일 서비스를 많이 이용한다.

### 2. 연락처 정보 수집 절차
알림을 보내기 위해서는 모바일 단말 토큰, 전화번호, 이메일 주소 등의 정보가 필요하다.
* 회원이 애플리케이션을 설치하거나 회원가입할 때 해당 정보들을 수집한다.
* 수집된 정보는 데이터베이스(사용자 테이블 및 단말 정보 테이블)에서 안전하게 관리한다.

### 3. 초기의 개략적 설계안 (단일 서버 가정 시의 문제점)
[서비스 1~N] ➔ [알림 서버 (단일)] ➔ [제3자 서비스 (APNS/FCM/Twilio 등)] ➔ [사용자 단말]

* **문제점 1 (SPOF):** 알림 서버가 하나뿐이므로, 이 서버에 장애가 발생하면 전체 알림 시스템이 중단된다.
* **문제점 2 (확장성 부족):** 한 대의 서비스로 푸시 알림에 관계된 모든 것을 처리하므로, 데이터베이스나 캐시 등 중요 컴포넌트의 규모를 개별적으로 늘릴 방법이 없다.
* **문제점 3 (성능 병목):** 제3자 서비스의 응답을 기다리는 건 시간이 많이 걸릴 수 있다. 요청이 몰리는 피크 타임에는 전송 지연 및 서버 과부하가 발생한다.

### 4. 개선된 개략적 설계안
초기 모델의 단점을 보완하기 위해 **서버 분리, 수평 확장, 메시지 큐(Message Queue)**를 도입한다.

* 데이터베이스와 캐시를 알림 시스템의 주 서버에서 분리한다.
* 알림 서버를 증설하고 로드밸런서를 두어 수평적 규모 확장(Horizontal Scaling)이 가능하게 한다.
* 메시지 큐를 이용해 시스템 컴포넌트 사이의 강한 결합을 끊는다.

#### [컴포넌트별 기능]
* **알림 서버:** 
  * **알림 전송 API:** 인증된 클라이언트나 사내 서비스만 API를 호출할 수 있도록 설계(스팸 방지).
  * **알림 검증:** 이메일 주소, 전화번호 등에 대해 기본적인 형식을 검증.
  * **데이터베이스 또는 캐시 질의:** 알림에 포함시킬 사용자 정보나 템플릿 데이터를 조회.
  * **알림 전송:** 검증된 알림 데이터를 각 알림 유형에 맞는 **메시지 큐**로 전달.
* **메시지 큐 (Message Queue):** 컴포넌트 간 의존성을 제거하고, 다량의 알림이 폭주할 때 버퍼 역할을 수행 (종류별로 큐를 두어 하나의 장애가 다른 유형에 영향을 주지 않도록 함).
* **작업 서버 (Worker):** 메시지 큐에서 전송할 알림 이벤트를 꺼내서 제3자 서비스로 전달하는 역할.

## 상세 설계

### 1. 안정성 (Robustness)

#### 데이터 손실 방지 (데이터 보관 및 재시도)
* 어떠한 상황에서도 알림이 소실되면 안 된다. 지연이나 순서가 틀리는 것은 허용되지만 무단 유실은 곤란하다.
* 이를 위해 작업 서버는 알림 전송 상태를 **알림 로그 데이터베이스(Notification Log DB)**에 기록해야 한다. 제3자 서비스 전송이 실패하더라도 로그를 바탕으로 재시도 메커니즘을 가동한다.

#### 알림 중복 전송 방지
* 분산 시스템 특성상 동일한 알림이 중복 전송되는 현상을 100% 막기는 어렵다. (At-least-once 전달 모델)
* 중복 빈도를 줄이기 위해 **중복 탐지 메커니즘**을 도입한다. 보내야 할 알림이 도착하면 **이벤트 ID(Deduplication Key)**를 검사하여, 최근에 이미 발송된 이벤트를 찾아내고 중복 시 버리는 로직을 구현한다.

### 2. 추가 컴포넌트 및 고려사항

#### 알림 템플릿
* 하루에도 수백만 건 이상의 알림을 처리할 때 메시지의 형식이 대부분 비슷하므로 템플릿을 쓰면 유용하다.
* 파라미터(이름, 날짜 등)나 스타일, 추적 링크만 조정하여 알림 메시지를 동적으로 생성하므로, 일관성을 유지하고 오타 등의 오류 가능성을 줄인다.

#### 알림 설정 (User Settings)
* 사용자가 받는 알림 피로도를 줄이기 위해 알림 설정을 세분화하여 조정할 수 있도록 한다. 사용자 테이블이나 별도 설정 테이블에 채널별(푸시, SMS, 이메일) 수신 유무를 관리하고, 알림 발송 전 이 설정을 반드시 확인한다.

#### 전송률 제한 (Rate Limiting)
* 한 사용자가 짧은 시간 동안 받을 수 있는 알림의 빈도를 제한한다. 너무 자주 알림을 보내면 사용자가 서비스 알림을 아예 꺼버리거나 앱을 삭제할 수 있기 때문이다.

#### 재시도 방법 (Retry Mechanism)
* 제3자 서비스가 알림 전송에 실패하면, 해당 알림을 즉시 버리지 않고 **재시도 전용 큐(Retry Queue)**에 넣는다. 만약 지속적으로 실패하여 최대 재시도 횟수를 초과하면 개발자가 인지할 수 있도록 데드 레터 큐(Dead Letter Queue)에 넣고 알람을 보낸다.

#### 푸시 알림과 보안
* iOS와 안드로이드 앱의 경우, 알림 전송 API는 `appKey`와 `appSecret` 등 인증 토큰을 사용하여 보안을 유지한다. 이를 통해 검증된 사내 서버나 승인된 클라이언트만 알림을 발송할 수 있도록 제어한다.

#### 큐 모니터링 (Queue Monitoring)
* 알림 시스템의 핵심 메트릭 중 하나는 큐에 쌓인 **대기 오프셋(Lag) 및 알림 개수**이다. 이 숫자가 너무 크다면 작업 서버(Worker)들이 이벤트를 빠르게 처리하지 못하고 있다는 뜻이므로, 작업 서버를 동적으로 스케일 아웃(Scale-out)해야 하는 지표가 된다.

## SSE (Server-Sent Events) 개요 및 차이점

### 1. SSE (Server-Sent Events) 란?
SSE는 서버가 클라이언트로 데이터를 **실시간 단방향 스트리밍**할 수 있도록 하는 HTTP 표준 기술이다. 클라이언트가 서버와 최초 한 번 HTTP 연결을 맺고 나면, 서버는 웹 소켓(WebSocket)처럼 연결을 계속 유지하면서 이벤트가 발생할 때마다 클라이언트에 데이터를 밀어줄(Push) 수 있다.

### 2. 본 알림 시스템 설계안 vs SSE의 차이점

| 비교 항목 | 본 알림 시스템 설계안 (대규모 알림 시스템) | SSE (Server-Sent Events) |
| :--- | :--- | :--- |
| **통신 방향 및 메커니즘** | 다양한 플랫폼(OS 내장 푸시, 이메일, SMS)의 외부 제3자 API를 거쳐 단말로 도달하는 간접 구조 | 서버와 웹 브라우저(클라이언트)가 직접 HTTP 지속 연결을 맺고 데이터를 스트리밍하는 구조 |
| **주요 대상 기기** | 앱이 꺼져 있거나 기기가 잠겨 있는 **모바일 단말기(iOS/Android)** 중심 | 주로 웹 브라우저나 앱이 **현재 켜져 있는 상태(Active)**의 대시보드 및 웹 애플리케이션 중심 |
| **연결 지속 여부** | 서버가 단말과 항상 연결을 유지할 필요 없음 (이벤트 발생 시 외부 인프라에 요청 위임) | 클라이언트와 서버가 상시 **HTTP Persistent Connection**을 유지해야 하므로 동시 접속 동시성이 중요함 |
| **활용 예시** | 마케팅 푸시, 결제 완료 문자, 비밀번호 변경 안내 이메일 발송 등 | 주식 시세 실시간 변동, 실시간 알림 피드 피드 업데이트, 실시간 채팅(수신만) 등 |

**결론:** 책 내용은 앱이 꺼져 있거나 백그라운드 상태일 때도 통신사나 OS 레벨 인프라를 통해 도달해야 하는 **시스템/마케팅 알림**에 적합하고, SSE는 사용자가 웹 페이지를 보고 있는 동안 서버에서 발생하는 이벤트를 실시간으로 화면에 반영해야 하는 **화면 내 실시간 인터랙션**에 특화된 기술이다.
