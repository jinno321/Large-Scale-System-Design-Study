# 분산 키-값 저장소 설계 요약

키-값 저장소(Key-Value Store)는 키-값 데이터베이스라고도 불리는 비관계형(NoSQL) 데이터베이스이다. 이 시스템에서 키값은 **고유(Unique)**해야 하며, 키-값 쌍(Key-Value Pair)이라는 연결 관계를 갖는다. 키는 일반 텍스트일 수도 있고 해시 값일 수도 있으며, **키의 길이가 짧을수록 성능이 좋아진다.**

---

## 1. 단일 서버 키-값 저장소
한 대의 서버만 사용하는 키-값 저장소를 설계하는 것은 간단하다. 모든 데이터를 메모리에 해시 테이블로 저장하면 된다. 그러나 이 접근법은 모든 데이터를 메모리 안에 두기 때문에 데이터 크기가 커지면 용량 및 성능 상의 이유로 불가능해질 수 있다.

이를 개선하기 위해 다음과 같은 방법을 사용한다.
* **데이터 압축**
* **자주 쓰이는 데이터만 메모리에 두고, 나머지는 디스크에 저장**

하지만 이 방식도 데이터 량이 계속 늘어나면 곧 한 대의 서버로는 부족한 시점이 찾아온다. 따라서 대규모 데이터를 저장하기 위해서는 **분산 키-값 저장소**를 만들 필요가 있다.

---

## 2. 분산 키-값 저장소와 CAP 정리
분산 키-값 저장소를 설계할 때는 **CAP 정리**를 반드시 이해하고 있어야 한다.

### CAP 정리의 3가지 요소
* **데이터 일관성 (Consistency):** 분산 시스템에 접속하는 모든 클라이언트는 어떤 노드에 접속했느냐에 관계없이 언제나 같은 데이터를 보게 되어야 한다.
* **가용성 (Availability):** 분산 시스템에 접속하는 클라이언트는 일부 노드에 장애가 발생하더라도 항상 응답을 받을 수 있어야 한다.
* **파티션 감내 (Partition Tolerance):** 두 노드 사이에 통신 장애(파티션)가 발생해도 시스템은 계속 동작하여야 한다.

> **CAP 정리의 핵심:** 이 세 가지 요구사항을 동시에 만족하는 분산 시스템을 설계하는 것은 불가능하다.

따라서 이 중 2가지를 채택하는 시스템을 가져가야 하며, 현실적으로는 **CP 시스템** 또는 **AP 시스템**을 선택하게 된다. 현실 세계에서 네트워크 장애(파티션)는 피할 수 없는 일로 여겨지므로, 가용성과 일관성을 모두 만족하면서 파티션을 무시하는 **CA 시스템은 존재할 수 없다.**

### 실세계의 분산 시스템 예시
이상적인 상태에서는 네트워크 파티션이 일어나지 않으므로 $n_1 \sim n_3$ 노드가 있을 때, $n_1$에 데이터가 기록되면 $n_2, n_3$에 자동으로 복제된다. 그러나 실세계에서는 파티션 문제를 피할 수 없다. 

만약 $n_3$에 파티션 문제가 발생하면 $n_1, n_2$와 통신할 수 없게 된다. 이때 $n_3$에 데이터가 기록되었으나 아직 $n_1, n_2$로 전송되지 않았다면, $n_1$과 $n_2$는 오래된 사본을 갖고 있게 된다.

* **CP 시스템 (일관성 + 파티션 감내):** 금융권(은행) 등에서 주로 선택한다. 온라인 뱅킹 시스템에서 최신 계좌 정보(잔액 등)를 정확하게 출력하지 못한다면 큰 문제가 발생하기 때문이다. 따라서 데이터 동기화가 완벽해질 때까지 읽기/쓰기를 차단하고 오류를 반환하며 가용성을 포기한다.
* **AP 시스템 (가용성 + 파티션 감내):** 낡은 데이터를 반환할 위험이 있어도 시스템의 중단 없는 서비스를 위해 계속 읽기/쓰기 연산을 허용한다.

## 3. 핵심 컴포넌트

### 데이터 파티션 (Data Partition)
전체 데이터를 한 대의 서버에 저장하는 것은 불가능하므로, 데이터를 작은 파티션들로 분할하여 여러 대의 서버에 나누어 저장해야 한다. 이때 아래 두 가지 사항을 고려해야 한다.
1.  데이터를 여러 서버에 고르게 분산할 수 있는가?
2.  노드가 추가되거나 삭제될 때 데이터의 이동을 최소화할 수 있는가?

이 문제를 해결하기 위해 **안정 해시(Consistent Hashing)**를 사용할 수 있다. 안정 해시의 특징상 규모 확장 자동화를 쉽게 이뤄낼 수 있으며, 각 서버의 용량에 맞게 **가상 노드(Virtual Node)**의 수를 조정하여 균등한 분산이 가능하다.

### 데이터 다중화 (Data Replication)
높은 가용성과 안정성을 확보하려면 데이터를 $N$개의 서버에 비동기적으로 다중화해야 한다. 그래야 한 서버에 파티션(장애)이 발생해도 다른 서버에서 데이터를 가져올 수 있기 때문이다.

* **다중화 방법:** 키를 해시 링(Ring) 위에 배치한 후, 그 지점으로부터 **시계 방향**으로 링을 순회하면서 만나는 첫 $N$개의 서버에 데이터 사본을 저장한다.

## 4. 데이터 일관성

여러 노드에 다중화된 데이터는 적절히 동기화되어야 한다. 이를 위해 정족수 합의 프로토콜을 사용한다.

* $N$ = 사본 개수
* $W$ = 쓰기 연산에 대한 정족수 (쓰기가 성공했다고 간주하려면 적어도 $W$개의 서버로부터 성공 응답을 받아야 함)
* $R$ = 읽기 연산에 대한 정족수 (읽기가 성공했다고 간주하려면 적어도 $R$개의 서버로부터 성공 응답을 받아야 함)

이 과정은 클라이언트와 노드 사이에서 프록시 역할을 해주는 **중재자(Coordinator)**가 관리한다. $W$와 $R$의 값을 정하는 것은 **응답 지연(Latency)과 데이터 일관성(Consistency) 사이의 트레이드오프**를 조절하는 중요한 과정이다.

* $R=1, W=N$ : 빠른 읽기 연산 최적화 시스템
* $R=N, W=1$ : 빠른 쓰기 연산 최적화 시스템
* $W + R > N$ : **강한 일관성(Strong Consistency)** 보장 (보통 $N=3, W=2, R=2$ 조합을 사용)

### 일관성 모델 (Consistency Models)
* **강한 일관성 (Strong Consistency):** 모든 읽기 연산은 가장 최근에 갱신된 결과를 반환한다. 클라이언트는 절대로 낡은 데이터를 보지 못한다.
* **약한 일관성 (Weak Consistency):** 읽기 연산이 가장 최근에 갱신된 결과를 반환하지 못할 수 있다.
* **결과적 일관성 (Eventual Consistency):** 약한 일관성의 한 형태로, 당장은 일치하지 않더라도 갱신 결과가 결국에는 모든 사본에 반영되는 모델이다.

> 강한 일관성을 달성하는 가장 쉬운 방법은 모든 사본에 쓰기가 끝날 때까지 데이터에 **락(Lock)**을 거는 것인데, 이는 고가용성 시스템에는 부적합하다. 따라서 다이나모(DynamoDB)나 카산드라(Cassandra) 같은 저장소는 **결과적 일관성 모델**을 채택하고 있다.

## 5. 고급 메커니즘 (충돌 및 장애 처리)

### 데이터 버저닝과 벡터 시계 (Vector Clock)
데이터를 다중화하면 가용성은 높아지지만 사본 간의 일관성이 깨질 수 있다. 이를 해결하기 위해 **버저닝(Versioning)**과 **벡터 시계(Vector Clock)**를 사용한다. 버저닝은 데이터를 변경할 때마다 해당 데이터의 새로운 버전을 만드는 방식이다.



#### 충돌 발생 시나리오
$n_1$과 $n_2$에 똑같은 데이터가 있을 때, 다른 두 클라이언트가 동시에 데이터를 각각 $v_1, v_2$로 바꾸는 요청을 보낸다면 두 노드의 데이터가 서로 달라진다. 이 충돌을 발견하고 자동으로 해결하기 위해 벡터 시계를 사용한다.

* **표현 방식:** `D([S1, v1], [S2, v2], ...)`의 형태로 데이터에 [서버, 버전] 순서쌍을 매긴다. ($D$: 데이터, $S_i$: 서버 번호, $v_i$: 버전 카운터)
* **동작 원리:** 데이터 $D$를 서버 $S_i$에 기록할 때, 순서쌍 목록에 $S_i$가 있으면 $v_i$를 증가시키고, 없으면 `[S_i, 1]`을 새로 추가한다.

이를 통해 어떤 버전 $X$가 버전 $Y$의 이전 버전인지, 후행 버전인지, 아니면 충돌 관계인지 판별할 수 있다. $Y$의 벡터 시계 구성요소 중 $X$의 동일 서버 구성요소보다 작은 값을 갖는 것이 하나라도 있다면 선후 관계가 깨진 충돌 상태로 판단한다.

#### 벡터 시계의 단점
1.  충돌 감지 및 해소 로직이 클라이언트 쪽에 포함되어야 하므로 구현이 복잡해진다.
2.  `[서버: 버전]` 순서쌍의 개수가 너무 빠르게 늘어난다. 이를 해결하기 위해 임계치(Threshold)를 설정하고 길이가 길어지면 오래된 순서쌍을 제거해야 하는데, 이 경우 선후 관계 결정의 정확도가 떨어져 충돌 해소 효율이 낮아질 수 있다. *(참고: 아마존 다이나모 DB에 의하면 실서비스에서 이로 인한 문제가 벌어진 적은 거의 없다고 한다.)*

### 장애 감지 (Failure Detection)
분산 시스템에서는 단순히 하나의 서버가 특정 서버를 "죽었다"고 판단해서 바로 장애 처리를 하지 않는다. **네트워크 일시적 단절이나 지연으로 인한 오판일 수 있기 때문에, 보통 두 대 이상의 노드가 똑같이 해당 서버의 장애를 보고해야 실제로 장애가 발생했다고 간주한다.**

장애 감지를 위해 모든 노드가 서로 통신하는 멀티캐스팅(Multicasting) 방식은 서버 수가 늘어날 때 기하급수적으로 연결이 늘어나 비효율적이다. 따라서 **가십 프로토콜(Gossip Protocol)** 같은 분산형 장애 감지 솔루션을 사용한다.

* **가십 프로토콜의 동작:** 각 노드는 멤버십 목록과 **박동 카운터(Heartbeat Counter)**를 유지한다. 각 노드는 주기적으로 자신의 박동 카운터를 증가시키고, 무작위로 선정된 이웃 노드들에게 자신의 카운터 목록을 보낸다. 어떤 멤버의 박동 카운터 값이 지정된 시간 동안 갱신되지 않으면, 시스템은 해당 멤버를 장애 상태로 판단한다. (마치 주기적으로 부모님께 생존 신고를 하는 것과 유사하다.)

### 장애 처리 (Failure Handling)

#### 1. 일시적 장애 처리: 단서 후 임시 위탁 (Hinted Handoff)
엄격한 정족수 접근법을 쓴다면 장애 발생 시 읽기/쓰기 연산을 금지해야 하지만, 가용성을 높이기 위해 **느슨한 정족수(Sloppy Quorum)** 접근법을 쓴다면 해시 링에서 장애 서버를 대신해 읽기/쓰기를 수행할 임시 서버들을 고른다.

장애 서버가 복구될 때까지 임시 서버가 요청을 대신 처리하며, 이때 발생한 변경 사항은 임시로 **단서(Hint)**라는 로그로 남겨둔다. 이후 장애 노드가 복구되면 이 단서들을 일괄 반영하여 일관성을 보존한다. 이를 **단서 후 임시 위탁(Hinted Handoff)** 기법이라고 한다.

#### 2. 영구적 장애 처리: 머클 트리 (Merkle Tree)
노드의 영구적인 장애 상태는 **반-엔트로피(Anti-Entropy) 프로토콜**을 구현하여 사본들을 동기화한다. 이때 사본 간의 일관성이 깨진 상태를 탐지하고 동기화 시 전송되는 데이터 양을 줄이기 위해 **머클 트리(해시 트리)**를 사용한다.



* **머클 트리 구축 단계:**
    1.  키 공간을 여러 개의 버킷(Bucket)으로 나눈다.
    2.  버킷에 포함된 각각의 키에 균등 분포 해시 함수를 적용하여 해시 값을 계산한다.
    3.  버킷별로 해시 값을 계산하여 해당 해시 값을 레이블로 갖는 루트 직전의 노드를 만든다.
    4.  자식 노드의 레이블로부터 새로운 해시 값을 계산하여, 이진 트리를 상향식(Bottom-up)으로 구성해 나간다.

* **비교 및 동기화:** 두 서버의 머클 트리 루트(Root) 노드의 해시 값을 비교한다. 값이 일치하면 두 서버의 데이터는 완전히 일치하는 것이다. 값이 다르면 자식 노드로 내려가면서 해시 값을 비교하며, 최종적으로 **데이터가 다른 버킷만 찾아내어 해당 버킷의 데이터만 동기화**하므로 네트워크 전송량을 획기적으로 줄일 수 있다.

## 6. 쓰기 경로와 읽기 경로 (카산드라 사례)

### 쓰기 경로 (Write Path)
1.  쓰기 요청이 들어오면 우선 재해 복구용 **커밋 로그(Commit Log)** 파일에 기록된다.
2.  이후 데이터가 메모리 캐시인 **MemTable**에 기록된다.
3.  MemTable이 가득 차거나 사전에 정의된 임계치(Threshold)에 도달하면, 데이터는 디스크에 있는 **SSTable (Sorted-String Table)** 구조로 플러시(Flush)되어 영구 저장된다.

### 읽기 경로 (Read Path)
1.  먼저 데이터가 메모리 캐시(**Row Cache** 등)에 있는지 검사하고, 있으면 즉시 반환한다.
2.  메모리에 없다면 데이터가 디스크의 어느 고정된 위치에 있는지 효율적으로 찾기 위해 **블룸 필터(Bloom Filter)**를 검사한다.
3.  블룸 필터를 통해 해당 키를 보관하고 있을 가능성이 있는 **SSTable**을 알아낸다.
4.  지정된 SSTable에서 데이터를 가져와 최종적으로 클라이언트에게 반환한다.
