키-값 저장소 ⇒ 비 관계형 데이터베이스, 즉 NoSQL DB

- 이 저장소에 저장되는 값은 고유 식별자를 키로 가져야함.
- 키와 값 사이의 연결 관계를 “키-값” 쌍이라고 지칭
- 키-값 쌍에서의 키는 유일해야 하며, 키를 통해서만 접근 가능.
    - 키는 일반 텍스트 일수도 있고, 해시값일 수도 있음
    - 그러나 성능상의 이유로 키가 짧을 수록 좋음
- DynamoDB, memcached, 레디스 등등

### 문제 이해 및 설계 범위 확정

<aside>

- 키-값 쌍의 크기는 10KB이하이다.
- 큰 데이터를 저장할 수 있어야한다.
- 높은 가용성 - 시스템에 장애가 있더라도 빨리 응답
- 높은 규모 확장성 - 트래픽 양에 따라 자동적으로 서버 증설/삭제
- 데이터 일관성 수준 조절 가능
- 응답 지연시간이 짧아야함
</aside>

### 단일 서버 키-값 저장소

- 서버 한 대만 사용해서 설계하는 것.
- 단순한 방법 ⇒ 키-값 쌍 전부를 메모리에 해시 테이블로 저장
    - 빠른 속도를 보장하지만 모든 데이터를 메모리 안에 두는 것이 불가능할 수도 있음
        - 해결 - 데이터 압축 + 자주 쓰는 데이터만 메모리, 그 이외는 디스크 저장
        - 그러나 개선해도 부족할 수가 있음 ⇒ 분산 키-값 저장소 사용

### 분산 키-값 저장소

- 분산 키-값 저장소 = 분산 해시 테이블
    
    ⇒ 키-값 쌍을 여러 서버에 분산시키는 것
    

### CAP 정리

- 데이터 일관성(Consistency), 가용성(availability), 파티션 감내(partition tolerance)라는 세가지 요구사항을 동시에 만족하는 분산 시스템을 설계하는 것은 불가능 하다는 정리

<aside>

- 데이터 일관성
    - 분산 시스템에 접속하는 모든 클라이언트는 어떤 노드에 접속했느냐에 관계없이 언제나 같은 데이터를 보게 되어야 한다.
- 가용성
    - 분산 시스템에 접속하는 클라이언트는 일부 노드에 장애가 발생하더라도 항상 응답을 받을 수 있어야 한다.
- 파티션 감내
    - 파티션은 두 노드 사이에 통신 장애가 발생하였음을 의미.
    - 즉, 네트워크에 파티션이 생기더라도 시스템은 계속 동작하여야 한다는 것. (피할 수 없음)
</aside>

<aside>

- CP 시스템 ⇒ 일관성 + 파티션 감내 지원. 가용성 희생

> 
> 
> 
> **결제 시스템** — 잔액 불일치를 용납할 수 없습니다. 파티션 발생 시 거래를 거절하고 일관성을 지킵니다.
> 
> **분산 설정 관리 (Zookeeper)** — 모든 노드가 동일한 리더·설정값을 공유해야 합니다.
> 
> **재고 관리** — 남은 재고 1개를 두 곳에서 동시에 구매 허용하면 안 됩니다.
> 
- AP 시스템 ⇒ 가용성 + 파티션 감내 지원. 일관성 희생

> 
> 
> 
> **SNS 좋아요 수** — 잠깐 숫자가 틀려도 괜찮습니다. 서비스가 멈추는 게 더 나쁩니다.
> 
> **장바구니** — 오래된 데이터가 보여도 결제 직전 최종 확인으로 충분합니다. 무중단이 중요합니다.
> 
> **DNS** — 업데이트 전파에 시간이 걸려도 항상 응답해야 합니다. Eventual Consistency의 대표 사례입니다.
> 
- CA 시스템 ⇒ 일관성 + 가용성 지원. 파티션 감내 x
    - 통상적으로 네트워크 장애는 피할 수 없으므로 분산 시스템은 반드시 파티션 문제를 감내할 수 있도록 설계되어야 함.
    - 즉, 실세계에서 CA 시스템은 존재하지 않음.

> 
> 
> 
> **단일 노드 RDBMS** — 서버가 하나라면 파티션 자체가 없으므로 C+A 모두 가능합니다. 하지만 수평 확장이 불가합니다.
> 
> **분산 시스템에서는 불가능** — 네트워크 파티션은 언제든 발생합니다. CA는 단일 머신 시나리오에서만 의미가 있습니다.
> 
</aside>

### 데이터 파티션

- 대규모 애플리케이션 → 전체 데이터를 한 대 서버에 욱여넣는 것이 불가능
- 해결 - 데이터를 작은 파티션들로 분할한 다음 여러 대 서버에 저장하는것

<aside>

고려해야하는 포인트

- 데이터를 여러 서버에 고르게 분산할 수 있는가?
- 노드가 추가되거나 삭제될 떄 데이터의 이동을 최소화할 수 있는가?

⇒ 둘다 만족하는 해시링 사용

- 규모 확장 자동화 가능 - 시스템 부하에 따라 서버가 자동으로 추가/삭제 가능
- 다양성 - 각 서버의 용량에 맞게 가상 노드 수를 조정 가능
</aside>

### 데이터 다중화

- 높은 가용성 + 안정성을 확보하기 위해 데이터를 N개 서버에 비동기적으로 다중화할 필요가 있음
- N → 튜닝한 값
- N개의 서버를 선장하는 방법
    
    <aside>
    
    1. 어떤 키를 해시 링 위에 배치
    2. 그 지점으로부터 시계 방향으로 링 순회 해서 만나는 N개의 서버에 데이터 사본을 저장
    </aside>
    

### 데이터 일관성

- 여러 노드에 다중화된 데이터는 적절히 동기화가되어야 한다.
- 정족수 합의(Quorum Consensus) 프로토콜 사용해서 읽기/쓰기 연산에 일관성 보장

<aside>

N = 사본 개수

W = 쓰기 연산에 대한 정족수.

- 쓰기 연산이 성공한 것으로 간주되려면 적어도 W개의 서버로부터 쓰기 연산이 성공했다는 응답을 받아야 함.

R = 읽기 연산에 대한 정족수. 

- 읽기 연산이 성공한 것으로 간주되려면 적어도 R개의 서버로부터 응답을 받아야 함.
</aside>

<aside>

R=1, W=N: 빠른 읽기 연산에 최적화된 시스템

W=1, R=N: 빠른 쓰기 연산에 최적화된 시스템

W+R > N: 강한 일관성이 보장됨 (보통 N=3, W=R=2)

W+R ≤ N: 강한 일관성이 보장되지 않음

→ 요구되는 일관성 수준에 따라 W, R, N의 값을 조정한다!

</aside>

### 일관성 모델

<aside>

- 강한 일관성
    - 모든 읽기 연산은 가장 최근에 갱신된 결과를 반환한다.
    클라이언트는 절대 낡은 데이터를 보지 못한다.
    - 모든 사본에 쓰기 연산의 결과가 반영될 때까지 해당 데이터에 대한 읽기/쓰기 금지
        - 고가용성 시스템에는 적합하지 않음.
- 약한 일관성
    - 읽기 연산은 가장 최근에 갱신된 결과를 반환하지 못할 수 있다.
- 결과적 일관성
    - 약한 일관성의 한 형태로, 갱신 결과가 결국에는 모든 사본에 반영(즉, 동기화)되는 모델
    - 쓰기 연산이 병렬적으로 발생
    - 비일관성 → 클라이언트가 해결
</aside>

**비일관성 해소 기법 ⇒ 데이터버저닝**

#### 벡터시계

→ [서버, 버전]의 순서쌍을 데이터에 매단 것.
어떤 버전이 선행인지 후행인지, 아니면 다른 버전과 충돌이 있는지 판별.

`D([S1,v1], [S2,v2], ..., [Sn,vn])` 으로 표현을 한다고 가정했을때

- [Si, vi]가 있으면 vi를 증가시킨다.
- 그렇지 않으면 새 항목 [Si, 1]을 만든다.

→ 벡터 시계를 사용하면 어떤 버전 X가 버전 Y의 이전 버전인지 쉽게 판단 가능.

- 버전 Y에 포함된 모든 구성요소의 값이 X에 포함된 모든 구성요소의 값보다 같거나 큰지 확인

**단점**

1. 클라이언트가 충돌 감지 및 해소 로직을 가짐. → 구현 복잡
2. [서버: 버전]의 순서쌍 개수가 굉장히 빨리 늘어남.
    1. 임계치를 설정하고 임계치 이상으로 길이가 길어지면 오래된 순서쌍을 벡터 시계에서 제거.

### 장애처리

#### 장애 감지

- 분산 시스템에서 어떤 서버가 죽었다는 것을 어떻게 판단할까?
- 멀티캐스팅
    - 서버 각각이 다른 서버 전부를 확인
- 가십 프로토콜
    - 멤버십 목록 + 심장 박동 카운터
    
    <aside>
    
    - 각 노드 멤버십 목록 유지.
        - 멤버십 목록 = 각 서버의 멤버 ID와 박동 카운터 쌍의 목록
    - 각 노드는 주기적으로 자신의 박동 카운터를 증가시킨다.
    - 각 노드는 무작위로 선정된 노드들에게 주기적으로 자기 박동 카운터 목록을 보낸다.
    - 박동 카운터 목록을 받은 노드는 멤버십 목록을 최신 값으로 갱신
    - 어떤 멤버의 박동 카운터 값이 지정된 시간 동안 갱신되지 않으면 해당 멤버가 장애 상태인 것으로 간주한다.
    </aside>
    

#### 일시적 장애 처리

- 가십 프로토콜로 장애를 감지 했을 경우 → 가용성 보장을 위해 필요한 조치를 취함.
- 엄격한 정족수 접근법을 쓴다면 읽기와 쓰기 연산을 금지…
- 느슨한 정족수 → 조건을 완화하여 가용성을 높인다.
    - 쓰기연산을 수행할 W개의 건강한 서버와 읽기 연산을 수행할 R개의 건강한 서버를 해시 링에서 고른다.
        - 장애 서버는 무시
    - 네트워크나 서버 문제로 장애 상태인 서버로 가능 요청을 다른 서버가 맡아서 처리함.
    - 복구 되었을때 발생한 변경사항을 일괄 반영 → 일관성 보존
        - 이를 위해 쓰기 서버에서는 쓰기 연산 에 대한 단서를 남겨둠.
        
        > 단서 후 임시 위탁 기법 (hinted handoff)
        > 

#### 영구적 장애 처리

- 영구적인 장애 상태는 어떻게 처리해야할까?
- 반-엔트로피 프로토콜을 구현하여 사본들을 동기화
    - 사본들을 비교하여 최신 버전으로 갱신
    - 사본 간의 일관성이 망가진 상태를 탐지하고 전송 데이터의 양을 줄이기 위해 머클트리 사용

**머클트리란?**

각 노드에 그 자식 노드들에 보관된 값의 해시(자식 노드가 리프노드인 경우), 또는 자식 노드들의 레이블로부터 계산된 해시값을 레이블로 붙여두는 트리.

→ 즉 해시의 해시트리

데이터가 조금만 바뀌어도 해시가 바뀌므로, 루트 노드 비교로 각 서버의 데이터 사본이 동일한지 비교가 가능함.
루트가 다를 경우에는 자식노드로 내려가면서 비교를 하면 데이터가 다른 부분을 logN 타임에 확인가능.
⇒ 해당 부분만 동기화를 해주면 된다.
