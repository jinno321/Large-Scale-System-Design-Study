수평적 규모 확장성을 달성하기 위해서는 요청 또는 데이터를 서버에 균등하게 분배하는 것이 중요하다.
안정 해시는 이러한 목표를 안정적이고, 나중의 위기를 대비하기에 좋은 해시 전략이라고 생각한다.

---

### 해시 키 재배치(Refresh) 문제

이전에 로드밸런서 얘기를 하며 키에 대한 분배가 중요하다고 말한 적이 있다. 캐시에서도 마찬가지다. 캐시 서버에 부하를 균등하게 나누는 방법은 `%` 연산자를 사용하는 것이다.

$$\text{serverIndex} = \text{hash(key)} \pmod N \quad (N = \text{서버 개수})$$

이렇게 되면 $N$개의 서버에 고르게 트래픽이 분배가 될 것이다. 하지만 이것이 잘 동작하는 경우는 서버 풀의 크기가 고정되어 있을 때이다. 서버 수가 추가되거나 기존 서버가 삭제되면 문제가 생긴다. 

예를 들어, 1번 서버가 장애를 일으켜 동작을 중단한다면, $N$이 $N-1$이 될 것이다. 이렇게 되면 나머지 상관없는 모든 키의 서버를 교체해 주어야 한다. 결국 대규모 캐시미스가 발생하게 될 것이고, 이는 서비스에 중대한 영향을 미칠 것이다.

## 안정 해시

안정 해시는 해시 키 재배치 문제를 최소화시키며 평균적으로 오직 $\frac{k}{n}$ 개의 키만 재배치하는 해시 기술이다. ($k = \text{키의 개수}$, $n = \text{슬롯의 개수}$)

### 해시 공간과 해시 링
SHA-1을 해시함수로 사용한다고 가정하자. 해시 공간의 범위는 $0 \sim 2^{160}-1$이다. 그렇다면 해시값은 이 사이를 가리키게 된다. 이 공간의 양 끝을 구부려 붙이면 **해시 링(Hash Ring)**이 된다.

### 해시 서버와 키
해시 함수 $f$를 사용해서 서버 IP나 이름을 링 위의 어떤 위치에 대응시킬 수 있다. 이때 해시 키는 모듈러 연산을 따로 하지 않은 순수 키이다. 어떤 키가 저장되는 서버는, 해당 키의 위치로부터 **시계방향**으로 링을 탐색해 나가면서 만나는 첫 번째 서버다.

### 서버 추가와 제거
이때, 서버를 추가하더라도 키 가운데 일부만 재배치하면 된다. 서버가 추가되어도 추가된 서버와 반시계방향의 가장 가까운 서버 사이에 있는 키값만 조정을 해주면 된다. 제거될 때도 마찬가지다. 서버가 제거되면 자신을 가리키고 있는 키값들만 재배치 해주면 된다.

### 기본 구현법의 두 가지 문제
안정 해시 알고리즘의 기본 절차는 다음과 같다.
* 서버와 키를 균등 분포 해시 함수를 사용해 해시 링에 배치한다.
* 키의 위치에서 링을 시계 방향으로 탐색하다 만나는 최초의 서버가 키가 저장될 서버다.

여기에도 문제가 있다. 서버가 추가, 삭제되는 상황을 감안하면 파티션의 크기를 균등하게 유지하는 것은 불가능하다. 어떤 서버는 작은 해시 공간을, 어떤 서버는 큰 해시 공간을 부여받을 것이다. 

또한, 키의 균등 분포를 달성하기가 어렵다. 어떤 서버는 아무 데이터도 갖지 않을 수도 있고, 어떤 서버는 대부분의 키를 부여받을 수 있다. 이를 해결하기 위해 **가상 노드(Virtual Node)** 또는 복제 기법이 제안되었다.

### 가상 노드
가상 노드는 실제 노드 혹은 그 서버를 가리키는 노드이다. 하나의 서버는 링 위에 여러 개의 가상 노드를 가진다. 

원래 서버 0을 링에 배치하기 위해 `s0` 하나만 쓰는 대신, `s0_0`, `s0_1`, `s0_2`와 같이 세 개의 가상 노드를 사용하는 것이다. 따라서 각 서버는 하나가 아닌 여러 개의 파티션을 관리해야 한다. 이렇게 되면 키의 위치로부터 시계 방향으로 링을 탐색하다 만나는 최초의 '가상 노드'가 해당 키가 저장될 서버가 되는 것이다. 가상 노드의 개수를 늘리면 키의 분포는 표준 편차가 작아져서 점점 더 균등해질 것이다.

**💡 가상 노드 개수와 표준 편차의 트레이드오프**
100~200개의 가상 노드를 사용했을 경우 표준 편차 값은 5% ~ 10% 사이다. 그러나 이렇게 가상 노드를 늘릴수록 가상 노드 데이터를 저장할 공간은 더 많이 필요해질 것이다. 이때 트레이드오프가 필요하다.

### Discord에서 해시 링 사용

앞서 말한 안정 해시 링은 이론만이 아니라 실제 대규모 서비스에서도 널리 쓰인다. 대표적으로 디스코드(Discord)가 수백만 명의 동시 접속자를 버티기 위해 이 방식을 적극적으로 도입했다.

디스코드는 수많은 노드로 이루어진 분산 시스템 환경이라, 특정 디스코드 서버(길드) 데이터가 정확히 어느 원격 노드에 있는지 찾아내는(Lookup) 인프라가 필요했다. 이를 위해 전체 시스템 수준에서 안정 해시 기반의 링 구조를 만들어 관리하고 있다.

여기에 추가로 디스코드는 대규모 메시지 전송 부하를 줄이기 위해 자체 개발한 `Manifold` 라이브러리를 쓴다. 메시지를 보낼 대상 프로세스들을 노드별로 그룹화한 뒤, 각 노드 안에서 안정 해시 함수(`:erlang.phash2/2`)를 돌려 CPU 코어 수에 맞춰 균등하게 분할(Hashing)해 병렬로 처리한다. 결국 안정 해시 덕분에 파티션 크기를 균등하게 유지하며 CPU 부하를 완벽히 분산시킬 수 있었던 것이다.

이 안정 해시 링을 조회하는 속도마저 아까워 디스코드는 `FastGlobal`이라는 라이브러리를 만들어 링 구조 자체를 읽기 전용 공유 힙에 상주시키는 최적화까지 진행했다.이전에는,데이터를 주고받거나 공용 저장소에서 데이터를 읽어올 때마다 매번 데이터를 자신의 공간으로 복사해야 하는데, 이러한 시간을 런타임에 동적으로 빌드 시켜서 12up => 0.3up 로 급격하게 줄였다.
