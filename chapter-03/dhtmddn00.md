# 안정 해시(Consistent Hashing)

## 안정 해시란?

해시 테이블이 재조정될 때 평균 k/n개의 키만 재배치되도록 설계된 기술이다. (k = 전체 키 수, n = 슬롯 수) 수평적 규모 확장을 위해 요청이나 데이터를 서버에 균등하게 나눌 때 보편적으로 사용된다.

## 왜 필요한가

N개의 캐시 서버에 부하를 균등하게 나누는 보편적인 방법은 다음과 같다.

```
serverIndex = hash(key) % N
```

이 방식은 서버 수가 고정되어 있을 때는 잘 동작한다. 문제는 서버가 추가되거나 삭제될 때 발생한다.

```
서버 4대일 때: hash(key) % 4
서버 3대로 줄면: hash(key) % 3

→ 거의 모든 키의 serverIndex가 달라짐
→ 캐시 서버에 저장된 데이터를 찾지 못하고 원본 DB에 요청이 쏟아짐
→ 대규모 캐시 미스 발생
```

서버 수가 자주 변하는 대규모 분산 환경에서 이 방식은 치명적이다. 안정 해시는 이 문제를 해결하기 위해 나온 기술이다.

## 안정 해시 알고리즘

### 해시링

해시 공간을 0부터 2^32-1까지의 값을 갖는 링(원형)으로 표현한다. SHA-1 같은 균등 분포 해시 함수를 사용하면 서버와 키 모두 이 링 위의 어떤 지점에든 배치될 수 있다.

```
서버를 링에 배치:
hash(서버 IP 또는 이름) → 링 위의 위치

키를 링에 배치:
hash(key) → 링 위의 위치
```

### 키의 서버 결정 방법

키의 위치에서 링을 시계 방향으로 탐색하다 만나는 최초의 서버가 그 키를 저장할 서버다.

```
링 위 배치 (시계 방향):
... → 키A → 서버B → 키C → 서버D → ...

키A는 시계 방향으로 탐색했을 때 처음 만나는 서버B에 저장
키C는 시계 방향으로 탐색했을 때 처음 만나는 서버D에 저장
```

### 서버 추가/삭제 시 재배치 최소화

서버가 추가되거나 삭제될 때, 영향을 받는 키는 해당 서버의 인접 구간에 있는 키들뿐이다. 나머지 키는 그대로 같은 서버에 남아있는다.

```
서버C가 삭제되면:
→ 서버C가 담당하던 키들만 다음 서버(D)로 이동
→ 나머지 키들은 변화 없음
```

## 기본 안정 해시의 문제

### 파티션 크기 불균등

해시 함수는 서버를 링 위에 무작위로 배치한다. 서버 수가 적을 때는 운에 따라 어떤 서버는 링의 큰 구간을, 어떤 서버는 작은 구간을 담당할 수 있다.

```
서버 3대 배치 예시 (링을 0~360도로 표현):
A(30도), B(80도), C(290도)

A의 담당 구간: 290도 ~ 30도  → 100도 (28%)
B의 담당 구간: 30도  ~ 80도  → 50도  (14%)
C의 담당 구간: 80도  ~ 290도 → 210도 (58%)
```

C에 트래픽의 58%가 몰리는 구조가 된다.

### 키의 균등 분포 달성의 어려움

파티션 크기가 불균등하면, 키가 링 위에 아무리 균등하게 뿌려져도 서버별 키 수는 불균등해진다. 위 예시에서 키의 58%가 C에 몰린다.

## 가상 노드(Virtual Node)

### 개념

서버 하나를 링 위에 여러 지점에 배치하는 기법이다. 서버 A를 3개의 가상 노드로 표현하면 A1, A2, A3이 링의 서로 다른 위치에 배치된다.

```
가상 노드 3개씩 배치:
A1(30도), A2(150도), A3(270도)
B1(90도), B2(200도), B3(320도)
C1(50도), C2(170도), C3(240도)

→ 링 위에 9개의 포인트가 고르게 분포
→ A, B, C 각각의 담당 구간이 대략 1/3에 근접
```

키를 저장할 서버를 찾을 때는 동일하게 시계 방향으로 탐색해서 처음 만나는 가상 노드의 실제 서버에 저장한다.

### 왜 균등 분포가 되는가

핵심은 **대수의 법칙(Law of Large Numbers)** 이다.

해시 함수는 입력값을 링 위에 무작위로 배치한다. 서버 포인트가 3개뿐일 때는 이 무작위성이 불균등하게 나타날 확률이 높다. 동전을 3번 던지면 앞/앞/앞이 나올 수도 있는 것처럼.

하지만 포인트가 300개가 되면, 무작위 배치라도 링 전체에 골고루 퍼질 가능성이 압도적으로 높아진다. 동전을 300번 던지면 앞이 150번 전후에서 수렴하는 것과 같은 원리다.

수치로 표현하면, 가상 노드 수를 V라 할 때 각 서버의 담당 구간 편차는 1/√V에 비례해서 줄어든다.

```
V = 100일 때 표준편차: σ
V = 400일 때 표준편차: σ/2   (4배 늘리면 편차 절반)
```

실무에서 노드당 100~200개 정도면 충분히 균등하다고 본다.

### 트레이드오프

가상 노드 수를 V, 서버 수를 N이라 하면:

- 링 위의 포인트 관리에 O(V × N) 메모리 필요
- 키의 서버 탐색 비용 O(log(V × N))

V가 커질수록 분포는 균등해지지만 메모리와 탐색 비용이 늘어난다. 균등 분포와 자원 사용량 사이의 트레이드오프가 존재한다.

## 안정 해시의 이점

서버가 추가되거나 삭제될 때 재배치되는 키의 수가 최소화된다. 단순 해시(hash % N)는 서버 수 변경 시 거의 모든 키가 재배치되지만, 안정 해시는 평균 k/n개만 재배치된다.

데이터가 보다 균등하게 분포하게 되므로 수평적 규모 확장이 쉬워진다. 서버를 추가해도 기존 서버들의 부하가 자연스럽게 분산된다.

핫스팟 키 문제를 줄인다. 특정 샤드에 대한 접근이 지나치게 빈번하면 서버 과부하 문제가 생길 수 있는데, 균등한 분포가 이를 완화한다.

## 실제 사용 사례

안정 해시는 대규모 분산 시스템에서 널리 쓰인다.

- 아마존 다이나모DB(DynamoDB): 데이터 파티셔닝에 안정 해시 사용
- 아파치 카산드라(Cassandra): 클러스터 노드 간 데이터 분산
- 디스코드(Discord): 채팅 메시지 라우팅
- 아카마이(Akamai) CDN: 엣지 서버 부하 분산
- 매그레프(Maglev): 구글의 네트워크 부하 분산기
