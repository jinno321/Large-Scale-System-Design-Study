# 분산 키-값 저장소(Distributed Key-Value Store)

## 문제 이해 및 설계 범위 확정

설계에 앞서 요구사항을 확정한다.
- 키-값 쌍의 크기는 10KB 이하
- 큰 데이터를 저장할 수 있어야 한다
- 높은 가용성(availability)을 제공해야 한다
- 높은 규모 확장성(scalability)을 제공해야 한다
- 데이터 일관성(consistency) 수준은 조정이 가능해야 한다
- 응답 지연시간이 짧아야 한다

## 단일 서버 키-값 저장소

가장 단순한 구현은 키-값 쌍 전부를 메모리에 해시 테이블로 저장하는 것이다. 빠른 속도는 보장하지만, 모든 데이터를 메모리 안에 두는 것은 현실적으로 불가능하다.

개선책으로는 데이터를 압축하거나, 자주 쓰이는 데이터만 메모리에 두고 나머지는 디스크에 저장하는 방법이 있다. 하지만 이렇게 해도 한 대의 서버로는 결국 부족해진다. 그래서 데이터를 여러 서버에 나눠 저장하는 분산 키-값 저장소가 필요하다. 이를 분산 해시 테이블(distributed hash table)이라고도 부른다.

## CAP 정리

분산 시스템을 설계할 때는 CAP 정리를 반드시 이해해야 한다. CAP 정리는 데이터 일관성(Consistency), 가용성(Availability), 파티션 감내(Partition tolerance) 세 가지를 동시에 만족하는 분산 시스템 설계는 불가능하다는 정리다.

- **데이터 일관성**: 모든 클라이언트는 어떤 노드에 접속하든 항상 같은 데이터를 본다.
- **가용성**: 일부 노드에 장애가 발생하더라도 클라이언트는 항상 응답을 받을 수 있다.
- **파티션 감내**: 두 노드 사이에 통신 장애(파티션)가 생기더라도 시스템은 계속 동작한다.

세 가지 중 두 가지만 충족할 수 있고, 나머지 하나는 반드시 희생해야 한다.

- **CP 시스템**: 일관성 + 파티션 감내. 가용성을 희생.
- **AP 시스템**: 가용성 + 파티션 감내. 일관성을 희생.
- **CA 시스템**: 일관성 + 가용성. 파티션 감내를 포기. 그러나 네트워크 장애는 피할 수 없는 일이므로 분산 시스템은 반드시 파티션을 감내하도록 설계되어야 한다. 따라서 실세계에 CA 시스템은 존재하지 않는다.

### 왜 "파티션(partition)"이라는 이름일까

네트워크의 일시적 장애를 왜 하필 "파티션"이라 부르는지 헷갈리기 쉽다. 여기서 파티션은 "저장 공간을 나눈다"는 디스크 파티션의 의미가 아니라, **네트워크가 분단(分斷)된다**는 의미다.

노드 사이의 통신이 끊기면, 전체 클러스터가 서로 연결된 하나의 그룹이 아니라 **서로 통신하지 못하는 여러 개의 섬(분단된 그룹)으로 쪼개진다.** 예를 들어 노드 5개 중 통신 장애로 {n1, n2}와 {n3, n4, n5}가 서로 못 보게 되면, 네트워크가 두 개의 파티션으로 나뉜 것이다. 즉 "네트워크 그래프가 둘 이상으로 분할된 상태"를 가리키는 말이라서 파티션이라고 부른다.

### 이상적 상태 vs 실세계

이상적이라면 네트워크 파티션은 절대 일어나지 않고, n1에 기록된 데이터는 자동으로 n2, n3에 복제된다.

하지만 실세계의 분산 시스템에서는 반드시 네트워크 파티션이 생긴다. 파티션이 발생하면 일관성과 가용성 중 하나를 선택해야 한다.

- **일관성을 택하면**: 데이터 불일치를 막기 위해 쓰기 연산을 전부 금지한다. 데이터는 항상 정확하지만 가용성을 잃는다.
- **가용성을 택하면**: 읽기/쓰기를 계속 허용한다. 대신 파티션 때문에 사본 간 동기화가 안 되므로 클라이언트가 낡은 데이터를 볼 수 있다.

### 파티션 감내는 미래에 해결될 수 있을까

파티션 자체를 없애는 건 물리 법칙상 불가능에 가깝다. 네트워크 파티션은 결국 노드 사이에 메시지가 제때 못 가는 상황인데, 그 원인(빛의 속도 한계, 중간 장비 고장, 소프트웨어 버그, 자연재해 등)은 100% 제거가 불가능하다. 그래서 CA 시스템이라는 선택지가 실세계에서 사라진 것이다.

대신 발전하는 방향은 트레이드오프를 더 정교하게 다루는 쪽이다.

- **파티션 확률 자체를 낮추기**: 구글의 Spanner가 대표적이다. 전용 광케이블 네트워크와 GPS·원자시계(TrueTime)로 데이터센터 간 시간을 극도로 정밀하게 맞춰, 파티션 발생 확률을 극단적으로 낮춘다.
- **PACELC 정리**: CAP는 파티션이 났을 때만 다루지만, 현실에선 파티션이 없는 평상시에도 트레이드오프가 있다. PACELC는 "파티션이 있으면(P) 가용성(A) vs 일관성(C), 그렇지 않으면(E) 지연시간(L) vs 일관성(C)"으로 이를 확장한다.
- **CRDT(Conflict-free Replicated Data Type)**: 충돌이 수학적으로 절대 안 나도록 설계된 자료구조. 파티션이 나도 각자 쓰고, 나중에 어떤 순서로 합쳐도 같은 결과가 나온다.

## 시스템 컴포넌트

분산 키-값 저장소는 다음 요소들로 구성된다. 다이나모(DynamoDB), 카산드라(Cassandra), 빅테이블(Bigtable)의 사례를 참고한다.

- 데이터 파티션
- 데이터 다중화(복제)
- 일관성
- 일관성 불일치 해소
- 장애 처리
- 시스템 아키텍처
- 쓰기 경로 / 읽기 경로

## 데이터 파티션

대규모 애플리케이션이면 전체 데이터를 한 대 서버에 욱여넣는 건 불가능하다. 두 가지를 만족해야 한다.

- 데이터를 여러 서버에 고르게 분산
- 노드가 추가/삭제될 때 데이터 이동을 최소화

**안정 해시(consistent hashing)** 가 이 문제를 푸는 데 적합하다.

- **규모 확장 자동화**: 시스템 부하에 따라 서버가 자동으로 추가/삭제되도록 할 수 있다.
- **다양성(heterogeneity)**: 각 서버의 용량에 맞게 가상 노드 수를 조정할 수 있다.

## 데이터 다중화(Replication)

높은 가용성과 안정성을 확보하려면 데이터를 비동기적으로 N개의 서버에 다중화한다. 안정 해시 링에서 키의 위치로부터 시계 방향으로 순회하면서 만나는 첫 N개 서버에 데이터 사본을 보관한다.

같은 데이터 센터 노드들은 정전, 네트워크 이슈, 자연재해 등의 문제를 동시에 겪을 수 있으므로, 사본은 서로 다른 데이터 센터에 보관한다.

> AWS의 가용 영역(AZ)이 정확히 이 개념이다. 한 AZ에 정전이나 재해가 나도 다른 AZ의 사본으로 서비스를 이어갈 수 있도록, 사본을 여러 AZ에 분산 배치하는 것이 표준 패턴이다.

## 데이터 일관성

다중화된 데이터는 정족수 합의(quorum consensus) 프로토콜로 일관성을 보장한다.

- **N**: 사본의 개수
- **W**: 쓰기 정족수. W개 서버로부터 쓰기 성공 응답을 받아야 성공으로 간주.
- **R**: 읽기 정족수. R개 서버로부터 응답을 받아야 성공으로 간주.

### W + R > N 이면 강한 일관성

`W + R > N`이면 쓰기에 참여한 W개 집합과 읽기에 참여한 R개 집합이 반드시 최소 하나 겹친다. 그 겹치는 노드가 최신 데이터를 갖고 있으므로 읽기는 항상 최신 값을 보게 된다.

```
N = 3, W = 2, R = 2 → W + R = 4 > 3 → 강한 일관성
쓰기: {n1, n2} 성공
읽기: {n2, n3} 수행
→ n2가 겹침 → 최신 데이터 보장
```

정족수 설정에 따른 특성:
- `R = 1, W = N`: 빠른 읽기에 최적화
- `W = 1, R = N`: 빠른 쓰기에 최적화
- `W + R > N`: 강한 일관성 보장
- `W + R <= N`: 강한 일관성 보장 안 됨

## 일관성 모델

- **강한 일관성(strong consistency)**: 모든 읽기 연산이 가장 최근에 갱신된 결과를 반환한다.
- **약한 일관성(weak consistency)**: 읽기 연산이 최신 갱신 결과를 반환하지 못할 수 있다.
- **결과적 일관성(eventual consistency)**: 약한 일관성의 한 형태로, 갱신 결과가 결국에는 모든 사본에 반영된다.

> 결과적 일관성은 "지금 당장은 사본마다 값이 다를 수 있지만, 새로운 쓰기가 멈추고 시간이 충분히 지나면 결국 모든 사본이 같은 값으로 수렴한다"는 뜻이다. SNS 프로필 사진 변경처럼 잠깐의 불일치는 허용하되 언젠가는 맞춰지는 경우가 대표적이다.

강한 일관성을 보장하는 일반적 방법은 모든 사본에 현재 쓰기 결과가 반영될 때까지 읽기/쓰기를 금지하는 것인데, 고가용성 시스템에는 적합하지 않다.

## 일관성 불일치 해소: 데이터 버저닝

### 벡터 시계(Vector Clock)

벡터 시계는 버전 간 선후 관계를 추적하기 위한 도구다. `[서버, 버전]` 순서쌍의 목록을 데이터에 매달아둔다.

```
D([Sx, 1])         → 서버 Sx가 1번 갱신한 버전
D([Sx, 2], [Sy, 1]) → Sx가 2번, Sy가 1번 갱신한 버전
```

한 벡터 시계가 다른 쪽을 "포함"하면 선후 관계가 명확해 자동으로 최신 것을 택한다. 포함 관계가 아니면 **충돌(conflict)** 이 발생한 것으로 별도의 해소가 필요하다.

### 벡터 시계의 단점

1. **충돌 해소 로직이 클라이언트에 들어가야 한다.** 충돌이 났다는 건 어느 쪽이 맞는 값인지 시스템이 자동으로 결정할 수 없다는 의미다. 데이터의 의미를 아는 애플리케이션(클라이언트)만 판단할 수 있으므로 클라이언트 구현이 복잡해진다.
2. **순서쌍 개수가 빠르게 늘어난다.** 임계치 이상으로 길어지면 오래된 쌍을 제거해야 하는데, 그러면 버전 간 선후 관계를 정확히 결정할 수 없게 된다.

## 장애 처리

### 장애 감지

분산 시스템에서는 두 대 이상이 똑같이 서버 A의 장애를 보고해야 실제 장애로 간주한다. 보고한 노드 자신의 네트워크가 문제일 수 있기 때문이다.

**가십 프로토콜(gossip protocol)** 이 효율적인 분산형 장애 감지 솔루션이다.

- 각 노드는 `[멤버 ID, 박동(heartbeat) 카운터]` 쌍의 멤버십 목록을 유지한다.
- 주기적으로 자신의 박동 카운터를 증가시키고, 무작위로 선정된 노드들에게 박동 카운터 목록을 보낸다.
- 어떤 멤버의 박동 카운터가 지정된 시간 동안 갱신되지 않으면 장애 상태로 간주한다.

모든 노드가 모든 노드에 직접 연결되지 않아도 무작위 전파만으로 정보가 클러스터 전체에 퍼지므로 확장성이 좋다.

### 일시적 장애 처리: 단서 후 임시 위탁(Hinted Handoff)

장애 상태인 서버로 가야 할 요청을 다른 서버가 잠시 대신 처리한다. 임시로 처리한 서버는 "원래 누구 것인지" 단서(hint)를 남겨두고, 원래 서버가 복구되면 그동안 쌓인 변경분을 인계한다.

```
노드 s2가 장애
→ s2로 갈 읽기/쓰기를 s3가 임시로 처리 (s3는 "이건 s2 것" 단서를 남김)
→ s2가 복구되면 s3가 그동안 쌓인 변경분을 s2로 인계
```

### 영구 장애 처리: 반-엔트로피 프로토콜과 머클 트리

영구적인 장애는 **반-엔트로피(anti-entropy) 프로토콜**로 사본들을 다시 동기화한다. 사본 간 불일치를 탐지하고 전송 데이터 양을 줄이기 위해 **머클 트리(Merkle tree)** 를 사용한다.

머클 트리는 데이터를 잘게 나눠 각 조각의 해시값을 잎으로 두고, 자식 노드들의 해시를 합쳐 해시한 값을 부모 노드로 올리는 트리다. 루트 해시 하나가 전체 데이터를 대표한다.

```
                루트 해시
              /          \
        해시(AB)        해시(CD)
        /     \         /     \
    해시A   해시B    해시C   해시D
```

두 사본을 비교할 때 루트 해시만 먼저 비교하고, 다르면 자식으로 내려가며 다른 가지만 추적한다. 실제로 차이 나는 데이터 조각만 골라서 전송하므로 비교/전송 비용이 크게 줄어든다.

```
1. 루트 해시 비교 → 다름
2. 해시(AB) 같음, 해시(CD) 다름 → CD 쪽만 파고듦
3. 해시C 같음, 해시D 다름 → 조각 D만 동기화
→ A, B, C는 건드리지 않음
```

### 데이터 센터 장애 처리

데이터 센터 전체가 한꺼번에 내려갈 수도 있으므로, 데이터를 여러 데이터 센터에 다중화하는 것이 중요하다.

## 시스템 아키텍처

클라이언트는 두 가지 API와 통신한다.
- `get(key)`: 키에 해당하는 값을 읽는다
- `put(key, value)`: 키-값 쌍을 저장한다

핵심 특징:
- **중재자(coordinator)**: 클라이언트 요청을 받아 적절한 노드로 라우팅하는 프락시 노드
- 노드는 안정 해시를 통해 해시 링 위에 분포
- 완전히 분산(decentralized)되어 단일 장애점(SPOF) 없음
- 모든 노드가 동일한 책임을 짐

## 쓰기 경로

```
쓰기 요청
  ├──→ [디스크] 커밋 로그(commit log)에 기록
  └──→ [메모리] 멤테이블(memtable)에 기록
              ↓ (멤테이블이 가득 차면)
       [디스크] SSTable로 플러시(flush)
```

1. **커밋 로그**: 디스크에 순차적으로 쌓는 로그. 서버가 죽어도 데이터를 복구할 수 있는 안전장치.
2. **멤테이블**: 메모리상의 정렬된 자료구조. 빠른 쓰기를 위해 메모리에 먼저 기록.
3. **SSTable(Sorted-String Table)**: 멤테이블이 가득 차면 디스크로 내려씀. 키가 정렬된 상태로 저장되며 한 번 쓰면 수정하지 않는 불변(immutable) 파일. 순차 쓰기만 발생하므로 쓰기 성능이 좋다.

## 읽기 경로

먼저 멤테이블(메모리)을 확인하고, 없으면 디스크의 SSTable에서 가져온다. SSTable이 여러 개 쌓여 있을 때 어느 SSTable에 키가 있는지 효율적으로 찾기 위해 **블룸 필터(Bloom filter)** 를 사용한다.

블룸 필터는 "어떤 원소가 집합에 확실히 없는지, 아니면 아마 있는지"를 빠르고 적은 메모리로 판별하는 확률적 자료구조다.

- **거짓 음성(false negative)은 없다**: "없다"고 하면 진짜 없는 것이므로 해당 SSTable은 안전하게 건너뛸 수 있다.
- **거짓 양성(false positive)은 있다**: 가끔 "있을 것 같다"고 했는데 실제로 없는 경우가 있지만 그 SSTable을 한 번 헛읽는 정도의 손해라 큰 문제가 아니다.

```
키 조회 시:
→ 블룸 필터에 물어봄
→ "확실히 없음" → 이 SSTable 건너뜀
→ "아마 있음" → 이 SSTable 실제로 확인
```

## 기능과 구현 기술 요약

| 기능 / 목표 | 사용 기술 |
|---|---|
| 대규모 데이터 저장 | 안정 해시로 서버들에 부하 분산 |
| 읽기 연산의 높은 가용성 | 데이터를 여러 데이터 센터에 다중화 |
| 쓰기 연산의 높은 가용성 | 버저닝 및 벡터 시계를 통한 충돌 해소 |
| 데이터 파티션 | 안정 해시 |
| 점진적 규모 확장성 | 안정 해시 |
| 다양성(서버 용량별 부하 조정) | 안정 해시(가상 노드 수 조정) |
| 조절 가능한 데이터 일관성 | 정족수 합의(W/R 조정) |
| 일시적 장애 처리 | 느슨한 정족수 + 단서 후 임시 위탁 |
| 영구적 장애 처리 | 머클 트리 기반 반-엔트로피 프로토콜 |
| 데이터 센터 장애 대응 | 여러 데이터 센터에 걸친 데이터 다중화 |

## 정리

분산 키-값 저장소는 단일 서버의 한계를 넘기 위해 데이터를 여러 노드에 분산·복제하는 시스템이다. CAP 정리상 파티션 감내는 필수이므로, 일관성과 가용성 사이에서 선택하게 된다. 데이터 분산에는 안정 해시, 사본 동기화에는 정족수와 벡터 시계, 장애 감지에는 가십 프로토콜, 장애 복구에는 단서 후 임시 위탁과 머클 트리 기반 반-엔트로피 프로토콜이 쓰인다. 다이나모, 카산드라, 빅테이블이 이런 설계의 대표 사례다.
