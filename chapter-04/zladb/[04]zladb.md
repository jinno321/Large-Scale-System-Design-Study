### 처리율 제한 알고리즘

#### 토큰 버킷 알고리즘

- 토큰 버킷 → 지정된 용량을 갖는 컨테이너
- 사전 설정된 양이 주기적으로 채워지고 꽉차면 더이상 추가되지 않음.
    - 각 요청이 처리도리 때마다 하나의 토큰을 사용.
    - 요청이 도착하면 버킷에 충분한 토큰이 있는지 검사함.
        - 있으면 토큰을 하나 꺼내고 요청을 시스템에 전달
        - 토큰이 없으면 해당 요청은 그냥 버려짐
    - 토큰 공급률을 정해서 초당 최대 몇개의 토큰(요청)이 사용될 수 있는지 제한함
    
    > 버킷 크기(버킷 최대 토큰 개수) + 토큰 공급률(초당 몇 개의 토큰이 버킷에 공급되는가?)
    > 
- 통상적으로 API 엔드포인트마다 별도의 버킷을 둔다.
    - 하루에 150명까지만 팔로잉 할 수 있음
    - 좋아요 버튼을 5개까지 누를 수 있음
    
    → 이거 두개 구현하려면 사용자마다 2개의 버킷을 두어야함.
    
    - IP 주소별로 처리율 제한가능
    - 시스템 처리율을 제한하려면 모든 요청이 하나의 버킷을 공유하게 함

<aside>

장점

- 구현 쉬움
- 메모리 사용 효율적
- 짧은 시간에 집중되는 트래픽 처리 가능 → 버킷에 토큰만 있으면 요청 전달가능

단점

- 버킷 크기와 토큰 공급률이라는 두가지의 인자를 적절하게 튜닝하는게 까다로움
</aside>

---

#### 누출 버킷 알고리즘

- 토큰 버킷 알고리즘과 비슷하나, 요청 처리율이 고정되어있음.
- 보통 FIFO로 구현

<aside>

1. 요청이 도착하면 큐가 가득 차 있는지 확인. 빈자리가 있으면 큐에 요청 추가
2. 큐가 가득 차있으면 새 요청 버림
3. 지정된 시간마다 큐에서 요청을 꺼내어 처리함
</aside>

> 버킷 크기 (큐 사이즈) + 처리율(지정된 시간당 몇 개의 항목을 처리할지? → 초단위)
> 

<aside>

장점

- 큐의 크기가 제한되어 있어 메모리 사용량 측면에서 효율적
- 고정된 처리율 → 안정적 출력이 필요한 경우 적합

단점

- 단시간에 많은 트래픽이 몰리면 큐에 오래된 요청들이 쌓이고, 그 요청을 제때 처리 못하면 최신 요청들이 다 버려짐
- 두 개의 인자를 튜닝하기 까다로움
</aside>

### 고정 윈도 카운터 알고리즘

- 타임라은을 고정된 간격의 윈도로 나누고, 각 윈도마다 카운터를 붙인다.
- 요청이 접수될 때마다 카운터의 값이 1 증가한다.
- 카운터의 값이 사전에 설정된 임계치에 도달하면 새로운 요청은 새 윈도가 열릴 때 까지 버려진다.

문제점 → 윈도의 경계 부근에 순간적으로 많은 트래픽이 집중될 경우, 윈도에 할당된 양보다 더 많은 요청이 처리될 수 있음.

<aside>

장점

- 메모리 효율이 좋음
- 이해하기 쉬움
- 윈도가 닫히는 시점에 카운터를 초기화하는 방식은 특정한 트래픽 패턴을 처리하기에 적합

단점

- 윈도 경계 부근에서 일시적으로 많은 트래픽이 몰려드는 경우 → 기대했던 시스템의 처리 한도보다 많은 양의 요청을 처리하게 됨.
</aside>

---

#### 이동 윈도 로깅 알고리즘

- 고정 윈도 카운터 알고리즘의 단점을 여기서 해결
- 요청의 타임스탬프를 추적함.
    - 레디스나 정렬 집합 같은 캐시에 저장
    
    > 타임 스탬프를 남기니까 로깅 알고리즘
    > 
- 새 요청이 오면 만료된 타임스탬프 제거.
    - 만료된 타임스탬프란 ? → 현대 윈도의 시작 시점보다 오래된 타임스탬프를 뜻함
- 새 요청의 타임스탬프를 로그에 추가
- 로그의 크기가 허용치보다 같거나 작으면 요청을 시스템에 전달
    - 그렇지 않으면 거부!

<aside>

장점

- 어느 윈도우에서든지 허용되는 요청의 개수는 시스템의 처리율 한도를 넘기지 않는다.

단점

- 다량의 메모리 사용 → 거부된 요청의 타임스탬프도 보관하기 때문
</aside>

---

#### 이동 윈도 카운터 알고리즘

- 고정 윈도 카운터 알고리즘 + 이동 윈도 로깅 알고리즘
- 연속적으로 이동하는 윈도우 + 그 윈도우에 카운터 알고리즘

<aside>

장점

- 이전 시간대의 평균 처리율에 따라 현재 윈도의 상태를 계산함. ⇒ 짧은 시간에 몰리는 트래픽에도 잘 대응
- 메모리 효율이 좋음

단점

- 직전 시간대에 도착한 요청이 균등하게 분포되어 있다고 가정한 상태에서 추정치를 계산. (다소 느슨)
    - 실제로 심각한건 아님. 클라우드플레어 시험에 따르면 0.003퍼만 문제가됨.
</aside>

- 카운터를 어디에 보관할 것인가?

    - REDIS → 메모리, 기한 만료 정책 지원(EXPIRE)

<aside>

동작 원리

- 클라이언트가 처리율 제한 미들웨어에게 요청을 보냄
- 처리율 제한 미들웨어는 레디스의 지정 버킷에서 카운터를 가져와서 한도에 도달했는지 아닌지 검사
    - 한도에 도달 → 거부
    - 도달 x → API 서버로 전달 + 카운터값 증가(INCR) 후 레디스에 저장
</aside>

#### 처리율 한도 초과 트래픽 처리

→ 한도 제한에 걸리면 API는 HTTP 429(too many requests)를 클라이언트로 보냄

**처리율 제한 장치가 사용하는 HTTP헤더**

- 클라이언트가 자기 요청이 처리율 제한에 걸리고 있는지 감지하는 방법

<aside>

X-Ratelimit-Remaining: 윈도 내에 남은 처리 가능 요청의 수

X-Ratelimit-Limit: 매 윈도마다 클라이언트가 전송할 수 있는 요청의 수

X-Ratelimit-Retry-After: 한도 제한에 걸리지 않으려면 몇 초 뒤에 요청을 다시 보내야 하는지 알림

</aside>

#### 상세설계

- 처리율 제한 규칙 → 디스크 보관
    - 작업 프로세스가 수시로 규칙을 디스크에서 읽어와 캐시에 저장
- 클라이언트가 요청을 서버에 보내면 요청은 먼저 처리율 제한 미들웨어에 도달
- 처리율 제한 미들웨어 → 제한 규칙을 캐시에서 가져옴 + 카운터 및 마지막 요청의 타임스탬프를 레디스 캐시에서 가져옴.
    - 해당 요청이 처리율 제한에 걸리지 않았나? → API 서버
    - 처리율 제한에 걸림 → 429 too many request 에러 클라이언트전송
        - 요청 버리거나 메시지 큐에 보관

#### 분산 환경에서의 처리율 제한 장치의 구현

발생하는 문제

1. 경쟁 조건
2. 동기화

#### 경쟁 조건

<aside>

1. 레디스에서 카운터 값을 읽는다
2. counter + 1의 값이 임계치를 넘는지 본다
3. 넘지 않는다면 레디스에 보관된 카운터 값을 1만큼 증가시킨다.
</aside>

- 해결 방법
    1. 락 사용
        - 시스템 성능을 상당히 떨어뜨림
    2. 루아스크립트 사용
        - 읽기와 쓰기를 하나로 묶어서 직렬 수행
    3. 정렬 집합 레디스 자료구조 사용
        - ㅇ

#### 동기화 이슈

- 분산 환경에서 고려해야할 중요한 요소.
- 수백만 사용자를 지원하려면 한 대의 처리율 제한 장치 서버로는 충분하지 않을 수 있음.
    - 처리율 제한 장치 서버 여러대 → 동기화 필요해짐
- 해결방법
    - 고정 세션 사용
        - 같은 클라이언트 요청 → 항상 같은 처리율 제한 장치로 전송
            - 수평 확장에 유연하지 않기 때문에 비추
    - 레디스와 같은 중앙 집중형 서버 사용
        - 굿

### 더 알면 좋은것

- 경성 또는 연성 처리율 제한
    - 경성 → 요청의 개수는 임계치를 절대 넘어설 수 없다.
    - 연성 → 요청 개수는 잠시 동안은 임계치를 넘어설 수 있다.
- 다양한 계층에서의 처리율 제한
    - 애플리케이션 계층에서 처리 (HTTP)
    - IP 주소에서 처리 (Iptables)
- 처리율 제한을 회피하는 방법 - 클라이언트를 어떻게 설계하는 것이 최선인가?
    - 클라이언트 측 캐시를 사용하여 API 호출 횟수를 줄인다
    - 처리율 제한의 임계치를 이해하고, 짧은 시간 너무 많은 메시지를 보내지 않도록 한다.
    - 예외나 에러를 처리하는 코드를 도입하여 클라이언트가 예외적 상황으로부터 우아하게 복구
    - 재시도 로직을 구현할 때 충분한 백오프 시간을 둔다.
