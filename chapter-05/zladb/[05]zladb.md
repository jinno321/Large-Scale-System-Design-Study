### 해시 키 재배치(rehash) 문제

`serverIndex = hash(key) % N`

- 위의 해시 함수를 통해 N개의 캐시 서버로 부하를 균등하게 나눔
- N은 서버의 개수

⇒ 서버 풀의 크기가 고정되어 있을때, 데이터 분포가 균등할때 잘 동작함.

- 하지만 서버가 추가되거나, 기존 서버가 삭제되면 문제가 발생
    
    > 1번 서버가 장애로 중단됨 → 서버 풀의 크기가 N-1로 변함. (슬롯 크기 변화)
    그 결과 키 해시 값에 대한 나머지 연산을 적용한 값(=서버 인덱스)이 달라짐
    → **모든 키가 다시 재배치 됨**.
    
    ⇒ 캐시 클라이언트가 데이터가 없는 엉뚱한 서버에 접속하게 되어 대규모 **캐시 미스**가 발생함
    > 

### 안정해시

- 테이블의 크기가 변경될 때 평균적으로 k/n개의 키만 재배치하는 해시 기술.
- K → 키의 개수, N → 슬롯의 개수

### 가상노드

**가상 노드(Virtual Node)란?**

하나의 물리 서버를 링 위의 여러 위치에 등록하는 기법입니다. 예를 들어 서버 A를 `A#1`, `A#2`, `A#3` 같은 이름으로 각각 **해시**해서 **서로 다른 각도에 배치**합니다. 키가 그 위치에 매핑되면, 내부적으로 모두 실제 서버 A로 라우팅됩니다.

**왜 필요한가?**

가상 노드가 1개(기본)일 때는 물리적 위치가 3곳 뿐이라 서버마다 담당하는 링 구간 크기가 제각각입니다. 어떤 서버는 큰 구간, 어떤 서버는 좁은 구간을 가져 부하가 불균등해집니다. 슬라이더를 올릴수록 각 서버의 구간이 잘게 쪼개져 링 전체에 고르게 퍼지고, 최대-최소 편차가 줄어드는 것을 확인할 수 있습니다.

**실제 시스템에서는?**

**Cassandra, DynamoDB, Redis Cluster** 같은 시스템에서 실제로 사용합니다. 보통 서버당 100~200개의 가상 노드를 설정하며, 서버 스펙이 다를 때 고성능 서버에 가상 노드를 더 많이 할당해 부하를 의도적으로 더 몰아줄 수도 있습니다.

### 안정 해시의 이점

- 서버가 추가되거나 삭제될 때 재배치되는 키의 수가 최소화된다.
- 데이터가 보다 균등하게 분포하게 되므로 수평적 규모 확장성을 달성하기 쉽다.
- 핫스팟 키 문제를 줄인다. (유명인의 데이터가 전부 같은 샤드에 몰리는 상황 등)
    - 데이터를 좀 더 균등하게 분배하므로 이런 문제가 생길 가능성을 낮춘다.
