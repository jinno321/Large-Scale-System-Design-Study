# URL 단축기(URL Shortener) 설계

## URL 단축기의 동작 예시

URL 단축기는 긴 URL을 짧은 URL로 바꿔주고, 짧은 URL로 접속하면 원래의 긴 URL로 리디렉션해주는 서비스다. (bit.ly, tinyurl 같은 서비스)

```
원본 URL: https://www.example.com/very/long/path?query=something&id=12345
단축 URL: https://tinyurl.com/qt6c2x9
→ 단축 URL 접속 시 원본 URL로 리디렉션
```

## 요구사항 및 개략적 추정

### 요구사항

- **트래픽 규모**: 매일 1억 개의 단축 URL 생성
- **단축 URL 길이**: 짧을수록 좋다

### 개략적 추정

```
쓰기 연산: 매일 1억 개
초당 쓰기 연산: 1억 / 24시간 / 3600초 ≈ 1160회/초
읽기:쓰기 비율을 10:1로 가정
초당 읽기 연산: 1160 × 10 ≈ 11,600회/초

10년간 운영 시 레코드 수:
1억 × 365 × 10 = 3650억 개

10년간 필요한 저장 용량:
3650억 × 100바이트 ≈ 36.5TB
```

읽기가 쓰기보다 훨씬 많은(읽기 중심) 시스템이라는 점이 설계에서 중요하다.

## API 엔드포인트

REST 스타일로 설계하며, 크게 두 개가 필요하다.

### 1. URL 단축용 엔드포인트

```
POST /api/v1/data/shorten
- 인자: {longUrl: 원본 URL}
- 반환: 단축 URL
```

### 2. URL 리디렉션용 엔드포인트

```
GET /api/v1/shortUrl
- 반환: HTTP 리디렉션. 목적지가 될 원본 URL
```

## URL 리디렉션 동작 절차

```
1. 클라이언트가 단축 URL(https://tinyurl.com/qt6c2x9)에 접속
2. 서버가 단축 URL에 대응하는 원본 URL을 찾음
3. 서버가 HTTP 리디렉션 응답(301 또는 302)을 반환
   → Location 헤더에 원본 URL을 담아서
4. 클라이언트(브라우저)가 그 원본 URL로 다시 요청
5. 원본 사이트에 도달
```

### 301 vs 302 리디렉션

- **301 (Permanent Redirect, 영구 이동)**: 브라우저가 응답을 캐시한다. 이후 같은 단축 URL 접속 시 서버를 거치지 않고 바로 원본 URL로 간다. 서버 부하를 줄여주지만, 클릭 통계(analytics) 수집이 어렵다.
- **302 (Temporary Redirect, 임시 이동)**: 브라우저가 캐시하지 않는다. 매번 서버를 거치므로 클릭 수 추적이 가능하다. 대신 서버 부하가 늘어난다.

서버 부하를 줄이는 게 목적이면 301, 클릭 통계가 중요하면 302를 선택한다.

## 상세 설계

### 데이터 모델

모든 <단축 URL, 원본 URL> 쌍을 해시 테이블에 저장하는 방법도 있지만, 데이터가 커지면 메모리에 다 담을 수 없고 비싸다. **관계형 데이터베이스에 저장하는 것이 더 좋다.**

```
url 테이블
| id | shortURL | longURL |
```

### 해시 함수 접근법

원본 URL을 해시 함수에 넣어 단축 URL(해시값)을 만드는 방식이다.

#### 해시값 길이

단축 URL은 [0-9, a-z, A-Z] 문자를 쓸 수 있고, 이는 총 62개 문자다.

```
62^n ≥ 3650억(10년치 레코드 수)
62^7 ≈ 3.5조 → 3650억보다 충분히 큼
→ 길이 7이면 충분
```

#### 해시 후 충돌 해소

MD5, SHA-1 같은 해시 함수의 결과에서 앞 7글자만 잘라 쓰면 충돌이 발생할 수 있다. 충돌이 나면 미리 정한 문자열을 원본 URL에 덧붙여 다시 해시하는 식으로 충돌이 없어질 때까지 반복한다. 확실하지만 매번 DB를 조회해 충돌 여부를 확인해야 해서 비용이 든다.

### base-62 변환 접근법

유일한 ID(10진수)를 62진수로 변환해서 단축 URL을 만드는 방식이다.

```
62진법 문자: 0-9(10개) + a-z(26개) + A-Z(26개) = 62개
예: ID 11157 를 62진법으로 변환
11157 → "2TX"
→ 단축 URL: https://tinyurl.com/2TX
```

### 두 접근법 비교

| | 해시 함수 | base-62 변환 |
|---|---|---|
| 단축 URL 길이 | 고정(7자) | ID 값에 따라 가변 |
| 충돌 | 발생 가능 → 해소 로직 필요 | 발생 안 함(유일 ID 기반) |
| 다음 단축 URL 예측 | 어려움(보안상 유리) | 순차 ID면 예측 가능(보안상 불리) |
| 유일 ID 생성기 필요 | 불필요 | 필요 |

base-62 변환은 충돌이 없다는 게 큰 장점이지만, 전역적으로 유일한 ID를 만들어주는 별도의 생성기가 필요하다.

## URL 단축 과정 상세

base-62 방식을 택했을 때 단축 URL을 만드는 흐름은 다음과 같다.

```
1. 입력: 원본 URL을 받는다
2. 확인: 이 원본 URL이 이미 DB에 있는지 조회
3-a. 있으면: DB에서 찾은 기존 단축 URL을 반환
3-b. 없으면:
   → 새로운 유일 ID 생성
   → 그 ID를 base-62로 변환해 단축 URL 생성
   → <ID, 단축 URL, 원본 URL>을 DB에 저장
   → 단축 URL 반환
```

### 유일 ID 생성기의 중요성

단축 URL을 만들 때 쓰는 ID는 **전역적으로 유일해야 한다.** 분산된 환경에서 이 유일성을 보장하는 건 어렵다. 그래서 앞서 다룬 **분산 유일 ID 생성기**(예: 트위터 스노플레이크)를 여기서 활용한다. ID 생성기가 유일한 번호를 발급하고, 그 번호를 base-62로 변환하는 구조다.

## URL 리디렉션 메커니즘 상세

리디렉션은 읽기 연산이고, 이 시스템은 읽기가 쓰기보다 훨씬 많다(약 10:1). 따라서 읽기 성능 최적화가 핵심이다.

### 캐시 활용

<단축 URL, 원본 URL> 쌍을 캐시에 저장해서 성능을 높인다. 자주 조회되는 URL을 캐시에 두면 DB 조회를 줄일 수 있다.

```
리디렉션 요청 처리 흐름:
1. 로드밸런서가 요청을 웹 서버 중 하나로 분산
2. 웹 서버가 먼저 캐시를 확인
3-a. 캐시에 있으면(cache hit): 바로 원본 URL 반환
3-b. 캐시에 없으면(cache miss):
   → DB에서 원본 URL 조회
   → 캐시에 저장(다음 요청을 위해)
   → 원본 URL 반환
4. 클라이언트를 원본 URL로 리디렉션
```

### 로드밸런서의 역할

초당 11,600회의 읽기를 한 대의 서버로 감당할 수 없으므로, 로드밸런서가 여러 웹 서버에 요청을 고르게 분산한다. 트래픽을 나눠 처리하고, 특정 서버 장애 시에도 다른 서버로 요청을 넘겨 가용성을 유지한다.

## 정리

URL 단축기는 읽기 중심(읽기:쓰기 ≈ 10:1)의 시스템이다. 단축 URL 생성에는 해시 함수 방식과 base-62 변환 방식이 있는데, base-62는 충돌이 없는 대신 전역 유일 ID 생성기가 필요하다. 리디렉션 성능은 캐시와 로드밸런서로 끌어올린다. 앞서 배운 분산 유일 ID 생성기가 이 설계의 핵심 부품으로 재사용된다.
