#### 개략적 추정

> 쓰기 연산 → 매일 1억개의 단축 URL 생성
초당 쓰기 연산 → 1억(100million)/24/3600 = 1160
읽기 연산 → 읽기:쓰기 = 10:1이라고 가정. 1160*10 ⇒ 11,600회 발생
URL 단축 서비스 10년간 운영 → 1억 *365*10 = 3650억개의 레코드 보관
축약 전 URL 평균 길이 100이라고 가정
10년 동안 필요한 저장 용량→ 3650억*100바이트 = 46.6TB

#### API 엔드포인트

- REST 스타일로 설계
1. URL 단축용 엔드포인트
    
    `POST /api/v1/data/shorten`
    
    - 인자: {longUrl: ongURLstring}
    - 반환: 단축 URL
2. URL 리다이렉션용 엔드포인트
    - 단축 URL에 대해서 HTTP 요청이 오면 원래 URL로 보내주기 위한 용도의 엔드포인트
    
    `GET /api/v1/shortUrl` 
    

#### URL 리다이렉션

1. 단축 URL(도메인 tinyurl) 방문
2. tinyurl 서버 : 상태코드 301, Location: 원래 URL
3. 원래 URL 방문

<aside>

302 Permanently Moved

- 해당 URL에 대한 HTTP 요청의 처리 정책이 영구적으로 Location 헤더에 반환된 URL에 이전되었다는 응답.
    - 영구 이전 → 브라우저는 이 응답을 캐시함.
    - 추후 같은 단축 URL에 요청을 보낼때 캐시된 URL로 보냄

302 Found

- 주어진 URL로의 요청이 ‘일시적으로’ Location 헤더가 지정하는 URL에 의해 처리되어야 한다는 응답.
- 클라이언트의 요청은 언제나 단축 URL 서버에 먼저 보낸후 원래 URL로 리다이렉션되어야함
    - 즉, 캐시 불가

#### URL 단축

> 긴 URL → 해시함수 → 단축 URL 생성
> 
- 입력으로 주어지는 긴 URL이 다른 값이면 해시 값도 달라야함
- 계산된 해시 값은 원래 입력으로 주어졌던 긴 URL로 복원될 수 있어야함.

### 상세 설계

### 데이터 모델

- 관계형 데이터베이스에 <단축URL, 원래URL>의 순서쌍을 저장.
    - id(pk), shortURL, longURL 세 개의 칼럼

### 해시 함수

- 원래 주소를 단축 주소로 변환하는데 쓰임
- 단축 주소 = hashValue

<aside>

**해시 값 길이**

- [0-9, a-z, A-Z]의 문자들로 구성.
    - 사용가능한 문자의 개수 = 10+26+26 = 62개
- 62^n ≥ 2650억인 n의 최소값을 찾아야함!
- n=7이면 3.5조개의 URL을 만들수 있다. → 적절하면 선택 ㄱㄱ

</aside>

#### 해시 후 충돌 해소 전략

!IMG_8925.JPG

- CRC32, MD5, SHA-1 등의 해시 함수 사용
    - 이렇게 나온 값의 길이가 7보다 길다. 어떻게 줄일까?
        - 해시 값에서 처음 7개 글자만 이용한다. ⇒ 충돌 확률 업
        - 충돌이 발생하면 해소될 때까지 사전에 정한 문자열을 해시값이 덧붙인다.
    - 충돌은 해소가 되지만 URL을 생성할 때마다 한번이상 DB에 질의해야함 → 오버헤드 큼
        - 해결 → 블룸 필터를 사용
            - 블룸 필터: 어떤 집합에 특정 원소가 있는지 검사하는 확률론에 기초한 공간 효율이 좋은 기술

### URL 단축기 상세 설계

- 62진법 변환 기법 사용
    
    !IMG_8926.JPG
    
1. 입력으로 긴 URL을 받는다.
2. 데이터베이스에 해당 URL이 있는지 검사
    1. 있으면 단축 URL을 가져와서 반환
