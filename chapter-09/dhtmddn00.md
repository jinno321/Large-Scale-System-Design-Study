# 웹 크롤러(Web Crawler) 설계

## 웹 크롤러란

검색 엔진에서 널리 쓰는 기술로, 웹에 새로 올라오거나 갱신된 콘텐츠를 찾아내는 것이 주된 목적이다. 콘텐츠는 웹 페이지, 이미지, 비디오, PDF 등 다양할 수 있다. 로봇(robot) 또는 스파이더(spider)라고도 불린다.

## 크롤러의 이용

- **검색 엔진 인덱싱**: 가장 보편적인 용례. 웹 페이지를 모아 검색 엔진을 위한 로컬 인덱스를 만든다.
- **웹 아카이빙**: 나중에 사용할 목적으로 장기 보관하기 위해 웹에서 정보를 모으는 절차.
- **웹 마이닝**: 인터넷에서 유용한 지식을 도출한다.
- **웹 모니터링**: 인터넷에서 저작권이나 상표권이 침해되는 사례를 모니터링한다.

### 웹 마이닝이란

웹에서 수집한 대량의 데이터에서 의미 있는 패턴이나 지식을 추출하는 것이다. 보통 세 가지로 나눈다.

- **웹 콘텐츠 마이닝**: 웹 페이지의 실제 내용(텍스트, 이미지)에서 정보를 추출. 예: 뉴스 기사를 모아 감성 분석, 상품 리뷰에서 여론 파악.
- **웹 구조 마이닝**: 페이지 간 하이퍼링크 구조를 분석. 구글의 PageRank가 대표적이다.
- **웹 사용 마이닝**: 사용자의 웹 이용 로그(클릭, 방문 경로)를 분석해 행동 패턴을 파악.

## 웹 크롤러의 기본 알고리즘

1. URL 집합이 입력으로 주어지면, 해당 URL에 해당하는 웹 페이지를 다운로드한다.
2. 다운로드한 웹 페이지에서 URL들을 추출한다.
3. 추출된 URL을 다운로드할 URL 목록에 추가하고, 위 과정을 반복한다.

규모가 커지면 수많은 문제가 생겨 이 단순한 알고리즘만으로는 부족하다.

## 요구사항 정의

- 크롤러의 용도: 검색 엔진 인덱싱
- 매달 수집 페이지 수: 10억 개
- 새로 만들어지거나 수정된 웹 페이지도 고려
- 수집한 웹 페이지를 5년간 저장
- 중복된 콘텐츠는 무시

### 좋은 웹 크롤러가 만족시켜야 할 속성

- **규모 확장성**: 병행성(concurrency)을 활용하면 효과적으로 크롤링할 수 있다.
- **안정성(robustness)**: 비정상적 입력과 환경에 잘 대응할 수 있어야 한다.
- **예의(politeness)**: 수집 대상 웹사이트에 짧은 시간 동안 너무 많은 요청을 보내서는 안 된다.
- **확장성(extensibility)**: 새로운 형태의 콘텐츠를 지원하기 쉬워야 한다.

## 개략적 규모 추정

```
매달 10억 개 웹 페이지 다운로드

QPS = 10억 / 30일 / 24시간 / 3600초 ≈ 400페이지/초
최대(peak) QPS = 2 × QPS = 800페이지/초

웹 페이지 크기 평균: 500KB
10억 페이지 × 500KB = 500TB/월

5년치: 500TB × 12개월 × 5년 = 30PB
```

> 최대 QPS를 2배로 잡는 이유: 트래픽은 하루 24시간 균등하지 않다. 시스템은 평균이 아니라 피크를 견딜 수 있게 설계해야 하므로 경험적 안전 계수로 2배를 적용한다.

30PB 규모는 단일 DB로 불가능하고, HDFS나 AWS S3 같은 분산 파일 시스템/객체 저장소가 답이다. 크롤링한 HTML은 대부분 "한 번 쓰고 가끔 읽는" 데이터라서 저렴한 콜드 스토리지 티어를 활용하는 게 비용 면에서 유리하다.

## 크롤러의 컴포넌트

### 시작 URL 집합(Seed URLs)

웹 크롤러가 크롤링을 시작하는 출발점이다. 가능한 한 많은 링크를 탐색할 수 있도록 URL을 고른다. 일반적으로 URL 공간을 작은 부분집합으로 나누는 전략을 쓴다(지역별, 주제별 분류 등).

### 미수집 URL 저장소(URL Frontier)

다운로드할 URL을 저장·관리하는 컴포넌트. FIFO 큐라고 생각하면 된다.

### HTML 다운로더

인터넷에서 웹 페이지를 다운로드하는 컴포넌트.

### 도메인 이름 변환기(DNS Resolver)

URL에 대응하는 IP 주소를 알아낸다.

### 콘텐츠 파서

다운로드된 웹 페이지를 파싱하고 검증한다. 크롤링 과정 자체가 느려질 수 있으므로 독립된 컴포넌트로 만들었다.

### 중복 콘텐츠 확인

공개된 연구 결과에 따르면 약 29%의 웹 페이지 콘텐츠가 중복이다. 해시 함수(MD5, SHA-1 등)로 각 문서의 해시값을 구해 해시 셋(hash set)에 보관하고, 새 문서의 해시값이 집합에 있는지 O(1)에 확인한다.

> 이 방식은 바이트 단위로 완전히 동일한 문서만 잡아낸다. "거의 비슷한" 문서까지 잡으려면 SimHash 같은 근사 중복 탐지 기법이 필요하다.

### 콘텐츠 저장소

인기 있는 콘텐츠는 메모리에 두어 접근 지연시간을 줄이고, 나머지 대부분은 디스크에 저장한다.

### URL 추출기

HTML 페이지를 파싱하여 링크를 골라내고, 상대 경로를 절대 경로로 변환한다.

```
상대 경로: /wiki/Main_Page
→ 절대 경로: https://en.wikipedia.org/wiki/Main_Page
```

### URL 필터

특정 콘텐츠 타입/파일 확장자, 오류 URL, 접근 제외 목록에 포함된 URL을 걸러낸다.

### URL 저장소

이미 방문한 URL을 보관한다. 같은 URL을 여러 번 처리하는 것을 막아 서버 부하를 줄이고 무한 루프를 방지한다.

## 작업 흐름

1. 시작 URL들을 미수집 URL 저장소에 저장한다.
2. HTML 다운로더는 미수집 URL 저장소에서 URL 목록을 가져온다.
3. HTML 다운로더는 DNS Resolver로 IP 주소를 알아내고 웹 페이지를 다운받는다.
4. 콘텐츠 파서가 HTML 페이지를 파싱하여 올바른 형식인지 검증한다.
5. 중복 콘텐츠인지 확인한다. 이미 저장된 콘텐츠면 버리고, 없으면 저장소에 저장 후 URL 추출기로 전달한다.
6. URL 추출기가 HTML 페이지에서 링크를 골라낸다.
7. 골라낸 링크를 URL 필터로 전달한다.
8. 필터링 후 남은 URL 중 이미 처리한 URL은 버리고, 새 URL은 URL 저장소와 미수집 URL 저장소에 저장한다.

## 상세 설계

### DFS vs BFS

웹은 유향 그래프(directed graph)와 같다. 페이지는 노드, 하이퍼링크는 에지다.

보통은 **BFS**를 쓴다. DFS로 하면 어느 정도로 깊숙이 들어가게 될지 알 수 없기 때문이다.

표준적 BFS에는 두 가지 문제가 있다.
1. **예의 문제**: 한 페이지에서 나오는 링크의 상당수가 같은 서버로 돌아가, 그 웹사이트에 부하를 너무 많이 줄 수 있다.
2. **우선순위 부재**: 모든 페이지를 동등하게 처리하는데, 실제로는 중요도가 다르다.

미수집 URL 저장소를 잘 활용하면 이 문제를 해결할 수 있다.

### 예의(Politeness)

동일 웹사이트에 한해서는 한 번에 한 페이지만 요청한다. 같은 웹사이트의 페이지는 시간 차를 두고 다운로드한다.

각 다운로드 스레드는 별도의 FIFO 큐를 가지고, 해당 큐에서 꺼낸 URL만 다운로드한다.

```
큐 라우터 → 매핑 테이블 → FIFO 큐(호스트별) → 큐 선택기 → 작업 스레드
```

- **큐 라우터**: 같은 호스트에 속한 URL은 언제나 같은 큐로 가도록 보장
- **매핑 테이블**: 호스트명 ↔ 큐의 관계를 저장
- **FIFO 큐**: 호스트 하나당 큐 하나
- **작업 스레드**: 한 큐에서만 URL을 꺼내므로 같은 호스트에 동시 요청이 가지 않는다

### 우선순위(Priority)

PageRank, 트래픽 양, 갱신 빈도 등 다양한 척도로 URL의 우선순위를 정한다. **순위 결정 장치(prioritizer)**가 이 역할을 담당한다.

### 전체 미수집 URL 저장소 설계

예의와 우선순위를 모두 반영하면 두 계층으로 구성된다.

```
순위 결정 장치
   ↓
전면 큐(front queue) — 우선순위 관리
   ↓
전면 큐 선택기
   ↓
후면 큐 라우터(back queue router) — 호스트별 분배
   ↓
후면 큐(back queue) — 예의 관리(호스트당 하나)
   ↓
후면 큐 선택기
   ↓
작업 스레드
```

### 신선도(Freshness)

웹 페이지는 수시로 추가·삭제·변경된다. 이미 다운로드한 페이지라 해도 주기적으로 재수집해야 한다.

최적화 전략:
- 변경 이력 활용: 자주 바뀌는 페이지는 자주, 안 바뀌는 페이지는 드물게 재수집
- 우선순위 활용: 중요한 페이지는 좀 더 자주 재수집

### 미수집 URL 저장소의 지속성

대부분의 URL은 디스크에 두지만, IO 비용을 줄이기 위해 메모리 버퍼에 큐를 두고 주기적으로 디스크에 기록하는 하이브리드 방식을 택한다.

## HTML 다운로더

### 로봇 제외 프로토콜(Robots Exclusion Protocol)

`robots.txt`는 웹사이트가 크롤러와 소통하는 표준적 방법이다. 크롤러는 웹사이트를 긁어가기 전에 이 파일의 규칙을 먼저 확인해야 한다. 거푸 다운로드하는 것을 피하기 위해 주기적으로 다시 다운받아 캐시에 보관한다.

```
User-agent: *
Disallow: /private/
Allow: /public/
```

### 성능 최적화

1. **분산 크롤링**: 크롤링 작업을 여러 서버에 분산. URL 공간을 작은 단위로 나눠 각 서버가 일부만 담당한다.

2. **DNS 조회 결과 캐시**: DNS 요청은 보통 10ms~200ms가 걸리고 블로킹될 수 있다. 도메인 이름과 IP 주소 사이의 관계를 캐시에 보관해두고, 크론 잡(cron job)으로 주기적으로 갱신한다.

3. **지역성(Locality)**: 크롤링 서버를 크롤링 대상 서버와 지역적으로 가깝게 배치하면 페이지 다운로드 시간이 줄어든다.

4. **짧은 타임아웃**: 응답이 느리거나 없는 서버에 최대 대기 시간을 정해두고, 초과 시 다음 페이지로 넘어간다.

### 안정성

- **안정 해시**: 다운로더 서버들에 부하를 분산할 때 적용. 서버를 쉽게 추가/삭제할 수 있다.
- **크롤링 상태 저장**: 장애가 생겨도 그 시점부터 다시 시작할 수 있도록 지속적 저장장치에 기록.
- **예외 처리**: 에러가 발생해도 전체 시스템이 중단되지 않고 그 작업만 처리하고 계속 진행.
- **데이터 검증**: 시스템 오류를 방지하기 위한 중요 수단.

### 확장성

새로운 형태의 콘텐츠를 쉽게 지원할 수 있도록 새 모듈을 끼워 넣는 방식으로 확장한다.

## 문제 있는 콘텐츠 감지 및 회피

### 1. 중복 콘텐츠

약 30%의 웹 페이지는 중복이다. 해시나 체크섬을 사용하면 중복을 쉽게 탐지할 수 있다.

> 해시(암호학적 해시)는 충돌 확률이 극도로 낮아 중복 탐지에 적합하다. 체크섬은 계산이 빠르지만 충돌이 상대적으로 잘 일어난다. 1차로 체크섬으로 후보를 걸러내고 2차로 해시를 비교하는 조합도 가능하다.

### 2. 거미 덫(Spider Trap)

크롤러를 무한 루프에 빠뜨리도록 설계한 웹 페이지다.

```
예: spidertrap.com/foo/bar/foo/bar/foo/bar/...
```

만능 해결책은 없다. URL의 최대 길이를 제한하거나, 수작업으로 탐색 대상에서 제외하거나 URL 필터 목록에 걸어둔다.

### 3. 데이터 노이즈

광고, 스팸 URL, 스크립트 코드 조각 등 가치가 없는 콘텐츠는 가능하다면 크롤링 대상에서 제외한다.

## 마무리: 추가로 논의할 점

### 서버 측 렌더링(Server-Side Rendering)

많은 웹사이트가 자바스크립트, AJAX 등으로 링크를 동적으로 생성한다. 크롤러가 HTML 원본만 파싱하면 자바스크립트가 실행된 후에야 나타나는 링크를 발견할 수 없다.

해결책: **헤드리스 브라우저(headless browser)**를 사용한다. 화면 없이 동작하는 실제 브라우저 엔진(예: Puppeteer로 제어하는 Chrome, Playwright)으로 자바스크립트를 실제로 실행한 뒤 완성된 DOM을 파싱한다.

```
기존:  URL → HTTP GET → 원본 HTML → 파싱 (동적 링크 놓침)
개선:  URL → 헤드리스 브라우저 → JS 실행 → 완성된 DOM → 파싱
```

브라우저를 띄우는 비용이 크므로 모든 페이지에 적용하기보다 필요한 사이트에만 선별적으로 쓴다.

### 그 외 고려사항

- **원치 않는 페이지 필터링**: 스팸 방지 컴포넌트를 두어 품질 낮은 페이지를 걸러낸다.
- **데이터베이스 다중화 및 샤딩**: 데이터 계층의 가용성, 규모 확장성, 안정성 향상.
- **수평적 규모 확장성**: 서버가 상태 정보를 유지하지 않도록(무상태) 만드는 것이 핵심.
- **데이터 분석 솔루션**: 시스템을 세밀히 조정(fine-tuning)하려면 데이터 수집과 분석이 필수.

## 정리

웹 크롤러는 단순해 보이는 알고리즘(다운로드 → 링크 추출 → 반복) 위에 수많은 현실적 문제가 얹힌 시스템이다. 예의(호스트별 큐), 우선순위(순위 결정 장치), 신선도(재수집 전략)를 미수집 URL 저장소로 해결하고, 성능은 분산 크롤링·DNS 캐시·지역성·짧은 타임아웃으로 끌어올린다. 중복 콘텐츠는 해시 비교로, 거미 덫은 필터로 방어한다. 앞서 배운 안정 해시가 다운로더 부하 분산에 재사용된다는 점도 눈여겨볼 만하다.
