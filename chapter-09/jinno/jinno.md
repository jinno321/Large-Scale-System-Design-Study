# [8장] URL 단축기 설계

## API 엔드포인트
URL 단축기는 기본적으로 2개의 엔드포인트를 필요로 한다.

1. URL 단축용 엔드포인트 : 클라이언트는 단축할 URL을 인자로 실어서 POST요청을 보낸다.
2. URL 리디렉션용 엔드포인트 : 단축 URL에 대해서 HTTP 요청이 오면 원래 URL로 보내주기 위한 엔드포인트

## URL 리디렉션
만약 브라우저에 단축되어 있는 URL을 호출하면, 단축 URL을 받은 서버는 그 URL을 원래 URL로 바꿔서 301응답의 Location 헤더에 넣어 반환한다.

### 301 vs 302

- 301 : Permanently Moved, 해당 URL에 대한 HTTP 요청이 처리 책임이 영구적으로 Location 헤더에 반환된 URL로 이전되었다는 것을 의미한다. 브라우저는 이 응답을 캐시하므로, 추후 같은 요청은 서버를 거치지 않고 바로 리디렉션된다. (서버 부하 감소)

- 302 : Found, URL의 변경이 일시적임을 의미한다. 따라서 향후 올 요청도 원래 URL 단축 서버로 먼저 보내진 후, Location 헤더에 담긴 URL로 리디렉션되어야 한다. (트래픽 분석 및 추적에 유리)

---

## 개략적 설계 안 (전체 흐름)

### 1. URL 단축 흐름
* 클라이언트가 원래 URL(longURL)을 담아 POST 요청을 보낸다.
* 서버는 이 URL을 받아 단축 URL(shortURL)로 변환 후, DB에 저장하고 클라이언트에게 응답한다.

### 2. URL 리디렉션 흐름
* 클라이언트가 단축 URL로 GET 요청을 보낸다.
* 로드밸런서가 요청을 웹 서버로 전달한다.
* 웹 서버는 해당 단축 URL에 매핑된 longURL을 찾아 301 또는 302 응답으로 리디렉션한다.

## 상세 설계

### 1. 데이터 모델
* 메모리는 한정되어 있으므로 `<단축 URL, 원래 URL>` 쌍을 관계형 데이터베이스(RDB)에 저장한다.
* 테이블은 단순하게 `id(PK)`, `shortURL`, `longURL` 3개의 컬럼을 가진다.

### 2. URL 단축 알고리즘 (길이 및 변환 방식)

#### URL 길이 결정 (추정치 계산)
* 매일 1억 개의 단축 URL을 생성해야 한다.
* 10년간 보관한다고 가정하면 총 $1억 \times 365 \times 10 = 3,650억$ 개의 URL을 감당해야 한다.
* 단축 URL문자로 사용할 수 있는 숫자가 영문자 개수는 총 62개(`0-9`, `a-z`, `A-Z`)이다. (Base62)
* 길이에 따른 표현 가능 수:
  * $62^{6} \approx 568억$ (부족함)
  * $62^{7} \approx 3조 5,200억$ (3,650억 개를 수용하기에 충분함)
* 따라서 단축 URL의 길이는 **최소 7글자**로 설정한다.

#### 방법 A: 해시 후 충돌 해소 방식
* CRC32, MD5, SHA-1 등의 해시 함수를 이용해 원래 URL을 변환하고, 앞 7글자만 잘라서 사용한다.
* **충돌 해결:** 서로 다른 원래 URL이 같은 해시 값을 가질 수 있으므로, 충돌이 발생하면 원래 URL 뒤에 사전에 정의한 고유 문자열을 덧붙여 다시 해시한다. 
* 디비 조회가 반복되므로 오버헤드가 크다. (블룸 필터를 사용해 최적화 가능)

#### 방법 B: 분산 카운터 방식 (유일 ID 생성기 + Base62 변환)
* 시스템 전역에서 유일성이 보장되는 **ID 생성기**를 이용해 숫자로 된 고유 ID를 먼저 발급받는다. (예: 트위터 스노우플레이크 등)
* 발급받은 10진수 ID를 **Base62로 변환**하여 단축 URL로 사용한다.
* ID 자체가 유일하므로 해시 충돌 가능성이 원천 차단된다.

### 3. URL 단축기 상세 흐름
1. 입력으로 원래 URL(longURL)을 받는다.
2. 입력된 URL이 DB에 있는지 확인한다.
3. 있다면 해당 단축 URL을 가져와서 반환한다.
4. 없다면, 유일 ID 생성기를 통해 새로운 ID를 발급받는다.
5. 발급받은 ID를 Base62로 인코딩하여 단축 URL을 만든다.
6. ID, 원래 URL, 단축 URL을 DB에 저장한다.

### 4. URL 리디렉션 상세 흐름
1. 클라이언트가 단축 URL로 요청을 보낸다.
2. 웹 서버는 먼저 **캐시**에서 해당 단축 URL이 있는지 확인한다.
3. 캐시에 있다면 즉시 원래 URL을 꺼내 리디렉션한다.
4. 캐시에 없다면 **데이터베이스**에서 조회한다. 
5. DB에도 없다면 잘못된 URL 에러를 반환하고, 있다면 캐시에 저장한 뒤 원래 URL로 리디렉션한다.

# [9장] 웹 크롤러 설계

## 크롤러의 주요 용도
* **검색 엔진 인덱싱:** 크롤러의 가장 보편적 용례. 웹 페이지를 모아 검색 엔진을 위한 로컬 인덱스를 만든다. ex) 구글의 Googlebot
* **웹 아카이빙:** 장기보관하기 위해 웹에서 정보를 모으는 절차. 많은 국립 도서관이 크롤러를 돌려 웹사이트 아카이빙을 진행 중이다.
* **웹 마이닝:** 크롤링을 통해 인터넷에서 유용한 지식을 도출해 낼 수 있다. 유명 금융 기업들은 크롤러를 사용해 주주 총회 자료 등 기업의 핵심 사업 방향을 알아냄.
* **웹 모니터링:** 예를 들면, 저작권이나 상표권이 침해되는 사례를 모니터링 할 수 있다.

### 기본 알고리즘과 웹 크롤러의 속성
1. URL 집합이 입력으로 주어지면, 해당 URL들이 가리키는 모든 웹 페이지를 다운로드한다.
2. 다운받은 웹 페이지에서 URL들을 추출한다.
3. 추출된 URL들을 다운로드할 URL 목록에 추가하고 위의 과정을 처음부터 반복한다.

사실, 이건 가장 간단한 알고리즘이고, 엄청난 규모 확장성을 갖는 웹 크롤러를 설계하는 것은 엄청 어려운 작업이다. 만약 웹 크롤러를 설계할 때는 다음과 같은 속성들에 관심을 가져야 한다.

* **규모 확장성:** 웹은 거대하다. 병행성(Parallelism)을 활용하면 보다 효과적으로 웹 크롤링을 할 수 있다.
* **안정성(Robustness):** 잘못 작성된 HTML, 아무 반응이 없는 서버, 악성 링크 등 함정이 가득하다. 크롤러는 이런 비정상적 입력이나 환경에 잘 대응할 수 있어야 한다.
* **예절(Politeness):** 크롤러는 수집 대상 웹 사이트에 짧은 시간 동안 너무 많은 요청을 보내서는 안 된다. (디도스 공격으로 오인될 수 있음)
* **확장성(Extensibility):** 새로운 형태의 콘텐츠(예: 이미지, PDF 등)를 지원하기 쉬워야 한다.

## 컴포넌트

크롤러를 구성하는 컴포넌트 각각이 어떤 기능을 수행하는지 살펴본다.

### 시작 URL 집합
웹 크롤러가 크롤링을 시작하는 출발점이다. 예를 들어, 어떤 대학 웹사이트로부터 찾아 나갈 수 있는 모든 웹 페이지를 크롤링하려면 해당 대학의 도메인 이름이 붙은 모든 페이지의 URL을 시작 URL로 쓰면 된다. 

이거는 가장 간단한 방법이고, 크롤러가 가능한 한 많은 링크를 탐색할 수 있도록 하는 URL을 고르는 것이 바람직해 보인다. 일반적으로 이것들을 작은 부분집합으로 나누는 전략을 쓴다. 주제별로 세분화할 수도 있고, 사실 정답이 있지는 않다.

### 미수집 URL 저장소 (URL Frontier)
대부분의 웹 크롤러는 크롤링 상태를 다운로드할 URL, 다운로드된 URL의 두 가지로 나눠 관리한다. 이 중 **다운로드할 URL**을 저장 관리하는 컴포넌트를 의미한다. 

### HTML 다운로더
인터넷에서 웹 페이지를 다운로드하는 컴포넌트이다. 미수집 URL 저장소에서 URL을 가져온다.

### 도메인 이름 변환기 (DNS Resolver)
웹 페이지를 다운받으려면 URL을 IP 주소로 변환해야 한다. 이때 URL로 IP 주소를 알아내는 컴포넌트이다.

### 콘텐츠 파서 (Content Parser)
웹 페이지를 다운로드하면 파싱과 검증 절차를 거쳐야 한다. 이상한 웹 페이지는 문제를 일으킬 수도 있고, 저장 공간을 낭비한다. 크롤링 서버 안에 콘텐츠 파서를 구현하면 크롤링 과정이 느려지므로 독립적으로 구성한다.

### 중복 콘텐츠 구별
29% 가량의 웹 페이지 콘텐츠는 중복이라고 한다. 같은 콘텐츠를 여러 번 저장하는 것을 방지하기 위해 웹 페이지의 해시값(예: MD5, 체크섬 등)을 비교해서 중복 콘텐츠인지를 구별해낸다.

### 콘텐츠 저장소
HTML 문서를 보관하는 시스템이다. 저장소를 구현하는 데 쓰일 기술을 고를 때는 저장할 데이터의 유형, 크기, 저장소 접근 빈도, 유효 기간 등을 종합적으로 고려해야 한다. 디스크, 메모리 아니면 둘 다 사용하는 저장소 등을 사용할 수 있다.
여기서는 다음 전략을 채택하겠다.
* 대부분의 콘텐츠는 디스크에 저장
* 인기 있는 콘텐츠는 메모리에 두어 접근 지연시간을 줄임

### URL 추출기
HTML 페이지를 파싱하여 링크들을 골라내는 컴포넌트이다. (상대 경로는 절대 경로로 변환하여 추출한다.)

### URL 필터
특정한 콘텐츠 타입이나 파일 확장자를 갖는 URL, 접속 시 오류가 발생하는 URL, 접근 제외 목록(Blacklist)에 포함된 URL 등을 배제하는 역할을 한다.

### 이미 방문한 URL 필터
이미 방문한 URL이나 미수집 URL 큐에 보관된 URL을 추적할 수 있도록 도와준다. 이렇게 되면, 서버 부하를 줄이고 시스템이 무한 루프에 빠지는 일을 방지할 수 있다. 해당 기능을 위해 **블룸 필터(Bloom Filter)**나 **해시 테이블**을 사용하겠다.

### URL 저장소
이미 방문한 URL을 보관하는 저장소다.

## 웹 크롤러 작업 흐름
[시작 URL 집합] ➔ [미수집 URL 저장소] ➔ [HTML 다운로더] ➔ [콘텐츠 파서]
│               │
(DNS 변환기)    (중복 콘텐츠 확인)
│
[미수집 URL 저장소] 🔀 [이미 방문한 URL 필터] ⬅ [URL 필터] ⬅ [URL 추출기]

1. 시작 URL들을 미수집 URL 저장소에 저장한다.
2. HTML 다운로더는 미수집 URL 저장소에서 URL 목록을 가져온다.
3. HTML 다운로더는 도메인 이름 변환기를 사용하여 URL의 IP 주소를 알아내고, 해당 IP 주소로 접속하여 웹 페이지를 다운받는다.
4. 콘텐츠 파서는 다운된 HTML 페이지를 파싱하여 올바른 형식을 갖춘 페이지인지 검증한다.
5. 파싱, 검증이 끝나면 중복 콘텐츠인지 확인하는 절차를 개시한다.
6. 중복 콘텐츠인지 확인하기 위해서, 해당 페이지가 이미 저장소에 있는지 본다.
   * 이미 저장소에 있는 콘텐츠라면 처리하지 않고 버린다.
   * 그렇지 않다면 저장소에 저장하고 URL 추출기로 전달한다.
7. URL 추출기는 해당 HTML 페이지에서 링크를 골라내 URL 필터로 전달한다.
8. 필터링이 끝나면 중복 URL을 판별한다.
9. 이미 처리한 URL(이미 방문한 URL 필터에 존재)인지 확인한다.
10. 저장소에 없다면, URL 저장소에 저장하고 미수집 URL 저장소에 전달한다.

## 상세 설계

### DFS vs BFS
웹은 거대한 그래프라고 할 수 있다. 페이지는 노드이고, 하이퍼링크를 에지라고 보면 된다. 크롤링 프로세스가 이 거대한 그래프를 탐색하는 과정이다. 결론부터 말하면 **DFS(깊이 우선 탐색)는 좋은 선택지가 아니다.** 그래프 크기가 클 경우 어느 정도의 depth까지 가게 될지 가늠하기 어렵기 때문이다.

그래서 보통 **BFS(너비 우선 탐색)**를 사용한다. 하지만 기본적인 FIFO(First-In-First-Out) 큐를 사용하는 알고리즘은 두 가지 문제점이 있다.
* **예절 문제:** 한 페이지에서 나오는 링크의 상당수는 같은 서버로 되돌아간다. 이때, 이 링크들을 병렬로 처리하면 그 서버는 수많은 요청으로 과부하에 걸린다. (Impolite 크롤러)
* **우선순위 문제:** 기본적인 FIFO에는 우선순위가 없다. 하지만 모든 웹 페이지가 같은 수준의 품질, 중요성을 갖는 것은 아니다. 페이지랭크나 트래픽 양에 따라 처리 우선순위를 구별하는 것이 좋아 보인다.

### 미수집 URL 저장소 활용
이런 문제는 미수집 URL 저장소를 활용하면 쉽게 해결이 가능하다. 다운로드할 URL을 보관하는 장소인데, URL 사이의 우선순위와 신선도를 구별하며, 예의를 갖춘 크롤러를 구현 가능하다.

#### 1. 예의 (Politeness)
수집 대상 서버로 짧은 시간 안에 너무 많은 요청을 보내는 것을 삼가야 한다. 때로 디도스(DDoS) 공격으로 간주되기도 하니까.. 그래서 **동일 웹 사이트(호스트)에 대해서는 한 번에 한 페이지만 요청**해야 한다.

그러기 위해서는 웹사이트의 호스트명과 다운로드를 수행하는 작업 스레드(Worker thread) 사이의 관계를 유지하면 된다. 즉, 각 다운로드 스레드는 별도 큐를 갖고, 해당 큐에서 꺼낸 URL만 다운로드한다.
* **큐 라우터 (Queue Router):** 같은 호스트에 속한 URL이 언제나 같은 큐로 가도록 보장해 준다.
* **매핑 테이블:** 호스트 이름과 큐 사이의 관계를 보관한다.
* **FIFO 큐 ($B_1$ to $B_n$):** 같은 호스트에 속한 URL은 언제나 같은 큐에 보관한다.
* **큐 선택기 (Queue Selector):** 큐들을 순회하며 큐에서 URL을 꺼내서 해당 URL을 다운로드하도록 지정된 작업 스레드에 전달한다.

#### 2. 우선순위 (Priority)
애플 제품에 대한 사용자 의견이 올라오는 페이지는 애플 홈페이지보다 더 높은 우선순위를 가질 것이다. 둘 다 애플이 키워드긴 하지만.. 그래서, 페이지랭크(PageRank), 트래픽 양, 갱신 빈도 등 다양한 척도를 사용한다.
* **순위결정장치 (Prioritizer):** URL의 우선순위를 정한다.
* **우선순위별 큐 ($f_1$ to $f_n$):** 순위결정장치가 우선순위를 계산하고, 우선순위별로 큐가 하나씩 할당된다. (우선순위가 높으면 선택될 확률이 높아진다.)

#### 3. 신선도 (Freshness)
웹 페이지의 데이터들은 수시로 추가되고, 삭제되고, 변경된다. 그래서 데이터의 신선함을 유지하기 위해서는 다운로드한 페이지라고 해도 주기적으로 재수집 해야 한다. 웹 페이지의 변경 이력 등을 활용하거나, 우선순위를 활용하여 중요한 페이지는 좀 더 자주 재수집하는 둥 해야 한다.

#### 4. 미수집 URL 저장소를 위한 지속성 저장장치
검색 엔진을 위한 크롤러의 경우, 처리해야 하는 URL의 수는 수억 개 이상이다. 이것들을 모두 메모리에 보관하는 것은 안정성이나 규모 확장성 측면에서 바람직하지 않다. 전부 디스크에 저장하는 것도 성능 병목지점이 되어서다. 

따라서 **대부분의 URL은 디스크에 두지만, I/O 비용을 줄이기 위해 메모리 버퍼에 큐를 둔다.** 버퍼에 있는 데이터는 주기적으로 디스크에 기록한다.

## HTML 다운로더
* HTTP 프로토콜을 통해 웹 페이지를 내려받는다.

### robots.txt (로봇 제외 표준)
검색엔진 크롤러 같은 웹 로봇이 웹사이트에 접근할 수 있는 범위와 경로를 지정하는 텍스트 파일이다. 여기에 사용자는 크롤러가 수집해도 되는 페이지 목록을 넣어 둔다. 웹 사이트를 긁기 전에 크롤러는 해당 파일에 나열된 규칙을 확인해야 한다. (자주 다운로드하는 것을 피하기 위해 `robots.txt` 파일은 주기적으로 크롤링하여 캐시에 보관한다.)

### 성능 최적화
1. **분산 크롤링:** 성능을 높이기 위해 크롤링 작업을 여러 서버에 분산한다. 여러 스레드를 돌려 다운로드 작업을 처리한다. 이 구성을 위해 URL 공간을 작은 단위로 분할하여, 각 서버는 그중 일부의 다운로드를 담당하도록 한다.
2. **도메인 이름 변환 결과 캐시 (DNS 캐시):** DNS 요청을 보내고 결과를 받는 작업은 동기적(Synchronous)이다. DNS 요청의 결과를 받기 전까지는 다음 작업을 진행할 수 없다. 이 때문에 성능 병목 지점이다. 크롤러 스레드 가운데 어느 하나라도 이 작업을 하고 있으면 다른 DNS 요청이 전부 블록된다. 그래서 도메인 이름과 IP 주소 관계를 캐싱해두고, 크론 잡(Cron job) 등을 돌려 주기적으로 갱신하면 성능이 높아진다.
3. **지역성 (Locality):** 크롤링 작업을 수행하는 서버를 지역별로 분산할 수도 한다. 크롤링 서버가 크롤링 대상 서버와 지역적으로 가까우면 페이지 다운로드 시간이 줄어드니까..
4. **짧은 타임아웃:** 웹 서버에 요청을 했는데, 응답이 느리거나 응답하지 않는 경우가 있다. 이런 경우에 대기 시간이 길어지므로, 최대 타임아웃 시간을 미리 정해두고 이 시간 내에 응답하지 않으면 다음 URL로 넘어간다.

## 안정성 (Robustness)

최적화된 성능뿐 아니라 시스템의 안정성도 매우 중요한 고려 사항이다.
* **안정 해시 (Consistent Hashing):** 다운로더 서버들에 부하를 분산할 때 적용 가능하다. 다운로더 서버를 쉽게 추가하고 삭제할 수 있어 규모 확장성에 유리하다.
* **크롤링 상태 및 수집 데이터 저장:** 장애가 발생한 경우에도 쉽게 복구할 수 있도록 크롤링 상태와 수집된 데이터를 지속적 저장장치(디스크)에 기록해 두는 것이 좋다. (중단되었던 지점부터 다시 시작 가능)
* **예외 처리:** 예외가 발생해도 전체 시스템이 중단되지 않도록 방어적으로 코드를 짜야 한다.

## 확장성 (Extensibility)

시스템 설계할 때 새로운 형태의 콘텐츠를 쉽게 지원할 수 있도록 해야 한다. 이때 유연하게 **확장 모듈(플러그인)**들을 끼워넣으면 좋다.
* 예: PNG/JPEG 파일 같은 이미지를 다운로드하고 파싱하는 플러그인 모듈 추가
* 예: 웹을 모니터링해서 저작권/상표권 침해 등을 막는 모니터링 모듈 추가

### 문제 있는 콘텐츠 감지 및 회피
1. **중복 콘텐츠:** 해시나 체크섬(Anti-duplication)을 사용하면 중복 콘텐츠를 쉽게 탐지하고 회피할 수 있다.
2. **거미 덫 (Spider Trap):** 거미 덫은 크롤러를 무한 루프에 빠뜨리도록 설계한 웹 페이지다. (예: `spidertrap.com/a/b/c/a/b/c/...` 처럼 무한히 깊은 디렉터리 구조를 만드는 링크)
   * **회피 방법:** URL의 최대 길이를 제한하면 덫을 회피할 수 있다. 하지만 만능의 해결책은 없으므로, 이런 덫을 자동으로 피해가는 알고리즘을 만들어내는 것은 힘들고 결국 사람의 수작업(Blacklist 등록 등)이 필요해 보인다.
3. **데이터 노이즈:** 광고, 스팸 URL, 의미 없는 스크립트 등 가치 없는 콘텐츠는 과감히 배제해야 한다.
