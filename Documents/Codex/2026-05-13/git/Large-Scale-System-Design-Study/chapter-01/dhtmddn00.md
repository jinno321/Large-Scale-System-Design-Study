# 대규모 시스템 설계 기초
> 《가상 면접 사례로 배우는 대규모 시스템 설계 기초》 1장 내용을 정리한 문서.
---
## 1. 단일 서버 구조
가장 단순한 형태. 웹 서버와 데이터베이스가 한 대의 서버에서 동작한다.
```
사용자 → DNS 조회 (도메인 → IP) → 웹 서버에 요청 → 응답 반환
```
사용자가 늘어나면 서버 한 대로는 감당하기 어려워진다. 이 때 일반적으로 **웹 서버**와 **데이터베이스 서버**를 분리한다. 분리하면 각각을 독립적으로 확장할 수 있다.
---
## 2. 데이터베이스 종류
### 관계형 데이터베이스 (RDBMS)
- MySQL, PostgreSQL, Oracle 등
- 데이터를 테이블 형태로 저장하고 **JOIN** 연산으로 여러 테이블을 결합할 수 있다.
- 스키마가 명확하고, ACID 트랜잭션을 지원한다.
### 비관계형 데이터베이스 (NoSQL)
- MongoDB, Cassandra, Redis, DynamoDB 등
- 일반적으로 JOIN을 지원하지 않는다.
- 다음과 같은 상황에서 선택을 고려한다.
  - 비정형 데이터를 다뤄야 할 때
  - 아주 낮은 지연 시간이 요구될 때
  - 아주 많은 양의 데이터를 저장해야 할 때
  - 스키마를 자주 바꿔야 할 때
---
## 3. 수직적 확장 vs 수평적 확장
### 수직적 확장 (Scale Up)
서버 한 대의 사양을 높이는 방식. CPU 업그레이드, RAM 증설 등.
- **장점**: 구현이 단순하다. 애플리케이션 코드 변경이 거의 없다.
- **단점**:
  - 하드웨어 스펙에 물리적 한계가 있다.
  - 단일 서버이므로 장애 발생 시 서비스 전체가 중단된다. (단일 장애 지점, SPOF)
### 수평적 확장 (Scale Out)
서버 대수를 늘리는 방식.
- 대규모 트래픽에 현실적으로 대응 가능하다.
- 한 서버가 죽어도 나머지가 요청을 처리할 수 있다.
- **로드 밸런서**가 필요하다.
---
## 4. 로드 밸런서
사용자 트래픽을 여러 서버에 고르게 분산시키는 장치.
```
사용자 → 로드 밸런서 (공인 IP) → 웹 서버 1 (사설 IP)
                               → 웹 서버 2 (사설 IP)
```
- 사용자는 로드 밸런서의 **공인 IP(Public IP)** 로 접근한다.
- 로드 밸런서와 서버 간 통신에는 **사설 IP(Private IP)** 를 사용한다. 외부에서 직접 서버에 접근할 수 없어 보안이 향상된다.
- 서버 한 대가 다운되면 나머지 서버로 트래픽을 자동으로 보낸다. (Failover)
- 서버를 추가하면 로드 밸런서가 자동으로 분산 대상에 포함시킨다.
---
## 5. 데이터베이스 다중화 (Replication)
데이터베이스 서버를 **주(Master) / 부(Slave, Replica)** 로 나누어 운영하는 방식.
| 구분 | 역할 |
|---|---|
| Master | 쓰기(INSERT, UPDATE, DELETE) 전담 |
| Slave | 읽기(SELECT) 전담, Master로부터 데이터 복제 |
대부분의 애플리케이션은 읽기 연산이 쓰기보다 압도적으로 많다. 따라서 Slave 서버를 여러 대 두어 읽기 부하를 분산하는 것이 일반적이다.
### 장점
- **성능 향상**: 읽기/쓰기 연산을 병렬로 처리한다.
- **안정성**: 일부 서버가 파괴되어도 다른 서버에 데이터가 남아 있다.
- **가용성**: 한 서버가 다운되어도 다른 서버가 요청을 처리한다.
### 주의 사항
- Slave 서버의 데이터는 Master와 **약간의 시간 차(복제 지연, Replication Lag)** 가 있을 수 있다. 최신 데이터가 필요한 읽기라면 Master에서 직접 조회해야 한다.
- Master가 다운되면 Slave 중 하나를 승격(Promote)해 새 Master로 만들어야 한다. 이 과정에서 아직 복제되지 않은 데이터가 유실될 수 있다. 복구 스크립트나 **멀티 마스터(Multi-Master)**, **원형 복제(Circular Replication)** 같은 방식을 고려할 수 있다.

## 6. 캐시 (Cache)
자주 조회되는 데이터나 연산 비용이 비싼 결과를 메모리에 저장해두고, 이후 요청을 빠르게 처리하는 저장소. Redis, Memcached 등이 대표적이다.
### 캐시 읽기 전략 (Cache-Aside / Look-Aside)
```
요청 → 캐시 확인
         ├─ Hit: 캐시에서 바로 반환
         └─ Miss: DB 조회 → 캐시에 저장 → 반환
```
### 캐시 사용 시 고려 사항
- **도입 기준**: 읽기가 쓰기보다 훨씬 빈번한 데이터에 적합하다. 자주 변경되는 데이터에는 효과가 떨어진다.
- **영속성**: 캐시는 휘발성 메모리(RAM)에 저장된다. 재시작하면 사라지므로, 반드시 영구 보존이 필요한 데이터는 캐시에만 두면 안 된다.
- **만료 정책 (TTL, Time-To-Live)**: 너무 짧으면 캐시 미스가 잦아지고, 너무 길면 오래된 데이터가 제공될 수 있다.
- **일관성**: DB와 캐시의 데이터가 불일치할 수 있다. 쓰기 시 캐시를 같이 갱신하거나 무효화(Invalidation)하는 전략이 필요하다.
- **단일 장애 지점**: 캐시 서버가 한 대뿐이면 장애 시 전체에 영향을 준다. 여러 대로 분산하거나 클러스터링을 고려한다.
- **메모리 크기**: 메모리가 꽉 찼을 때 어떤 데이터를 제거할지 **방출 정책(Eviction Policy)** 을 정해야 한다. 대표적인 정책은 LRU(Least Recently Used), LFU(Least Frequently Used), FIFO 등이 있다.
### 캐시 서버의 메모리와 디스크
캐시는 메모리를 주 저장소로 사용하지만, 디스크도 함께 활용한다.
Redis를 예로 들면 두 가지 방식으로 디스크에 데이터를 저장한다.
- **RDB (Redis Database)**: 주기적으로 메모리 전체를 디스크에 스냅샷으로 저장
- **AOF (Append Only File)**: 모든 쓰기 명령을 디스크에 순서대로 기록
Redis가 재시작되면 디스크에 저장된 내용을 읽어 메모리를 복구한다. 완전한 휘발성은 아닌 셈이다.
그럼에도 "캐시는 휘발성"이라고 표현하는 이유는, 디스크 저장은 어디까지나 **복구용 보조 수단**이고 메모리가 꽉 차면 디스크 저장 여부와 관계없이 **Eviction(방출)** 이 일어나기 때문이다. "영속적으로 보관할 데이터는 캐시에만 두면 안 된다"는 말은 바로 이 Eviction을 포함한 의미다.
| | 메모리 | 디스크 |
|---|---|---|
| 역할 | 실제 데이터 저장 및 조회 | 장애/재시작 시 복구용 |
| 속도 | 빠름 | 느림 |
| 용량 | 제한적 (비쌈) | 큼 (쌈) |

## 7. CDN (Content Delivery Network)
정적 콘텐츠(이미지, CSS, JS, 동영상 등)를 전 세계에 분산된 서버에 캐싱하여 사용자에게 가장 가까운 서버에서 제공하는 네트워크.
```
사용자 (한국) → 가까운 CDN 엣지 서버 (서울) → 정적 파일 반환
              (없으면 Origin 서버에서 가져와 CDN에 캐싱)
```
### 고려 사항
- **비용**: CDN은 트래픽 기반으로 요금이 발생하므로, 잘 쓰이지 않는 파일은 굳이 올릴 필요가 없다.
- **만료 시간 설정**: 콘텐츠 신선도(Freshness)를 유지하기 위해 적절한 TTL을 설정해야 한다.
- **장애 대처**: CDN 장애 시 클라이언트가 원본(Origin) 서버에서 직접 자원을 가져올 수 있도록 폴백(fallback)을 구현해야 한다.
- **캐시 무효화(Invalidation)**: 파일을 업데이트했을 때 CDN에 남아 있는 이전 버전을 제거하거나, URL에 버전 파라미터(`?v=2`)를 붙이는 방식으로 우회할 수 있다.
### CDN vs 데이터 센터
| | CDN | 데이터 센터 |
|---|---|---|
| 대상 | 정적 콘텐츠 (이미지, JS, CSS) | 서비스 전체 인프라 |
| 방식 | 콘텐츠를 엣지 서버에 캐싱 | 서버/DB 자체를 여러 지역에 배치 |
| 주목적 | 지리적 레이턴시 감소 | 레이턴시 감소 **+** 장애 대응 |
| 거점 수 | 수백~수천 곳 | 수십 곳 |
| 구축 비용 | 낮음 | 매우 높음 |
### 엣지 컴퓨팅 (Edge Computing)
CDN 엣지 서버에서 직접 코드를 실행하는 방식. Cloudflare Workers, AWS Lambda@Edge 등이 대표적이다.
엣지에서 처리 가능한 작업: 인증 토큰 검증, 사용자 위치 기반 응답 분기, A/B 테스트, 요청/응답 헤더 조작 등.
무거운 연산이나 DB 접근은 여전히 데이터 센터에서 처리해야 한다.

## 8. 무상태 웹 계층 (Stateless Web Tier)
### 상태 유지(Stateful) 아키텍처의 문제
사용자 세션을 특정 서버에 저장하면 **고정 세션(Sticky Session)** 이 필요해진다.
- 부하 분산 효율이 떨어지고, 해당 서버 장애 시 세션이 사라진다.
- 서버 확장/축소가 어렵다.
### 무상태(Stateless) 아키텍처
세션 정보를 **공유 저장소(Redis, DB 등)** 에 저장한다.
- 어떤 서버가 요청을 받아도 동일하게 처리할 수 있다.
- 서버 확장/축소가 자유롭고 구조가 단순하다.
---
## 9. 데이터 센터
글로벌 서비스는 여러 지역에 데이터 센터를 두고 **지리적 라우팅(GeoDNS Routing)** 으로 가장 가까운 센터에 연결한다.
### 다중 데이터 센터 구성 시 해결해야 할 문제
- **트래픽 우회**: 장애 발생 시 정상 데이터 센터로 자동 전환
- **데이터 동기화**: 여러 센터에 걸친 DB 복제 전략 필요
- **테스트와 배포**: 모든 센터에 동일 코드가 배포되어야 하며 배포 자동화가 중요
---
## 10. 메시지 큐 (Message Queue)
시스템 컴포넌트를 분리(Decoupling)하고 비동기 처리를 가능하게 하는 미들웨어. Kafka, RabbitMQ, Amazon SQS 등.
```
Producer → [메시지 큐] → Consumer
```
### 장점
- **느슨한 결합**: Producer와 Consumer가 독립적으로 확장/배포 가능
- **메시지 무손실**: Consumer가 다운되어도 메시지는 큐에 남아 있다
- **비동기 처리**: 오래 걸리는 작업을 큐에 넣고 즉시 응답 반환
---
## 11. 로그, 메트릭, 자동화
- **로그**: 여러 서버의 로그를 단일 서비스에 모아 검색/분석 (ELK Stack, Datadog 등)
- **메트릭**: CPU/메모리/디스크 같은 호스트 메트릭, DB/캐시 성능, DAU 같은 비즈니스 메트릭
- **자동화**: CI/CD 파이프라인으로 빌드·테스트·배포 자동화
---
## 12. 데이터베이스 규모 확장
### 수평적 확장 — 샤딩 (Sharding)
데이터를 여러 DB 서버(샤드)에 나누어 저장. **샤딩 키(Sharding Key)** 로 어느 샤드에 저장할지 결정한다.
> **파티셔닝 vs 샤딩**
> - 파티셔닝: 하나의 DB 서버 안에서 논리적으로 분할
> - 샤딩: 여러 서버에 걸쳐 물리적으로 분산
### 샤딩의 단점
- **재샤딩(Resharding)**: 데이터가 불균등해지면 샤드 재배치 필요 → 일관된 해싱(Consistent Hashing)으로 완화
- **핫스팟 키**: 특정 샤드에 트래픽 집중 (예: 팔로워 수백만인 유명인 데이터)
- **조인 어려움**: 샤드 간 JOIN 불가 → 비정규화(Denormalization)로 완화
---
## 13. 정리
| 기법 | 목적 |
|---|---|
| 무상태 웹 계층 | 서버 확장 용이 |
| 다중화 (Redundancy) | 가용성, 장애 대응 |
| 캐시 적극 활용 | 응답 속도, DB 부하 감소 |
| 다중 데이터 센터 | 지역 장애 대응, 레이턴시 감소 |
| 정적 콘텐츠 CDN 배포 | 응답 속도 |
| 데이터 계층 샤딩 | DB 수평 확장 |
| 계층별 독립 서비스 분리 | 독립적 확장, 유지보수 |
| 모니터링 + 자동화 | 운영 효율, 빠른 장애 감지 |
