# 처리율 제한 장치(Rate Limiter)의 설계

## 처리율 제한 장치란?

특정 클라이언트나 API가 단위 시간당 보낼 수 있는 요청 수를 제한하는 장치다. 예를 들어 "1초에 최대 10건"처럼 정해두고, 초과하는 요청은 거부(또는 지연)한다.

## 왜 필요한가

- DoS 공격이나 봇으로 인한 자원 고갈을 막을 수 있다. 그래서 대부분의 기업이 공개하는 API는 어떤 형태로든 rate limit을 가지고 있다.
- 비용 절감: 외부 써드파티 API에 사용료를 지불하고 쓰는 경우, 비용을 예상하거나 제한할 수 있어야 한다.
- 서버 과부하 방지.

## 설계 전 확인해야 할 질문들

면접이든 실제 설계든 아래 질문으로 요구사항을 좁혀가는 게 첫 단계다.

- 클라이언트 측에서 제한할지, 서버 측에서 제한할지. 보통 서버 측이 신뢰할 수 있는 위치이므로 기본은 서버 측이다.
- 어떤 기준으로 제한할지: IP, 사용자 ID, API 엔드포인트 등 여러 조합이 가능하다.
- 시스템 규모(요청량, 사용자 수)와 분산 환경 여부. 서버가 여러 대라면 카운터를 어디서 어떻게 공유할지가 설계의 핵심이 된다.
- 처리율 제한 장치를 독립된 서비스로 둘지, 애플리케이션 코드 안에 내장할지.
- 제한에 걸린 사용자에게 알릴지 여부와 방식. 보통 HTTP 429와 헤더로 한도 정보를 알려준다.

## 처리율 제한 알고리즘

| 알고리즘 | 특징 |
|---|---|
| 토큰 버킷(Token Bucket) | 일정 속도로 토큰이 채워지고 요청마다 소비. 토큰이 없으면 거부. 버스트 트래픽을 어느 정도 허용 |
| 누출 버킷(Leaky Bucket) | 요청을 큐에 담아 일정한 속도로 처리(FIFO). 큐가 가득 차면 새 요청을 버림. 출력 속도가 일정 |
| 고정 윈도 카운터(Fixed Window Counter) | 시간을 고정 구간으로 나눠 카운트. 구현이 단순하지만 윈도 경계에서 트래픽이 두 배로 몰릴 수 있음 |
| 이동 윈도 로그(Sliding Window Log) | 요청마다 타임스탬프를 로그로 저장하고 윈도 안의 로그 개수를 셈. 정확하지만 메모리 사용량이 큼 |
| 이동 윈도 카운터(Sliding Window Counter) | 고정 윈도와 이동 윈도 로그의 절충. 이전/현재 윈도 카운트를 가중 평균. 메모리 효율과 정확도의 균형이 좋음 |

### 토큰 버킷 (Token Bucket)

버킷에 용량(capacity)이 있고, 일정 속도로 토큰이 채워진다. 예: 용량 4개, 초당 2개씩 충전.

요청이 오면 토큰 1개를 소비하고 통과시킨다. 토큰이 없으면 거부.

```
시간 0초: 토큰 4개 (가득 찬 상태)
→ 요청 4개가 동시에 와도 전부 통과 (버스트 허용)
→ 토큰 0개

시간 0.5초: 토큰 1개 충전됨 (초당 2개 = 0.5초당 1개)
→ 요청 1개 오면 통과, 토큰 0개
```

핵심은 평소에 안 쓰던 토큰이 쌓여있다가, 순간적으로 몰리는 트래픽을 흡수할 수 있다는 점이다. "1초에 평균 2건"이라는 제약은 지키면서도, 잠깐의 버스트는 봐준다.

### 누출 버킷 (Leaky Bucket)

요청이 오면 일단 큐(버킷)에 쌓아두고, 고정된 속도로 하나씩 꺼내서 처리한다. 큐가 가득 차면 새 요청은 버린다.

```
큐 용량 4개, 처리 속도 초당 2개

시간 0초: 요청 4개 도착 → 큐에 [1,2,3,4]
시간 0.5초: 큐에서 1개 꺼내 처리 → 큐 [2,3,4]
시간 1.0초: 큐에서 1개 꺼내 처리 → 큐 [3,4]
...
```

토큰 버킷과의 차이는, 토큰 버킷은 들어오는 요청을 토큰만 있으면 즉시 통과시키지만(버스트 가능), 누출 버킷은 출력 속도가 항상 일정하다는 점이다. 큐에 쌓였다 처리되니까 일종의 평탄화(smoothing) 효과가 있다.

### 고정 윈도 카운터 (Fixed Window Counter)

시간을 1분, 1초 같은 고정된 구간으로 자르고, 그 구간 안의 요청 수를 센다. 예: "1분당 5건".

```
00:00:00 ~ 00:00:59 → 카운터 0부터 시작, 5건까지 허용
00:01:00 ~ 00:01:59 → 카운터 리셋, 다시 5건까지 허용
```

문제는 윈도 경계에서 발생한다.

```
00:00:55 ~ 00:00:59 사이에 5건 요청 → 통과
00:01:00 ~ 00:01:04 사이에 5건 요청 → 새 윈도라서 또 통과

→ 실제로는 00:00:55 ~ 00:01:04, 즉 10초 동안 10건이 처리됨
→ "1분당 5건" 의도와 다르게 짧은 시간에 두 배가 몰릴 수 있음
```

### 이동 윈도 로그 (Sliding Window Log)

요청마다 타임스탬프를 기록해두고, 현재 시점부터 거꾸로 윈도 크기만큼의 구간에 타임스탬프가 몇 개 있는지 센다.

```
"1분당 5건" 제한, 현재 시각 00:01:30

저장된 타임스탬프: [00:00:40, 00:00:50, 00:01:10, 00:01:25, 00:01:29]
→ 윈도 범위: 00:00:30 ~ 00:01:30
→ 이 범위 안의 타임스탬프 5개 → 한도 도달, 새 요청 거부

시간이 흘러 00:01:45이 되면
→ 윈도 범위: 00:00:45 ~ 00:01:45
→ 00:00:40은 범위 밖으로 빠짐 → 4개만 남음 → 새 요청 1건 허용
```

고정 윈도의 경계 문제가 사라진다. 항상 "지금부터 정확히 1분 전까지"를 기준으로 세기 때문이다. 하지만 사용자마다 모든 요청의 타임스탬프를 저장해야 하므로, 트래픽이 많을수록 메모리 사용량이 커진다.

### 이동 윈도 카운터 (Sliding Window Counter)

고정 윈도처럼 구간별로 카운트만 저장하면서(메모리 절약), 이동 윈도처럼 경계 문제를 줄이기 위해 이전 구간과 현재 구간을 가중 평균한다.

```
"1분당 5건" 제한
이전 윈도(00:00:00~00:00:59): 3건
현재 윈도(00:01:00~00:01:59): 2건, 현재 시각 00:01:30 (현재 윈도의 50% 지점)

추정 요청 수 = 현재 윈도 카운트 + 이전 윈도 카운트 × (1 - 현재 윈도 진행률)
            = 2 + 3 × (1 - 0.5)
            = 2 + 1.5
            = 3.5

→ 5보다 작으므로 통과
```

이전 윈도의 요청들이 아직 일부 영향을 미치고 있다고 가정해서 비율로 깎아 더하는 방식이다. 타임스탬프를 전부 저장하지 않고 구간별 카운트 2개만 저장하면 되므로 메모리 효율이 좋고, 고정 윈도의 경계 문제도 상당 부분 완화된다. 다만 균등 분포를 가정한 근사치라서 이동 윈도 로그처럼 100% 정확하지는 않다.

### 정리: 무엇을 트레이드오프하는가

| | 정확도 | 메모리 | 버스트 허용 |
|---|---|---|---|
| 토큰 버킷 | - | 적음 | 허용 (의도적) |
| 누출 버킷 | - | 적음 (큐 크기만큼) | 허용 안 함 |
| 고정 윈도 카운터 | 낮음 (경계 문제) | 가장 적음 | 경계에서 의도치 않게 허용됨 |
| 이동 윈도 로그 | 가장 높음 | 가장 많음 | 정밀하게 제어 |
| 이동 윈도 카운터 | 중간 (근사치) | 적음 | 어느 정도 완화됨 |

토큰 버킷/누출 버킷은 트래픽을 어떻게 흘려보낼지(버스트 허용 여부)에 초점이 있고, 나머지 셋은 주어진 시간 윈도 안의 요청 수를 얼마나 정확하게/효율적으로 셀지에 초점이 있다고 보면 이해하기 쉽다.

## 카운터는 어디에 저장하나

보통 인메모리 캐시에 저장한다. 시간 기반 만료(expiration) 정책을 기본으로 지원하고, 읽기/쓰기 속도가 매우 빠르기 때문이다. 이 두 특성이 rate limiter가 요구하는 스펙과 정확히 맞아떨어진다.

그래서 실무에서는 Redis를 많이 사용한다. `INCR`로 카운터를 증가시키고, `EXPIRE`로 윈도 시간이 지나면 자동으로 키를 삭제하는 방식이 대표적이다.

## 분산 환경에서의 처리율 제한

### 생기는 문제

서버가 한 대일 때는 메모리에 카운터를 두면 충분하지만, 서버가 여러 대로 늘어나면 두 가지 문제가 생긴다.

- **경쟁 조건(Race Condition)**: 여러 요청이 동시에 카운터를 읽고 +1 해서 다시 쓰는 과정이 겹치면, 실제로는 한도를 초과했는데도 카운터에 반영이 안 되는 경우가 생긴다. "읽기 → 계산 → 쓰기"가 원자적이지 않아서 발생.
- **동기화 문제(Synchronization)**: 로드밸런서 뒤에 여러 대의 rate limiter 서버가 있고 각자 자기 메모리에 카운터를 들고 있다면, 요청이 어느 서버로 가느냐에 따라 카운터 값이 달라진다. 결국 전체 한도가 의미 없어진다.

### 경쟁 조건 해결: Lua 스크립트

가장 흔한 해법은 Redis의 Lua 스크립트로 "읽기-계산-쓰기"를 하나의 원자적 연산으로 묶는 것이다. Redis는 단일 스레드로 명령을 처리하므로, 스크립트가 끝나기 전까지 다른 클라이언트의 요청은 큐에서 대기한다. 즉 애초에 동시 접근이 일어나지 않는 실행 단위로 묶이는 것이라서, 락을 걸고-작업하고-푸는 별도의 단계 자체가 필요 없다.

다른 방법으로 정렬 집합(sorted set)을 이용한 분산 락 구현도 있다. 이건 락 획득 시도 → 획득 여부 확인 → 작업 → 락 해제까지 여러 번의 Redis 왕복(round trip)이 필요하고, 클라이언트가 중간에 죽으면 락이 풀리지 않는 데드락 문제도 신경 써야 한다. 그래서 성능 면에서는 Lua 스크립트 쪽이 더 낫다.

```lua
local c = redis.call('INCR', KEYS[1])
if c == 1 then redis.call('EXPIRE', KEYS[1], ARGV[1]) end
return c
```

### 동기화 문제 해결: 중앙 집중식 저장소

여러 rate limiter 서버가 개별 메모리 대신 Redis 같은 중앙 집중식 캐시를 사용하도록 한다. 모든 서버가 같은 Redis를 보고 카운터를 읽고 쓰기 때문에, 사용자가 어떤 서버로 라우팅되든 일관된 카운터 값을 보게 된다.

다만 모든 요청마다 Redis 호출이 필요해 네트워크 지연이 추가된다. 이를 완화하려고 sticky session(같은 사용자를 같은 서버로 라우팅)을 쓰는 경우도 있지만, 확장성과 가용성 측면에서 트레이드오프가 있다.

## 아키텍처 구성

```
클라이언트 → 로드밸런서 → [Rate Limiter] → 애플리케이션 서버
                              ↓
                        Redis (카운터 저장)
```

요청이 들어오면 rate limiter가 Redis에서 해당 사용자/IP의 카운터를 확인하고, 한도 내면 통과시켜 애플리케이션 서버로 보내고, 한도 초과면 429 응답을 바로 반환한다. 한도 초과 요청은 애플리케이션 서버까지 도달하지 않아 서버 부담을 줄여준다.

Rate limiter는 미들웨어 형태로 둘 수도 있고, API 게이트웨이의 기능으로 둘 수도 있다. 마이크로서비스 환경이라면 API 게이트웨이에서 한 번에 처리하는 게 일반적이다.

## 처리율 제한 정보 전달 방식

요청이 거부됐을 때 클라이언트에게 알려주는 게 좋은 경험을 제공한다. 보통 HTTP 응답 헤더에 다음 정보를 담는다.

- `X-Ratelimit-Remaining`: 윈도 내 남은 허용 횟수
- `X-Ratelimit-Limit`: 윈도 내 허용되는 요청 횟수
- `X-Ratelimit-Retry-After`: 다시 요청 가능해지는 시점까지 남은 시간

한도를 초과하면 `429 Too Many Requests`와 함께 `Retry-After` 헤더를 반환하는 것이 표준적인 방식이다.

## 성능 최적화

- **에지 서버 활용**: 클라이언트와 가장 가까운 에지 서버에서 처리율 제한을 적용하면, 요청이 origin 서버까지 가지 않고 더 빨리 걸러진다. CDN/edge 수준에서 1차 필터링을 하고, 더 세밀한 제한은 애플리케이션 레벨에서 처리하는 다단계 구조가 실무에서 많이 쓰인다.
- **최종 일관성(eventual consistency) 모델 사용**: 분산 환경에서 모든 노드의 카운터를 매번 강한 일관성으로 동기화하면 성능 비용이 크다. 약간의 오차를 허용하는 최종 일관성 모델을 쓰면 성능과 정확도 사이에서 합리적인 절충이 가능하다.
- **모니터링을 통한 규칙 조정**: 처리율 제한 규칙이 트래픽 패턴에 맞는지 지속적으로 모니터링하고, 너무 빡빡하거나 느슨하면 조정해야 한다.

## 경성(Hard) vs 연성(Soft) 처리율 제한

- **경성 제한(Hard Limiting)**: 요청 수가 한도를 절대 넘지 못하도록 강제. 한도를 넘는 순간 바로 차단.
- **연성 제한(Soft Limiting)**: 한도를 약간 넘는 것까지는 허용하는 정책. 트래픽이 일시적으로 튀는 상황(버스트)에 좀 더 유연하게 대응할 수 있다.

## 다양한 계층에서의 처리율 제한

이번 장에서는 주로 애플리케이션 계층에서의 처리율 제한을 다뤘다. 하지만 다른 계층에서도 처리율 제한이 가능하다.

- **네트워크 계층(IP 기반)**: `iptables` 같은 도구를 사용하면 IP 주소 단위로 처리율을 제한할 수 있다. OSI 7계층 중 애플리케이션 계층(7계층) 이전인 더 아래 계층에서 걸러내는 방식이라, 애플리케이션 서버까지 도달하기 전에 악성 트래픽을 차단할 수 있다는 장점이 있다.
- **API 게이트웨이 계층**: 마이크로서비스 환경에서 게이트웨이 단에서 한 번에 처리.
- **애플리케이션 코드 내부**: 비즈니스 로직에 맞춘 세밀한 제한(스코프별, 행동별 등).

여러 계층을 조합하면 더 견고한 방어가 가능하다 (네트워크 계층에서 큰 공격을 거르고, 애플리케이션 계층에서 세밀하게 제한).

## 처리율 제한을 회피하는 방법과 클라이언트 설계

클라이언트 입장에서 처리율 제한에 걸렸을 때 어떻게 대응하는 것이 좋은 설계인가도 함께 생각해볼 부분이다.

- **클라이언트 측 캐싱**: 같은 요청을 반복하지 않도록 캐시를 활용해 호출 자체를 줄인다.
- **재시도 시 충분한 대기 시간(backoff) 적용**: 429를 받으면 즉시 재시도하지 말고, 서버가 보내준 `Retry-After` 값을 참고해 대기한다.
- **지수 백오프(Exponential Backoff)**: 재시도할 때마다 대기 시간을 점진적으로 늘려서, 제한이 풀릴 때까지 서버에 부담을 주지 않는다.
- **에러 코드 확인**: 요청이 처리율 제한에 의해 걸러졌는지 사용자에게 알 수 있도록 클라이언트 코드에서 429와 관련 헤더(`X-Ratelimit-*`)를 명확히 처리한다.

이런 설계는 "처리율 제한을 회피"하는 것이 아니라, 제한이 걸렸을 때 서버와 클라이언트 모두에게 부담을 덜 주는 방향으로 협력하는 것에 가깝다. 반대로 여러 IP를 돌려가며 우회하거나 캡차 팜 등을 쓰는 방식은 처리율 제한의 취지를 무력화하는 것이고, 결국 서비스 제공자 입장에서는 더 강한 방어 계층(네트워크 단 차단, 행동 기반 분석 등)을 추가하는 식으로 대응하게 된다.

## 실제 적용 사례: chess-analysis 프로젝트

위에서 다룬 개념들이 실제 프로젝트에서 어떻게 구현되는지 살펴본다.

### 공통 메커니즘

Redis 기반 `INCR` + `EXPIRE` 원자적 카운터를 Lua 스크립트로 묶어 race condition을 방지하는 패턴을 두 서비스 모두 동일하게 사용한다.

```lua
local c = redis.call('INCR', KEYS[1])
if c == 1 then redis.call('EXPIRE', KEYS[1], ARGV[1]) end
return c
```

### AnalysisRateLimitService — AI 분석 요청 제한

이 제한을 건 이유는 AI API 비용과 스톡피시(Stockfish) CPU 연산 비용을 줄이기 위해서다. 분석 요청 한 번마다 외부 AI API 호출 비용과 체스 엔진의 CPU 연산 비용이 발생하므로, 이 두 비용을 통제 가능한 범위로 유지하는 게 목적이다. 앞서 다룬 "비용 절감을 위한 처리율 제한"의 실제 적용 사례인 셈이다.

| 스코프 | 제한 | 키 패턴 |
|---|---|---|
| 유저명 (fast) | 1일 3회 | `rate:analysis:username:{platform}:{username}:fast:{date}` |
| 유저명 (precise) | 1일 1회 | `rate:analysis:username:{platform}:{username}:precise:{date}` |
| IP | 1일 10회 | `rate:analysis:ip:{ip}:{date}` |
| 전체(글로벌) | 1일 200회 | `rate:analysis:global:{date}` |
| 대기열 크기 | 동시 30개 초과 시 거부 | (Redis 큐 길이 체크) |

- 기준 시간대: `Asia/Seoul`, 매일 자정 리셋
- 화이트리스트: `chess-analysis.rate-limit.whitelist.usernames` / `.ips` 환경변수로 우회 가능
- 초과 시 `AnalysisRateLimitException` 발생, 응답에 `scope/limit/current/remaining/resetAt` 포함

이 구조는 위에서 다룬 "다양한 기준(유저명/IP/글로벌)을 조합한 다단계 제한"과 "고정 윈도 카운터(날짜 단위로 리셋)"의 실제 예시다. 스코프별로 다른 한도를 두는 것은, 같은 사용자라도 더 비싼 연산(precise)에는 더 엄격한 제한을 거는 식의 비용 기반 설계로 볼 수 있다.

### SignupRateLimitService — 인증/커뮤니티 관련

| 동작 | 제한 | 윈도우 |
|---|---|---|
| 회원가입 | IP당 10회 | 1시간 |
| 인증메일 재발송 | 이메일당 5회 | 1시간 |
| 이메일 중복확인 | IP당 30회 | 1분 |
| 닉네임 중복확인 | IP당 30회 | 1분 |
| 비밀번호 재설정 요청 | 이메일당 5회 | 1시간 |
| 게시글 작성 | 유저당 5개 | 10분 |
| 채팅/댓글 | 유저당 20개 | 1분 |

초과 시 `429 TOO_MANY_REQUESTS`와 함께 "잠시 후 다시 시도해주세요." 메시지를 반환한다. 위에서 다룬 "처리율 제한 정보 전달 방식(429 + Retry 안내)"이 실제로 적용된 형태다.
