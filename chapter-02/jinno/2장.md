# 처리율 제한 장치 설계

### 배경
처리율 제한 장치란 네트워크 시스템에서 클라이언트 또는 서비스가 보내는 트래픽의 처리율을 제어하기 위한 장치다. 특정 API에 대한 요청 횟수가 임계치를 넘어서면 추가로 도달한 모든 호출을 block 시킨다. 이를 통해, 다음과 같은 여러 가지 이점을 얻을 수 있다.
- DoS(Denial of Service) 공격에 의한 자원 고갈을 방지할 수 있다.
- 외부 API 호출에 대한 비용 절감
- 서버 과부하 방지

---

### 문제 이해 및 설계 범위 확정
본 글에서는 다음과 같은 대규모 분산 환경의 서비스 요구사항을 기준으로 시스템을 설계하겠다.

- **클라이언트 측 제한 장치 vs 서버 측 제한 장치** : 서버 측 제한 장치 도입
- **제어 기준** : 다양한 형태의 제어 규칙을 정의할 수 있도록 유연한 시스템이어야 한다.
- **시스템 규모** : 대규모 요청을 처리할 수 있어야 한다.
- **분산 환경 동작 여부** : O (분산 환경 지원)
- **독립 서비스 여부** : 애플리케이션 코드에 포함하거나 독립 게이트웨이 서비스로 둘 수 있다.
- **처리율 제한 알림 여부** : 제한 장치에 의해 요청이 걸러진 경우 사용자에게 사실을 알려야 한다.

---

### 개략적 설계안 제시 및 동의 구하기

만약 처리율 제한 장치를 클라이언트 측에 둔다면 어떻게 될까? 클라이언트의 요청은 쉽게 위변조가 가능하기도 하고, 모든 클라이언트의 구현을 통제하는 것은 어렵기 때문에 처리율 제한을 안정적으로 두기 어렵다.

따라서 서버 측에 제한 장치를 두는 가장 쉬운 방법은 **미들웨어(Middleware)**의 형식이다. API 서버로 가는 요청을 중간에서 통제하도록 하는 것이다. 만약 초당 2개의 요청으로 서버의 처리율이 제한되어 있다면, 3번째 요청을 처리율 제한 장치에서 가로막고 429 status를 반환하면 된다. 

MSA(마이크로서비스 아키텍처) 환경에서는 보통 **API 게이트웨이(API Gateway)**에서 처리율 제한 장치가 구현된다. 사실 서버에 두어도 되고 게이트웨이에 두는지에 대한 정답은 없다. 회사에서 사용하는 기술 스택이나 엔지니어링 인력, 우선순위, 목표에 따라 달라질 수 있다고 생각한다.

### 알고리즘 정하기
처리율 제한 알고리즘엔 여러 가지가 있고, 각기 다른 장단점을 갖고 있으므로 요구사항에 맞게 하나를 선택하는 것이 중요해 보인다.
- 토큰 버킷(token bucket)
- 누출 버킷(leaky bucket)
- 고정 윈도 카운터(fixed window counter)
- 이동 윈도 로그(sliding window log)
- 이동 윈도 카운터(sliding window counter)

#### 1. 토큰 버킷 알고리즘
처리율 제한에 폭넓게 이용되고 있다. 아마존과 스트라이프가 API 요청을 통제하기 위해 이 알고리즘을 사용한다고 한다. 지정된 용량을 갖는 컨테이너라는 토큰 버킷이 존재하며, 사전 설정된 양의 토큰이 주기적으로 채워지고 꽉 찬 버킷에는 더 이상의 토큰은 추가되지 않는다. 



각 요청은 처리될 때마다 하나의 토큰을 사용한다. 요청이 도착하면 버킷에 충분한 토큰이 있는지 검사한다.
- 충분한 토큰이 있다면 버킷에서 토큰 하나를 꺼낸 후 요청을 시스템에 전달한다.
- 충분한 토큰이 없는 경우 해당 요청은 버려진다.

토큰 버킷 알고리즘은 2개의 파라미터를 받는다.
- **버킷 크기** : 버킷에 담을 수 있는 토큰의 최대 개수
- **토큰 공급률** : 초당 몇 개의 토큰이 버킷에 공급되는가

예를 들어서, 하루에 한 번만 포스팅을 할 수 있다는 요구사항이 있다면 버킷 크기는 1개이고, 토큰 공급률은 하루에 한 개가 될 것이다. 통상적으로 API 엔드포인트마다 요구사항이 다르기 때문에 별도의 버킷을 두며, IP 주소별 제한이 필요하다면 IP마다 버킷을 할당해야 한다. 만약 모든 요청을 일정 개수 이하로 묶고 싶다면 하나의 버킷을 공유하게 만들면 된다.

구현이 쉽고 메모리 사용 측면에서도 효율적이지만, 버킷 크기와 토큰 공급률이라는 두 개의 파라미터를 적절히 튜닝하는 게 까다로울 수 있다.

#### 2. 누출 버킷 알고리즘
토큰 버킷과 비슷하지만 요청 처리율이 고정되어 있다. FIFO(First-In-First-Out) 큐로 구현한다. 즉, 토큰 버킷은 단시간에 트래픽이 몰리는 폭발(Burst)을 허용하지만, 누출 버킷은 이를 허용하지 않고 고정된 속도로만 출력한다.



- 요청이 도착하면 큐가 가득 차 있는지 보고, 빈자리가 있는 경우에는 큐에 요청을 추가한다.
- 큐가 가득 차 있는 경우에는 새 요청을 버린다.
- 지정된 시간마다 큐에서 요청을 꺼내어 고정된 속도로 처리한다.

고정된 처리율을 갖고 있기 때문에 안정적 출력이 필요한 경우에 적합하지만, 단시간에 많은 트래픽이 몰리는 경우 큐에 오래된 요청들이 쌓이고, 이를 제때 처리 못하면 최신 요청들은 전부 버려지게 된다는 단점이 있다.

#### 3. 고정 윈도 카운터 알고리즘
타임라인을 고정된 간격의 윈도로 나누고, 각 윈도마다 카운터를 붙인다. 요청이 접수될 때마다 이 카운터의 값은 1씩 증가하며, 카운터의 값이 사전에 설정된 임계치에 도달하면 새로운 요청은 새 윈도가 열릴 때까지 버려진다.

만약 초당 3개의 요청이 밀려오는데 특정 초에 4개의 요청이 온다면 1개의 요청이 그대로 버려진다. 이 알고리즘의 치명적인 단점은 윈도 경계 부근에 트래픽이 몰릴 때 설정한 한도보다 더 많은 요청을 처리하게 된다는 점이다. 분당 10개의 요청을 제한으로 둔다고 가정하자. 1시 59분 59초에 10개, 2시 00분 01초에 10개의 요청이 들어온다면, 사실상 윈도 경계인 2초 사이에 총 20개의 요청이 들어온 것이나 마찬가지다. 시스템을 설계할 때 이러한 특징을 반드시 신경 써야 한다.

#### 4. 이동 윈도 로깅 알고리즘
이동 윈도 로깅 알고리즘은 앞선 고정 윈도 카운터의 경계 문제를 해결해 준다.
- 요청의 타임스탬프(Timestamp)를 추적하며, 이 데이터는 보통 레디스(Redis)의 정렬 집합(Sorted Set) 같은 캐시에 보관한다.
- 새 요청이 오면 만료된 타임스탬프를 제거한다. 여기서 만료란 현재 윈도의 시작 지점보다 오래된 타임스탬프를 말한다.
- 새 요청의 타임스탬프를 로그에 추가한다.
- 로그의 크기가 허용치보다 같거나 작으면 요청을 시스템에 전달하고, 그렇지 않은 경우에는 처리를 거부한다.

이 알고리즘은 어느 순간을 보더라도 허용되는 요청의 개수가 시스템의 처리율 한도를 넘지 않아 매우 정교하다. 하지만 거부된 요청의 타임스탬프까지 전부 보관하기 때문에 다량의 메모리를 사용하게 된다는 치명적인 단점이 있다.

#### 5. 이동 윈도 카운터 알고리즘
고정 윈도 카운터 알고리즘과 이동 윈도 로깅 알고리즘을 결합한 형태다. 처리율 제한 장치의 한도가 분당 7개라고 하고, 이전 1분 동안 5개의 요청이, 현재 1분 동안 3개의 요청이 왔다고 하자. 현재 1분의 30% 시점에 도착한 새 요청의 경우, 현재 윈도 내의 요청 수는 다음과 같은 공식을 쓴다.

- `현재 1분간의 요청 수 + 직전 1분간의 요청 수 * 이동 윈도와 직전 1분이 겹치는 비율`
- 이 공식에 대입하면 현재 윈도에 들어 있는 요청은 `3 + 5 * 70% = 6.5`개다. 소수점 처리에 따라 버림이나 올림을 적용해 판단하면 된다.

이 알고리즘을 사용한다면 완벽하지는 않지만 짧은 시간에 몰리는 트래픽 경계 문제에도 잘 대응하고 메모리 효율이 좋다. 하지만 직전 시간대에 도착한 요청이 균등하게 분포되어 있다고 가정한 상태에서 계산을 하기 때문에 100% 완벽한 정밀도는 아니다. 다만, 클라우드 플레어의 실험에 따르면 40억 개 요청 가운데 억까(오차로 인한 오판단)당한 건은 0.003%뿐이라고 하니 대규모 환경에서 쓰기 매우 잘 짜여진 알고리즘 같다.

### 개략적인 아키텍처
사실 알고리즘보다 중요한 것은 이 카운터 정보들을 어디에 보관하며 어떻게 구성할지이다. 데이터베이스는 디스크 접근 때문에 느려서 API의 전반적인 속도를 떨어뜨린다. 그렇기 때문에 메모리 기반 캐시가 바람직하다. 일례로, API 게이트웨이에서 레디스(Redis)로 이러한 정보들을 저장하고 접근하는 아키텍처가 많이 쓰인다. 이때 레디스의 `INCR`과 `EXPIRE` 두 가지 명령어를 활용한다.

- **INCR** : 메모리에 저장된 카운터의 값을 1만큼 증가시킨다. (++)
- **EXPIRE** : 카운터에 타임아웃 값을 설정하며, 설정된 시간이 지나면 카운터는 자동으로 삭제된다.



**동작 원리:**
1. 클라이언트가 미들웨어(게이트웨이)에 요청을 보낸다.
2. 미들웨어는 레디스의 버킷에서 카운터를 가져와서 한도 검사를 한다. 한도가 꽉 찼다면 거부된다.
3. 한도가 꽉 차지 않았다면 요청은 API 서버로 전달되고, 미들웨어는 카운터 값을 증가시킨 후 다시 레디스에 저장한다.

### 상세 설계
방금 본 개략적 아키텍처만으로는 처리율 제한 규칙이 어떻게 만들어지고 어디에 저장되는지, 그리고 처리가 제한된 요청들은 구체적으로 어떻게 클라이언트와 통신하는지 설명하기 어렵다.

#### 처리율 제한 규칙
리프트(Lyft)는 처리율 제한에 오픈 소스(Envoy Proxy Rate Limit)를 사용하고 있다. 설정 파일(.yml 등) 형태로 디스크에 저장하여 관리하는 것이 일반적이다. 예를 들어 시스템이 처리할 수 있는 마케팅 메시지의 최대치(`request_per_unit`)를 하루 5개로 제한하는 등의 디테일한 규칙을 정의해 두고 미들웨어가 이를 읽어와 적용한다.

#### 처리율 한도 초과 트래픽은 어떻게 처리하나?
기본적으로 `429 Too Many Requests` 상태 코드를 응답으로 보내야 한다. 경우에 따라서는 한도 제한이 걸린 메시지를 캐시나 메시지 큐 같은 곳에 보관했다가 나중에 비동기적으로 처리할 수도 있다. 예를 들어 시스템 과부하 때문에 결제나 주문 요청이 한도 제한에 걸렸다면, 무작위로 버리기보다 큐에 담아두고 나중에 순차적으로 처리하는 것이 비즈니스 측면에서 이롭기 때문이다.

클라이언트는 본인의 요청이 처리율 제한에 걸리고 있는지를 어떻게 감지할까? 처리율 제한 장치에서 다음과 같은 HTTP 헤더를 응답에 포함해 보내주면 된다.

- `X-Ratelimit-Remaining`: 윈도 내에 남은 처리 가능 요청의 수
- `X-Ratelimit-Limit`: 매 윈도마다 클라이언트가 전송할 수 있는 요청의 수
- `X-Ratelimit-Retry-After`: 한도 제한에 걸리지 않으려면 몇 초 뒤에 요청을 다시 보내야 하는지 알림

### 분산 환경에서의 핵심 난제

단일 서버에서는 카운터 관리가 쉽지만, 여러 대의 서버와 병렬 스레드를 지원하도록 시스템을 확장하는 순간 분산 환경의 독들이 터져 나오기 시작한다.

#### 1. 경쟁 조건 (Race Condition)
레디스에서 카운터 값을 읽고, `counter + 1`이 임계치를 넘지 않는지 확인한 뒤, 넘지 않는다면 요청을 통과시키고 카운터 값을 증가시키는 일련의 과정을 생각해 보자. 만약 병렬로 동작하는 두 개의 스레드가 거의 동시에 레디스의 카운터 값을 읽었다면 어떻게 될까? 두 스레드 모두 업데이트 전의 카운터 값을 읽었기 때문에, 실제로는 임계치를 넘었음에도 두 요청 모두 통과 처리되는 정합성 오류가 발생한다. 

가장 쉬운 해결책은 자원에 락(Lock)을 거는 것이지만, 락은 분산 환경에서 시스템 전체의 성능을 심각하게 떨어뜨린다. 대용량 처리를 해야 하는 상황에서 분산 락은 곧 독이 되기 때문에, 실무에서는 **루아 스크립트(Lua Script)**와 **정렬 집합(Sorted Set)**이라는 레디스의 기능을 사용해 이를 해결한다.

- **루아 스크립트:** 레디스는 단일 스레드로 명령어를 처리한다는 특징이 있다. 루아 스크립트를 사용하면 카운터를 읽고 조건 검사를 한 뒤 값을 증가시키는 여러 개의 명령어를 하나의 단일 명령어(Atomic Operation)처럼 묶어서 실행할 수 있다. 즉, 스레드 A의 루아 스크립트가 실행되는 동안 스레드 B의 접근이 원천 차단되므로 락 없이도 경쟁 조건을 완벽히 방지할 수 있다.
- **정렬 집합(Sorted Set):** 이동 윈도 로그 알고리즘 등을 구현할 때 레디스의 정렬 집합을 활용한다. 타임스탬프를 `Score`로 삼아 데이터를 저장하면 윈도 밖의 만료된 타임스탬프를 제거(`ZREMRANGEBYSCORE`)하고 현재 카운트를 세는(`ZCARD`) 모든 일련의 과정을 레디스 내부에서 원자적(Atomic)으로 안전하게 처리할 수 있어서 경쟁 조건 문제를 우아하게 비껴갈 수 있다.

#### 2. 동기화 이슈 (Synchronization)
처리율 제한 장치 서버가 여러 대라면 웹 계층은 무상태(Stateless)이므로 클라이언트의 요청이 매번 다른 처리율 제한 장치 서버로 라우팅될 수 있다. 만약 제한 장치들이 각각 자신만의 로컬 카운터를 가지고 있다면, 다른 장치 서버에 기록된 사용자의 이전 요청 횟수를 알지 못하므로 처리율 제한이 올바르게 수행될 수 없다.

이에 대한 해결책으로 로드밸런서에서 고정 세션(Sticky Session)을 활용해 클라이언트가 항상 같은 처리율 제한 장치 서버로만 가게 강제할 수도 있지만, 이 방식은 규모 면에서 스케일 아웃 확장 가능하지도 않고 유연하지도 않아 대용량 처리 환경에는 어울리지 않는다. 훨씬 더 나은 해결책은 모든 처리율 제한 장치 서버들이 로컬에 카운터를 두지 않고, 중앙 집중형 저장소인 공유 레디스(Redis)를 바라보게 구조를 설계하는 것이다. 이렇게 하면 어떤 제한 장치 서버로 트래픽이 튀더라도 항상 일관된 카운터 값을 기반으로 처리율을 올바르게 제어할 수 있게 된다.
